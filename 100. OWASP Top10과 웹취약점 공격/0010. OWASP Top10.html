0. OWASP 프로젝트의 이해 (Deep Dive into OWASP)

OWASP(The Open Worldwide Application Security Project, 오픈 소스 웹 애플리케이션 보안 프로젝트)는 2001년 설립된 비영리 재단으로, 전 세계 소프트웨어의 보안성을 향상시키는 것을 목표로 합니다. 특정 벤더에 종속되지 않는 중립성(Vendor-Neutral)을 원칙으로 하며, 수만 명의 보안 전문가들이 자발적으로 참여하는 글로벌 커뮤니티입니다.

Mission

보안 위협 가시화, 표준 가이드 제공, 보안 도구 개발 및 배포를 통해 안전한 소프트웨어 생태계 구축

Impact

PCI DSS, NIST, ISO 27001 등 주요 글로벌 컴플라이언스의 핵심 참조 표준으로 채택

주요 프로젝트 포트폴리오

Flagship
OWASP Top 10

웹 애플리케이션 보안 10대 위협 순위 및 기술 분석 가이드

A01:2021-Broken Access Control (취약한 접근 제어)

대표적 침해 사례 (Case Study)

Facebook(페이스북) "View As" 기능 악용 (2018): 공격자가 'View As(내 프로필 보기)' 기능의 취약한 접근 제어를 악용하여 약 5천만 명의 액세스 토큰(Access Token)을 탈취한 사건입니다. 사용자 권한 검증 로직의 미비로 타인의 토큰에 직접 접근이 가능했습니다.

공격 기법 (Techniques)

  • • IDOR(Insecure Direct Object Reference, 직접 객체 참조): 파라미터 변조를 통한 타인 데이터 열람
  • • CORS(Cross-Origin Resource Sharing, 교차 출처 리소스 공유) Misconfiguration: 잘못된 원격 자원 공유 설정으로 데이터 유출

취약한 코드 예시 (Vulnerable Code)

// 접근 권한 검사가 누락된 계정 삭제 로직
@PostMapping("/deleteAccount")
public String delete(@RequestParam("userId") String id) {
  userSvc.remove(id); // 세션 사용자와 파라미터 ID 일치 여부 검증 없음
  return "success";
}

방어 대응 대책 (Defense Strategy)

모든 요청에 대해 RBAC(Role-Based Access Control, 역할 기반 접근 제어) 및 ABAC(Attribute-Based Access Control, 속성 기반 접근 제어) 모델을 적용하십시오.

Apache Configuration
# 특정 관리자 리소스에 대한 접근 제한 설정
<Location "/api/admin">
  Require group administrators
  Require valid-user
</Location>

A02:2021-Cryptographic Failures (암호화 실패)

대표적 침해 사례 (Case Study)

Ashley Madison(애슐리 매디슨) 유출 사건 (2015): 데이터베이스가 유출되었을 때 비밀번호가 취약한 MD5(Message-Digest Algorithm 5, 메시지 다이제스트 알고리즘 5) 해시로 저장되어 있어, 순식간에 수백만 명의 비밀번호가 복구되었습니다. 이는 현대적인 암호화 알고리즘 부재로 인한 전형적인 사례입니다.

기술 분석 (Technical Analysis)

  • • Lack of Salt: 레인보우 테이블 공격에 취약한 솔트(Salt) 없는 해시
  • • Weak Protocol: TLS 1.0/1.1 등 지원 종료된 암호화 통신 프로토콜 사용

보안 권고 (Best Practice)

# 권장 비밀번호 해시 알고리즘 (Argon2 또는 BCrypt)
String hashedPw = BCrypt.hashpw(plainText, BCrypt.gensalt(12));

인프라 보안 설정 (Infrastructure)

# Nginx SSL/TLS 최적화 설정
ssl_protocols TLSv1.3;
ssl_ciphers 'TLS_AES_256_GCM_SHA384:TLS_CHACHA20_POLY1305_SHA256';
add_header Strict-Transport-Security "max-age=31536000" always;

A03:2021-Injection (인젝션)

대표적 침해 사례 (Case Study)

Heartland Payment Systems(하트랜드 페이먼트 시스템즈) 사건 (2008): 약 1억 3천만 개의 신용카드 정보가 SQL 인젝션(SQL Injection, SQL 삽입 공격)을 통해 탈취되었습니다. 이는 단일 인젝션 공격으로 인한 최대 규모의 피해 사례 중 하나로 기록되어 있습니다.

실무 취약점 진단 명령어

# 취약점 자동 진단 도구 (sqlmap) 사용 예시
$ sqlmap -u "http://target.com/view?id=1" --dbms=mysql --dbs

대응 핵심 기술

  • • Parameterized Query: 바인딩 매개변수를 사용하여 데이터와 쿼리 로직의 완전한 분리
  • • WAF(Web Application Firewall, 웹 애플리케이션 방화벽): 공격 패턴 기반 실시간 필터링

A10:2021-Server-Side Request Forgery (SSRF, 서버 측 요청 변조)

대표적 침해 사례 (Case Study)

Capital One(캐피털 원) 유출 사고 (2019): 공격자가 WAF 설정 오류와 SSRF(서버 측 요청 변조)를 악용하여 AWS(아마존 웹 서비스)의 Metadata Service(IMDS, 인스턴스 메타데이터 서비스)에 접근, 약 1억 명의 고객 정보를 포함한 S3 버킷 권한을 획득했습니다.

클라우드 환경의 SSRF 대응 전략

AWS IMDSv2 도입

기존의 단순 GET 요청 대신 세션 토큰이 필요한 IMDSv2(Instance Metadata Service Version 2)를 강제 활성화하여 SSRF 공격을 무력화합니다.

# AWS CLI를 통한 IMDSv2 강제 설정
$ aws ec2 modify-instance-metadata-options \
  --instance-id i-1234567890abcdef0 \
  --http-tokens required
