0. OWASP 프로젝트의 이해 (Deep Dive into OWASP)

OWASP(The Open Worldwide Application Security Project, 오픈 소스 웹 애플리케이션 보안 프로젝트)는 2001년 설립된 비영리 재단으로, 전 세계 소프트웨어의 보안성을 향상시키는 것을 목표로 합니다. 특정 벤더에 종속되지 않는 중립성(Vendor-Neutral)을 원칙으로 하며, 수만 명의 보안 전문가들이 자발적으로 참여하는 글로벌 커뮤니티입니다.

Mission (미션)

보안 위협 가시화, 표준 가이드 제공, 보안 도구 개발 및 배포를 통해 안전한 소프트웨어 생태계 구축

Impact (영향력)

PCI DSS, NIST, ISO 27001 등 주요 글로벌 컴플라이언스의 핵심 참조 표준으로 채택

주요 프로젝트 포트폴리오

Flagship

A01:2021-Broken Access Control (취약한 접근 제어)

대표적 침해 사례

Facebook(페이스북) "View As" 기능 악용 (2018): 권한 검증 로직 미비로 타인의 액세스 토큰을 탈취, 5천만 명의 계정에 무단 접근한 사건입니다.

공격 기법

  • • IDOR (Insecure Direct Object Reference): 직접 객체 참조 취약점
  • • BOLA (Broken Object Level Authorization): 객체 수준 권한 부여 실패

취약 코드

// 검증 없는 삭제
userService.delete(request.getParameter("id"));

A02:2021-Cryptographic Failures (암호화 실패)

대표적 침해 사례

Ashley Madison(애슐리 매디슨) (2015): 암호화되지 않거나 취약한 MD5 해시를 사용하여 사용자 비밀번호가 대량으로 복구된 사례입니다.

기술적 결함

  • • Weak Hash: MD5, SHA1 등 충돌 취약 해시 사용
  • • Missing Salt: 레인보우 테이블 공격 방어 실패

보안 권고

// BCrypt와 강력한 솔트 적용
BCrypt.hashpw(password, BCrypt.gensalt(12));

A03:2021-Injection (인젝션)

Heartland Payment Systems (2008)

SQL 인젝션을 통해 1억 3천만 건의 카드 정보가 탈취된 역사적 사고입니다.

방어 대책: Parameterized Query

String query = "SELECT * FROM users WHERE id = ?";
PreparedStatement pstmt = conn.prepareStatement(query);
pstmt.setString(1, userId);

A04:2021-Insecure Design (불안전한 설계)

구현의 결함이 아닌 설계 자체의 결함을 의미합니다. 보안이 개발 초기 단계부터 고려되지 않았을 때 발생하며, 소스 코드 수정만으로는 해결이 불가능한 경우가 많습니다.

설계 결함 사례

전자상거래 서비스에서 취소 요청 시 결제 수단에 대한 소유권 확인 설계가 누락되어 타인의 결제를 취소하거나 환급받는 로직 등

대응 방안: Threat Modeling

개발 전 단계에서 STRIDE (Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege) 모델을 활용한 위협 식별

A05:2021-Security Misconfiguration (보안 설정 오류)

대표적 침해 사례

AWS S3 Bucket 노출 사고: 수많은 기업이 클라우드 스토리지 설정 오류(Public Access 허용)로 인해 고객 데이터를 전 세계에 공개하는 전형적인 설정 오류 사례입니다.

기본 계정/비번
상세 에러 메시지
불필요한 포트 개방

AWS Config (유료 서비스: CI당 $0.003) 활용

지속적인 모니터링을 통해 비정상적인 설정 변경을 자동 탐지하고 차단합니다.

A06:2021-Vulnerable and Outdated Components (취약하고 오래된 구성 요소)

대표적 침해 사례

Log4Shell (2021): 자바 로깅 라이브러리인 Log4j의 취약점을 이용해 전 세계 서버가 원격 코드 실행(RCE) 위험에 노출된 사건입니다.

대응 프로세스

01

Inventory (자산 파악)

사용 중인 모든 라이브러리 버전 전수 조사

02

SCA Tooling (자동화)

Snyk, OWASP Dependency-Check 등 활용

03

Patch (패치 관리)

최신 보안 패치 즉시 적용 체계 수립

04

Unused removal (삭제)

사용하지 않는 기능 및 구성 요소 제거

A07:2021-Identification and Authentication Failures (식별 및 인증 실패)

대표적 침해 사례

Credential Stuffing (크리덴셜 스터핑): 타 사이트에서 유출된 계정 정보를 대입하여 로그인에 성공하는 공격으로, MFA 미적용 시 매우 치명적입니다.

취약점 유형

  • • 취약한 비밀번호 정책 (Brute-force 허용)
  • • 세션 타임아웃 미설정
  • • 자동화된 공격 시도에 대한 속도 제한(Rate Limit) 부재

전문가 권고

MFA (Multi-Factor Authentication, 다요소 인증) 도입은 인증 실패 리스크를 99.9% 감소시킵니다.

// 보안 세션 설정 예시
Cookie cookie = new Cookie("SESSIONID", id);
cookie.setHttpOnly(true);
cookie.setSecure(true);

A08:2021-Software and Data Integrity Failures (소프트웨어 및 데이터 무결성 실패)

SolarWinds(솔라윈즈) 공급망 공격 (2020)

소프트웨어 업데이트 서버 자체에 악성코드를 심어, 신뢰할 수 있는 배포 과정을 통해 위협이 확산된 대표적인 무결성 실패 사고입니다.

핵심 보호 기술: Digital Signature

SW Package
Hash Function
Private Key Sign

배포되는 소프트웨어는 서명 검증을 통해 변조 여부를 반드시 확인해야 합니다.

A09:2021-Security Logging and Monitoring Failures (보안 로깅 및 모니터링 실패)

평균 탐지 시간(MTTD) 200일 초과

침입이 발생했음에도 로깅 부족이나 모니터링 체계 미비로 인해 공격자가 수개월간 네트워크에 상주하는 원인이 됩니다.

로그에 포함되어야 할 정보

  • • WHO: 사용자 식별자, IP 주소
  • • WHEN: ISO 8601 표준 타임스탬프
  • • WHAT: 시도된 작업, 리소스 경로
  • • RESULT: HTTP 상태 코드 (성공/실패)

전문가 도구 (AWS 예시)

AWS CloudWatch Logs + GuardDuty: 실시간 이상 징후 탐지 및 경보 자동화

// 실패 로그 생성 예시
log.error("Login Fail | User: {} | IP: {} | Result: Forbidden", user, remoteIp);

A10:2021-Server-Side Request Forgery (SSRF, 서버 측 요청 변조)

Capital One (캐피털 원) 유출 사고 (2019)

공격자가 WAF 설정 오류와 SSRF를 악용하여 AWS 인스턴스 메타데이터 서비스(IMDS)에 접근, 1억 명의 고객 정보를 탈취했습니다.

대응: AWS IMDSv2

세션 토큰 기반의 IMDSv2를 강제하여 단순 GET 요청을 통한 정보 유출을 원천 차단합니다.

# 메타데이터 보안 강화 명령어
$ aws ec2 modify-instance-metadata-options \
  --instance-id i-1234567890abcdef0 \
  --http-tokens required
