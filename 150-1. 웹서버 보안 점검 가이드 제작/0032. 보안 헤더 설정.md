## 3.2 보안 헤더 (Security Headers) 설정

---

### **1. HTTP 보안 헤더의 정의와 방어 메커니즘**

**HTTP Security Headers (HTTP 보안 헤더)**는 웹 서버가 브라우저에 보내는 응답 메시지에 포함되어, 브라우저가 페이지 콘텐츠를 처리하는 방식을 지시하는 설정입니다. 이를 통해 애플리케이션 코드 수정 없이도 **XSS (Cross-Site Scripting, 교차 사이트 스크립팅)**나 **Clickjacking (클릭재킹, 사용자 클릭 유도 및 가로채기)**과 같은 클라이언트 측 공격을 효과적으로 방어할 수 있습니다.



* **보안 목표**: 브라우저의 보안 기능을 활성화하여 신뢰할 수 없는 스크립트 실행을 차단하고 데이터 유출을 방지합니다.
* **작동 원리**: 웹 서버 응답 헤더에 특정 정책을 삽입하면, 브라우저는 해당 정책을 해석하여 위반되는 동작을 차단합니다.

> **설명**: 웹 애플리케이션 자체의 취약점이 있더라도, 보안 헤더라는 '최후의 방어선'을 통해 공격자가 실질적인 피해를 입히는 것을 물리적으로 차단하는 전략입니다.

---

### **2. 핵심 보안 헤더 및 설정값**

가장 필수적인 3가지 보안 헤더의 역할과 권장 설정값은 다음과 같습니다.

* **CSP (Content Security Policy, 콘텐츠 보안 정책)**
    * **역할**: 브라우저가 로드할 수 있는 스크립트, 이미지, 스타일시트의 출처(Domain)를 제한합니다.
    * **방어**: **XSS (Cross-Site Scripting)** 공격을 방어하는 가장 강력한 수단입니다.
* **X-Frame-Options**
    * **역할**: 해당 웹 페이지가 `<iframe>` 등에 삽입되어 표시되는 것을 허용할지 결정합니다.
    * **방어**: 사용자 모르게 투명한 레이어를 클릭하게 만드는 **Clickjacking (클릭재킹)** 공격을 차단합니다.
* **X-Content-Type-Options**
    * **역할**: 서버가 보낸 **MIME (Multipurpose Internet Mail Extensions, 파일 형식 식별 표준)** 타입을 브라우저가 임의로 해석하지 못하도록 강제합니다.
    * **방어**: 텍스트 파일을 실행 파일로 오인하여 악성 코드가 실행되는 **MIME Sniffing (MIME 스니핑)** 공격을 방지합니다.



---

### **3. 웹 서버 적용 가이드 (Nginx 기준)**

Nginx 설정 파일의 `server` 블록에 아래 내용을 추가하여 보안 헤더를 일괄 적용합니다.

```bash
# Nginx 가상 호스트 설정 파일 수정
sudo vi /etc/nginx/sites-available/default

# 보안 헤더 추가 설정 예시
server {
    # 1. 클릭재킹 방어 (동일 도메인 내에서만 프레임 허용)
    add_header X-Frame-Options "SAMEORIGIN" always;

    # 2. MIME 스니핑 방지
    add_header X-Content-Type-Options "nosniff" always;

    # 3. CSP 설정 (자신과 특정 도메인의 스크립트만 허용)
    add_header Content-Security-Policy "default-src 'self'; script-src 'self' https://trusted.com;" always;

    # 4. 리퍼러 정책 설정 (개인정보 보호를 위해 최소 정보만 전송)
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
}

# 설정 문법 검사 및 서비스 재시작
sudo nginx -t && sudo systemctl restart nginx
```

> **설명**: `always` 옵션은 에러 페이지(404, 500 등)를 포함한 모든 응답에 해당 헤더를 누락 없이 포함하도록 보장합니다.

---

### **4. 설정 결과 검증 방법**

명령행 도구인 `curl`을 사용하여 서버 응답 헤더가 정상적으로 포함되었는지 확인합니다.

```bash
# 특정 URL의 응답 헤더만 출력하여 확인
curl -I https://example.com
```

### **[AWS 환경 보안 가이드 및 과금 안내]**

개별 웹 서버 설정 외에도 인프라 앞단에서 보안 헤더를 일괄 관리할 수 있습니다.

* **Amazon CloudFront (아마존 클라우드프론트 - 응답 헤더 정책)**
    * **Response Headers Policies** 기능을 사용하면 오리진 서버의 수정 없이 엣지 로케이션에서 보안 헤더를 자동으로 추가할 수 있습니다.
    * **과금 안내**: 정책 설정 및 적용 자체는 **무료**이나, CloudFront를 통해 전송되는 데이터량(Data Transfer Out)에 따라 비용이 청구됩니다. [공식 요금 페이지](https://aws.amazon.com/ko/cloudfront/pricing/)
* **AWS WAF (웹 애플리케이션 방화벽)**
    * 특정 조건에 맞는 요청에 대해 보안 헤더를 조작하거나 비정상적인 헤더를 가진 요청을 차단할 수 있습니다.
    * **과금 안내**: 웹 ACL 및 규칙 수, 요청 건수에 따라 과금됩니다.

---

**상세 설명 주석**
1. **XSS (Cross-Site Scripting)**: 웹 사이트에 악성 스크립트를 삽입하여 사용자의 쿠키를 탈취하거나 세션을 가로채는 공격 기법입니다.
2. **Clickjacking**: 웹 사용자를 속여서 보이지 않는 버튼이나 링크를 클릭하게 하여 비정상적인 동작(비밀번호 변경, 송금 등)을 유도하는 공격입니다.
3. **MIME Sniffing**: 브라우저가 파일의 확장자 대신 파일의 실제 내용을 분석하여 데이터 타입을 결정하는 기능으로, 이를 악용해 악성 코드가 포함된 파일을 실행시킬 수 있습니다.
4. **Referrer-Policy**: 사용자가 다른 사이트로 이동할 때 이전 페이지의 주소(Referrer) 정보를 얼마나 보낼지 제어하여 개인정보 노출을 방지합니다.
5. **nosniff**: 브라우저가 서버가 지정한 Content-Type을 따르도록 강제하여 MIME 스니핑을 허용하지 않겠다는 선언입니다.
