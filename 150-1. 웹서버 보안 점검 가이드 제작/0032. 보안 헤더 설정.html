1. HTTP 보안 헤더의 정의와 방어 메커니즘

HTTP Security Headers (HTTP 보안 헤더)는 웹 서버가 브라우저에 보내는 응답 메시지에 포함되어, 브라우저가 페이지 콘텐츠를 처리하는 방식을 지시하는 설정입니다. 이를 통해 애플리케이션 코드 수정 없이도 XSS (Cross-Site Scripting, 교차 사이트 스크립팅)나 Clickjacking (클릭재킹, 사용자 클릭 유도 및 가로채기)과 같은 클라이언트 측 공격을 효과적으로 방어할 수 있습니다.

정책 주입 (Policy Insertion)
HTTP 응답 + 보안 헤더

정책 해석 및
실행 제약 (Enforcement)

보안 목표 (Security Goal)

브라우저의 보안 기능을 활성화하여 신뢰할 수 없는 스크립트 실행을 차단하고 데이터 유출을 방지합니다.

작동 원리 (Workflow)

웹 서버 응답 헤더에 특정 정책을 삽입하면, 브라우저는 해당 정책을 해석하여 위반되는 동작을 차단합니다.

설명: 웹 애플리케이션 자체의 취약점이 있더라도, 보안 헤더라는 '최후의 방어선'을 통해 공격자가 실질적인 피해를 입히는 것을 물리적으로 차단하는 전략입니다.

XSS (Cross-Site Scripting, 교차 사이트 스크립팅): 웹 사이트에 악성 스크립트를 삽입하여 사용자의 쿠키를 탈취하거나 세션을 가로채는 공격 기법입니다.
Clickjacking (클릭재킹): 웹 사용자를 속여서 보이지 않는 버튼이나 링크를 클릭하게 하여 비정상적인 동작을 유도하는 공격입니다.

2. 핵심 보안 헤더 및 설정값

가장 필수적인 3가지 보안 헤더의 역할과 권장 설정값은 다음과 같습니다.

[도식: Core Security Headers Comparison (핵심 헤더 비교 분석)]

CSP
Content Security Policy

스크립트, 이미지, 스타일시트의 출처(Domain)를 제한하여 비정상적인 자원 로드를 차단합니다.

Anti-XSS Defense
X-Frame-Options
Iframe Control

해당 웹 페이지가 <iframe> 등에 삽입되어 표시되는 것을 허용할지 결정합니다.

Anti-Clickjacking
X-Content-Type
MIME Enforcement

서버가 보낸 MIME 타입을 브라우저가 임의로 해석하지 못하게 강제합니다.

Anti-MIME Sniffing
MIME (Multipurpose Internet Mail Extensions): 파일의 형식을 식별하기 위한 표준입니다. (예: text/html, image/png)
MIME Sniffing (MIME 스니핑): 브라우저가 파일 확장자 대신 내용을 분석하여 데이터 타입을 결정하는 기능으로, 보안상 위험할 수 있습니다.

3. 웹 서버 적용 가이드 (Nginx 기준)

Nginx 설정 파일의 server 블록에 아래 내용을 추가하여 보안 헤더를 일괄 적용합니다.

Nginx 보안 헤더 구현 (Implementation)

# 1. Nginx 가상 호스트 설정 파일 편집
sudo vi /etc/nginx/sites-available/default # 사이트별 설정 파일을 텍스트 에디터로 엽니다.

server { # 서버 가상 호스트 설정을 시작합니다.
# 1. 클릭재킹 방어 (동일 도메인 내에서만 프레임 허용)
add_header X-Frame-Options "SAMEORIGIN" always; # 외부 사이트에서의 iframe 로드를 차단합니다.

# 2. MIME 스니핑 방지 (서버 지정 타입 강제)
add_header X-Content-Type-Options "nosniff" always; # 브라우저가 파일 확장자를 임의로 해석하지 못하게 합니다.

# 3. CSP 설정 (자신과 특정 도메인의 스크립트만 허용)
add_header Content-Security-Policy "default-src 'self'; script-src 'self' https://trusted.com;" always; # 화이트리스트 기반 자원 로드 정책을 적용합니다.

# 4. 리퍼러 정책 설정 (개인정보 보호 강화)
add_header Referrer-Policy "strict-origin-when-cross-origin" always; # 외부 이동 시 도메인 정보 유출을 제한합니다.
} # 서버 설정을 마칩니다.

# 설정 검증 및 적용
sudo nginx -t && sudo systemctl restart nginx # 문법 체크 후 이상이 없으면 엔진엑스를 재시작합니다.

설명: always 옵션은 에러 페이지(404, 500 등)를 포함한 모든 응답에 해당 헤더를 누락 없이 포함하도록 보장합니다.

Referrer-Policy (리퍼러 정책): 사용자가 링크를 클릭하여 이동할 때, 이전 페이지의 주소를 얼마나 보낼지 제어하는 정책입니다.
nosniff (노스니프): 브라우저가 서버가 지정한 콘텐츠 형식을 엄격히 따르도록 강제하여 스니핑을 방지하는 설정값입니다.

4. 설정 결과 검증 방법

명령행 도구인 curl을 사용하여 서버 응답 헤더가 정상적으로 포함되었는지 확인합니다.

[도식: Response Header Validation Workflow (응답 헤더 검증 흐름)]

curl -I 요청 (Request)
HTTP/1.1 200 OK
CSP: 'self'...
X-Frame-Options: SAMEORIGIN
# 특정 도메인의 응답 헤더 정보만 출력하여 확인
$ curl -I https://example.com # -I 옵션은 메시지 본문을 제외한 헤더부만 요청합니다.
curl (Client URL, 클라이언트 URL 도구): 서버와 데이터를 주고받기 위한 명령행 기반 전송 도구로, 통신 헤더 분석 및 디버깅에 필수적입니다.

Web ACL (Web Access Control List): WAF에서 정의한 규칙 모음으로, 애플리케이션 자원을 보호하는 기준이 됩니다.
