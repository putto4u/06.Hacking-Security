보안 헤더 (Security Headers) 설정
클라이언트 측 공격 방어 및 클릭재킹 대응

SCORE: 10.0

System Architect Pro

1. 아키텍처 개요 (Architecture Overview)

HTTP Security Headers (HTTP 보안 헤더)는 웹 서버가 브라우저에 보내는 응답 메시지에 포함되어, 브라우저가 페이지 콘텐츠를 처리하는 방식을 지시하는 설정입니다. 이를 통해 애플리케이션 코드 수정 없이도 XSS (Cross-Site Scripting, 교차 사이트 스크립팅)나 Clickjacking (클릭재킹, 사용자 클릭 유도 및 가로채기)과 같은 클라이언트 측 공격을 효과적으로 방어할 수 있습니다.

보안 목표 (Security Goal)

브라우저의 보안 기능을 활성화하여 신뢰할 수 없는 스크립트 실행을 차단하고 데이터 유출을 방지합니다.

작동 원리 (Mechanism)

웹 서버 응답 헤더에 특정 정책을 삽입하면, 브라우저는 해당 정책을 해석하여 위반되는 동작을 차단합니다.

[도식: 보안 헤더 방어 프로세스]

Web Server
응답 헤더(Response Header) 주입
Security Policy
Web Browser
정책 해석 및 위반 동작 차단

웹 애플리케이션 자체의 취약점이 있더라도, 보안 헤더라는 '최후의 방어선'을 통해 실질적인 피해를 물리적으로 차단합니다.

2. 핵심 보안 헤더 및 설정 (Security Components)

CSP

콘텐츠 보안 정책

  • 역할: 리소스(스크립트 등) 로드 출처 제한
  • 방어: XSS 공격의 가장 강력한 방어 수단
XFO

프레임 옵션 설정

  • 역할: iframe 렌더링 허용 여부 결정
  • 방어: 클릭재킹 (Clickjacking) 공격 차단
XCTO

MIME 타입 보호

  • 역할: 브라우저의 임의 형식 해석 금지
  • 방어: MIME Sniffing (MIME 스니핑) 공격 방지

3. 실증 및 구현 (Implementation)

STEP 01

Nginx 보안 헤더 일괄 적용

Nginx 설정 파일의 server 블록에 아래 내용을 추가하여 보안 정책을 선언합니다.

Nginx Config
# 1. 클릭재킹 방어 (동일 도메인 내에서만 프레임 허용)
add_header X-Frame-Options "SAMEORIGIN" always;

# 2. MIME 스니핑 방지 (nosniff 강제)
add_header X-Content-Type-Options "nosniff" always;

# 3. CSP 설정 (자신과 특정 도메인의 스크립트만 허용)
add_header Content-Security-Policy "default-src 'self'; script-src 'self' https://trusted.com;" always;

# 4. 리퍼러 정책 설정 (개인정보 보호 최적화)
add_header Referrer-Policy "strict-origin-when-cross-origin" always;

※ always 옵션은 에러 페이지(404, 500 등)를 포함한 모든 응답에 헤더를 강제 포함시킵니다.

STEP 02

응답 헤더 검증 (Verification)

Console
$ curl -I https://example.com
HTTP/1.1 200 OK
X-Frame-Options: SAMEORIGIN
X-Content-Type-Options: nosniff
Content-Security-Policy: default-src 'self';...

4. 클릭재킹 심층 분석 (Clickjacking Deep Dive)

클릭재킹 (Clickjacking)은 '클릭(Click)'과 '하이재킹(Hijacking, 납치)'의 합성어로, 사용자가 인지하는 웹 페이지 위에 투명한 레이어를 겹쳐서 사용자가 의도하지 않은 클릭을 유도하는 공격 기법입니다. 이를 UI Redressing (User Interface Redressing, 사용자 인터페이스 재구성)이라고도 부릅니다.

[도식: 클릭재킹 작동 원리]

[가시적 레이어]
Visible to User
[비가시적 레이어 (iframe)]
Opacity: 0 (Hidden)

공격자는 CSS opacity: 0 속성을 사용하여 민감한 실제 페이지를 투명하게 겹칩니다.

주요 공격 사례

구분 시나리오
SNS 하이재킹 쿠폰 버튼 위에 '좋아요' 버튼을 겹쳐 팔로워 조작
계정 정보 수정 재생 버튼 위에 '계정 탈퇴' 확인 버튼을 겹쳐 장애 유발
이커머스 결제 할인 광고 위에 '즉시 구매' 버튼 배치하여 오결제 유도

핵심 방어 대책

  • X-Frame-Options

    DENY(전면 차단) 또는 SAMEORIGIN(자사 허용) 설정을 통해 렌더링 제어

  • CSP (frame-ancestors)

    'none' 또는 특정 도메인 지정을 통해 부모 프레임의 권한을 정교하게 제한

5. 아키텍트 인사이트 (Architect Insight)

AWS 환경 인프라 기반 보안 관리 및 과금

Amazon CloudFront (응답 헤더 정책)

Response Headers Policies를 사용하면 오리진 수정 없이 엣지에서 보안 헤더를 추가합니다. 정책 적용은 무료이나, 데이터 전송량(DTO)에 따른 비용이 발생합니다.

AWS WAF (웹 방화벽)

특정 조건에 맞춰 보안 헤더를 조작하거나 차단합니다. Web ACL 및 규칙 수, 요청 건수에 따라 정교하게 과금됩니다.

MIME Sniffing (MIME 스니핑) 주의

브라우저가 파일의 확장자 대신 실제 내용을 분석하여 데이터 타입을 결정하는 기능입니다. 이를 악용해 악성 코드가 포함된 텍스트 파일을 실행 파일로 오인하게 할 수 있으므로 nosniff 설정이 필수적입니다.

Referrer-Policy (개인정보 보호)

다른 사이트로 이동할 때 이전 페이지 주소(Referrer) 정보를 얼마나 보낼지 제어합니다. strict-origin-when-cross-origin은 보안과 편의성의 적절한 균형점입니다.
