## 2.1 정보 노출 방지 및 배너 숨기기

---

### **1. Banner Grabbing (배너 그래빙)의 위험성**

**Banner Grabbing (배너 그래빙, 서비스 정보 수집)**은 공격자가 대상 서버에 접속하여 서버 소프트웨어의 이름, 버전, 운영체제 정보를 알아내는 기법입니다. 공격자는 이 정보를 바탕으로 해당 버전의 알려진 취약점(Exploit)을 검색하여 공격 대상을 구체화합니다.



* **Footprinting (풋프린팅, 사전 조사)**: 공격의 첫 번째 단계로, 시스템의 구조와 소프트웨어 정보를 수집하는 행위입니다.
* **보안 목표**: 서버가 응답 헤더(Response Header)나 에러 페이지에서 자신의 정체를 드러내지 않도록 설정하여 공격자의 분석을 방해합니다.

> **설명**: 웹 서버의 이름과 구체적인 버전(예: Nginx 1.18.0)이 노출되면, 해당 버전에만 존재하는 특정 취약점을 이용한 정밀 공격에 무방비로 노출될 수 있습니다.

---

### **2. Nginx 서버 정보 은폐 설정**

Nginx는 기본적으로 응답 헤더의 `Server` 필드에 버전 정보를 포함합니다. 이를 비활성화하여 소프트웨어 이름만 노출되거나 아예 숨기도록 설정합니다.

* **Server Tokens 비활성화**
    `/etc/nginx/nginx.conf` 파일의 `http` 블록 내에서 설정합니다.

```bash
# Nginx 설정 파일 수정
sudo vi /etc/nginx/nginx.conf

# http { ... } 블록 안에 아래 내용 추가 또는 수정
server_tokens off;

# 설정 적용 후 서비스 재시작
sudo nginx -t
sudo systemctl restart nginx
```

> **설명**: `server_tokens off;` 설정은 응답 헤더에서 버전 번호를 제거합니다. 더 나아가 `more_set_headers` 모듈(추가 설치 필요)을 사용하면 'Server' 필드 자체를 임의의 문자열로 변경할 수도 있습니다.

---

### **3. Apache 서버 정보 은폐 설정**

Apache HTTP Server 역시 기본적으로 상세 정보를 노출합니다. `ServerTokens`와 `ServerSignature` 옵션을 조정하여 이를 차단합니다.

* **보안 강화 설정**
    주 설정 파일(예: `/etc/apache2/conf-enabled/security.conf`)에서 수정합니다.

```bash
# Apache 보안 설정 파일 수정
sudo vi /etc/apache2/conf-enabled/security.conf

# 서버 정보 노출 최소화 (Product 이름만 노출)
ServerTokens Prod

# 에러 페이지 하단의 서버 정보 서명 비활성화
ServerSignature Off

# 서비스 재시작
sudo systemctl restart apache2
```

> **설명**: **ServerSignature (서버 시그니처, 서버 서명)**는 서버가 생성하는 에러 페이지 하단에 서버 버전과 호스트명을 자동으로 삽입하는 기능입니다. 이를 끄는 것은 정보 유출 방지의 기본입니다.

---

### **4. 사용자 정의 에러 페이지 (Custom Error Pages) 적용**

서버 내부 오류(500)나 페이지 없음(404) 발생 시, 기본 에러 페이지 대신 관리자가 만든 페이지를 보여주어 시스템 내부 경로 정보나 서버 특성이 노출되는 것을 방지합니다.

```bash
# Nginx 예시: 404 에러 시 특정 HTML 파일로 대체
error_page 404 /custom_404.html;
location = /custom_404.html {
    root /var/www/html/errors;
    internal;
}
```

### **[AWS 환경 보안 가이드 및 과금 안내]**

인프라 수준에서 정보를 숨기거나 헤더를 조작하려면 AWS의 엣지(Edge) 서비스를 활용하는 것이 매우 효과적입니다.

* **AWS WAF (웹 애플리케이션 방화벽) - Custom Response**
    * 서버가 응답하기 전에 WAF에서 응답 헤더를 조작하여 `Server` 정보를 강제로 삭제하거나 변경할 수 있습니다.
    * **과금 안내**: AWS WAF는 웹 ACL당 **월 $5.00**, 규칙당 **월 $1.00**, 요청 100만 건당 **$0.60**의 비용이 발생합니다. [공식 요금 페이지](https://aws.amazon.com/ko/waf/pricing/)
* **Amazon CloudFront (아마존 클라우드프론트, 콘텐츠 전송 네트워크)**
    * **CloudFront Function** 또는 **Lambda@Edge**를 사용하여 원본 서버(Origin)의 헤더를 가공하고 보안 헤더를 추가할 수 있습니다.
    * **과금 안내**: CloudFront Function은 실행 100만 건당 약 **$0.10**의 저렴한 비용으로 헤더 조작이 가능합니다.

**상세 설명 주석**
1. **Banner Grabbing**: 텔넷(Telnet)이나 넷캣(Netcat) 등을 이용해 웹 서버 포트(80, 443)에 연결하여 서버가 보내는 초기 응답 메시지를 가로채 정보를 얻는 행위입니다.
2. **Server Tokens**: Nginx에서 서버의 버전 정보를 응답 헤더에 포함할지 여부를 결정하는 지시어입니다.
3. **Response Header**: 웹 서버가 브라우저의 요청에 응답할 때 보내는 메타데이터로, 여기에 서버의 소프트웨어 정보가 포함됩니다.
4. **Internal 지시어**: Nginx에서 외부 사용자가 직접 해당 경로(예: /custom_404.html)로 접속하는 것을 막고, 서버 내부적인 리다이렉션으로만 접근 가능하게 제한하는 설정입니다.

5. 웹 서버(Web Server)에서 발생하는 오류는 보안 관리 측면에서 매우 민감한 정보입니다. 기본적으로 제공되는 에러 페이지는 서버의 종류, 버전, 내부 경로 등을 노출할 수 있는 **Information Leakage(정보 유출, 정보 누출)**의 원인이 됩니다.

## 1. 웹 서버별 기본 에러 페이지 구성 및 경로

대부분의 웹 서버는 오류 발생 시 별도의 외부 파일을 불러오기보다, 서버 바이너리에 내장된 **Hardcoded(하드코딩, 코드 내에 고정됨)** 메시지를 먼저 출력하거나, 배포판에서 제공하는 기본 **HTML(HyperText Markup Language, 하이퍼텍스트 마크업 언어)** 파일을 참조합니다.

| 웹 서버 종류 | 기본 에러 페이지 처리 방식 및 경로 |
| :--- | :--- |
| **Apache HTTP Server** (아파치 HTTP 서버) | 별도 설정이 없으면 내장된 메시지를 출력합니다. 하지만 패키지 설치 시 `/usr/share/apache2/error/` (Ubuntu) 또는 `/var/www/error/` (CentOS) 경로의 다국어 지원 파일을 사용하기도 합니다. |
| **Nginx** (엔진엑스) | 기본적으로 `/usr/share/nginx/html/` 경로에 `50x.html` 파일을 생성하여 관리합니다. 파일이 없으면 서버 내부의 단순한 HTML 코드를 응답합니다. |
| **IIS** (Internet Information Services, 인터넷 정보 서비스) | 윈도우 서버의 경우 `C:\inetpub\custerr\` 디렉터리 하위에 상태 코드별(404, 500 등) 폴더를 만들어 관리합니다. |

## 2. 보안 위협: 기본 에러 페이지를 숨겨야 하는 이유

기본 에러 페이지를 그대로 사용할 경우, 공격자는 다음과 같은 정보를 획득하여 **Reconnaissance(정찰, 정보 수집)** 단계에서 활용할 수 있습니다.

* **서버 소프트웨어 및 버전**: "Apache/2.4.41 (Ubuntu)"와 같은 문구가 하단에 노출됩니다.
* **물리적 경로**: 스택 트레이스(Stack Trace) 등이 노출될 경우 서버의 절대 경로가 드러납니다.
* **백엔드 기술 스택**: 에러 메시지의 형식을 통해 **PHP(Hypertext Preprocessor, PHP: 하이퍼텍스트 전처리기)**나 **Python(파이썬)** 등의 사용 여부를 추측할 수 있습니다.

## 3. 에러 페이지 커스터마이징 구조

관리자가 정의한 페이지로 리다이렉트(Redirect)하는 과정은 웹 서버 설정 파일(예: `httpd.conf`, `nginx.conf`)에서 정의됩니다.

> **[클라이언트 요청]** (잘못된 URL 접근)
>       ↓
> **[웹 서버 판단]** (404 Not Found 발생)
>       ↓
> **[설정 확인]** (ErrorDocument 또는 error_page 지시어 확인)
>       ↓
> **[사용자 정의 페이지 응답]** (관리자가 만든 `/errors/404.html` 출력)

## 4. 실전 가이드: 설정 파일 수정 예시

각 서버의 설정 파일에 다음과 같은 지시어를 추가하여 기본 페이지 대신 안전한 페이지를 보여주어야 합니다.

### ① Apache (아파치)
`.htaccess` 또는 설정 파일(`sites-available/default`)에 작성합니다.
* `ErrorDocument 404 /custom_404.html`
* `ErrorDocument 500 /custom_500.html`

### ② Nginx (엔진엑스)
`server` 블록 내부에 작성합니다.
* `error_page 404 /custom_404.html;`
* `location = /custom_404.html { root /var/www/html; internal; }`
