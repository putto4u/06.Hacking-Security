3.1 SSL/TLS (Secure Sockets Layer / Transport Layer Security, 보안 소켓 계층 / 전송 계층 보안) 적용

1. SSL/TLS의 정의와 보안의 3요소

SSL/TLS (Secure Sockets Layer / Transport Layer Security, 보안 소켓 계층 / 전송 계층 보안)는 웹 브라우저와 웹 서버 간의 데이터를 암호화하여 전송하는 표준 보안 프로토콜입니다. 현대 웹 환경에서는 기존의 보안 취약점이 발견된 SSL을 대신하여 보다 안전한 TLS 프로토콜을 사용하며, 이를 통해 통신의 기밀성, 무결성, 가용성을 보장합니다.

  • Confidentiality (기밀성): 스니핑(Sniffing) 공격으로부터 데이터를 보호하기 위해 모든 통신 내용을 암호화합니다.
  • Integrity (무결성): 전송 중인 데이터가 변조되지 않았음을 보장합니다.
  • Authentication (인증): 접속하려는 서버가 신뢰할 수 있는 기관으로부터 인증받은 실제 서버임을 증명합니다.

설명: Handshake (핸드셰이크, 악수) 과정을 통해 클라이언트와 서버는 서로의 신원을 확인하고 통신에 사용할 암호화 키를 안전하게 공유합니다.

2. SSL/TLS 핸드셰이크 프로세스

데이터 전송 전, 클라이언트와 서버는 암호화 알고리즘과 키를 결정하는 복잡한 협상 과정을 거칩니다.

  1. Client Hello: 클라이언트가 지원 가능한 암호화 방식(Cipher Suite) 목록을 보냅니다.
  2. Server Hello & Certificate: 서버가 암호화 방식을 선택하고 자신의 Digital Certificate (디지털 인증서, 전자 서명된 보안 인증서)를 보냅니다.
  3. Key Exchange: 공개키 암호화 방식을 이용하여 데이터를 암호화할 대칭키(Session Key)를 생성 및 공유합니다.
  4. Cipher Spec Change: 이후 모든 통신은 공유된 대칭키로 암호화됨을 선언합니다.

설명: Cipher Suite (사이퍼 스위트, 암호화 알고리즘 세트)는 프로토콜 버전, 키 교환 방식, 대칭 암호화 방식, 해시 함수 등의 조합을 의미합니다.

3. 웹 서버 보안 설정 (Nginx 기준)

취약한 암호화 알고리즘(DES, RC4, MD5)을 제거하고 TLS 1.2 이상 버전만 사용하도록 설정하는 것이 보안의 핵심입니다.

# Nginx 가상 호스트 설정 파일 수정
sudo vi /etc/nginx/sites-available/default

# SSL 설정 블록 예시
server {
    listen 443 ssl;
    server_name example.com;

    # 인증서 경로 지정
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # 최신 보안 프로토콜만 허용 (TLS 1.2, 1.3)
    ssl_protocols TLSv1.2 TLSv1.3;

    # 취약한 암호화 알고리즘 제외 및 강력한 사이퍼 스위트 설정
    ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';
    ssl_prefer_server_ciphers on;
}

설명: CA (Certificate Authority, 인증 기관)로부터 발급받은 인증서와 개인키를 서버에 설치하여 HTTPS 서비스를 활성화합니다.

<아파치2의 경우> /etc/apache2/sites-available/default-ssl.conf

<VirtualHost *:443>
    ServerName example.com
    DocumentRoot /var/www/html

    # SSL 엔진 활성화
    SSLEngine on

    # 인증서 경로 지정
    SSLCertificateFile    /etc/letsencrypt/live/example.com/fullchain.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/example.com/privkey.pem

    # [보안 핵심 1] 최신 프로토콜만 허용 (TLS 1.2, 1.3)
    # 기존의 SSLv2, SSLv3, TLSv1, TLSv1.1은 명시적으로 제외(-) 처리
    SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1

    # [보안 핵심 2] 강력한 사이퍼 스위트 설정 (취약한 RC4, DES, MD5 제외)
    SSLCipherSuite ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384

    # [보안 핵심 3] 서버가 정의한 암호화 순서를 우선함 (Client 무시)
    SSLHonorCipherOrder on

    # 보안 헤더 설정 (HSTS 활성화 권장)
    Header always set Strict-Transport-Security "max-age=63072000"
</VirtualHost>

### **4. HSTS (HTTP Strict Transport Security) 적용**

**HSTS (HTTP Strict Transport Security, HTTP 엄격한 전송 보안)**는 브라우저가 해당 사이트에 접속할 때 항상 HTTPS 프로토콜만 사용하도록 강제하는 보안 헤더입니다. 이를 통해 HTTP를 HTTPS로 리다이렉트하는 과정에서 발생할 수 있는 **SSL Stripping (SSL 스트리핑, HTTPS를 HTTP로 하향 유도하는 공격)**을 원천 차단합니다.

```bash
# Nginx 설정에 HSTS 헤더 추가
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

[AWS 환경 보안 가이드 및 과금 안내]

AWS 환경에서는 서버 내부 설정보다 관리형 서비스를 이용하는 것이 가용성과 보안성 측면에서 유리합니다.

  • AWS Certificate Manager (ACM, AWS 인증서 관리자)
    • Public SSL/TLS 인증서를 무료로 발급 및 갱신해 줍니다. (ELB, CloudFront와 연동 필수)
    • 과금 안내: 인증서 발급 자체는 무료이나, 사설 CA(Private CA)를 구축하여 사용할 경우 월별 고정 비용 및 발급 비용이 발생합니다. 공식 요금 페이지
  • Application Load Balancer (ALB, 애플리케이션 로드 밸런서)
    • SSL Termination (SSL 종단) 기능을 제공하여 서버의 암복호화 부하를 대신 처리합니다.
    • 과금 안내: ALB는 시간당 사용 요금과 LCU (Load Balancer Capacity Units, 로드 밸런서 용량 단위) 소비량에 따라 과금됩니다.

Next Step: 보안 헤더 설정

상세 설명 주석 1. SSL Termination: 로드 밸런서에서 SSL 복호화를 수행하고, 내부 서버와는 HTTP로 통신하여 서버 자원을 절약하는 방식입니다. 2. Let's Encrypt: 무료로 SSL 인증서를 발급해주는 비영리 인증 기관으로, 90일마다 자동 갱신 설정을 통해 주로 사용됩니다. 3. Forward Secrecy (전방 향후 보안): 서버의 개인키가 유출되더라도 과거에 통신했던 암호화된 데이터는 복구할 수 없도록 보장하는 성질입니다. 4. max-age: HSTS 설정이 브라우저에 기억될 기간을 초 단위로 지정합니다. (31536000초 = 1년)
