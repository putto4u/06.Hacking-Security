1. SSL/TLS의 정의와 보안의 3요소 (Security Triad)

SSL/TLS (Secure Sockets Layer / Transport Layer Security, 보안 소켓 계층 / 전송 계층 보안)는 웹 브라우저와 웹 서버 간의 데이터를 암호화하여 전송하는 표준 보안 프로토콜입니다. 현대 웹 환경에서는 기존의 보안 취약점이 발견된 SSL을 대신하여 보다 안전한 TLS 프로토콜을 사용하며, 이를 통해 통신의 기밀성, 무결성, 가용성을 보장합니다.

[도식: Security Triad in SSL/TLS (보안 3요소 매커니즘)]

Confidentiality (기밀성)

스니핑(Sniffing) 공격 방어
모든 통신 데이터 암호화

Integrity (무결성)

데이터 변조 여부 보장
메시지 인증 코드(MAC) 활용

Authentication (인증)

서버 신뢰성 증명
디지털 인증서 기반 신원 확인

설명: Handshake (핸드셰이크, 악수) 과정을 통해 클라이언트와 서버는 서로의 신원을 확인하고 통신에 사용할 암호화 키를 안전하게 공유합니다.

Annotate 01. Handshake: 클라이언트와 서버가 암호화 통신을 시작하기 전, 프로토콜 버전과 암호화 알고리즘을 협상하는 절차입니다.

2. SSL/TLS 핸드셰이크 프로세스 (Handshake Process)

데이터 전송 전, 클라이언트와 서버는 암호화 알고리즘과 키를 결정하는 복잡한 협상 과정을 거칩니다.

[도식: SSL/TLS Handshake Sequential Flow (단계별 협상 과정)]

Step 01. Client Hello
Cipher Suite List, Random Number
Step 02. Server Hello & Certificate
Selected Cipher, Digital Certificate
Step 03. Key Exchange
Pre-master Secret (Encrypted)
Step 04. Cipher Spec Change
Encrypted Session Start

설명: Cipher Suite (사이퍼 스위트, 암호화 알고리즘 세트)는 프로토콜 버전, 키 교환 방식, 대칭 암호화 방식, 해시 함수 등의 조합을 의미합니다.

Annotate 02. Digital Certificate: 인증 기관(CA)이 서버의 공개키에 전자 서명을 하여 신뢰성을 보증하는 문서입니다.
Annotate 03. Cipher Suite: 보안 통신을 위해 사용되는 알고리즘들의 묶음으로, 협상 과정을 통해 결정됩니다.

3. 웹 서버 보안 설정 (Nginx 기준)

취약한 암호화 알고리즘(DES, RC4, MD5)을 제거하고 TLS 1.2 이상 버전만 사용하도록 설정하는 것이 보안의 핵심입니다.

[도식: Nginx SSL Configuration Architecture (보안 설정 아키텍처)]

Security Protocol
TLS 1.2, TLS 1.3 Only

취약한 하위 버전(SSL 3.0, TLS 1.0) 차단

Key Management
fullchain.pem / privkey.pem

CA 인증서 및 개인키 경로 지정

Cipher Suites
AES-GCM, SHA-384...

강력한 대칭 암호화 및 해시 함수 선택

Nginx SSL 설정 구현 (Implementation)

# Nginx 가상 호스트 설정 파일 수정
sudo vi /etc/nginx/sites-available/default # 기본 설정 파일을 편집기로 엽니다.

server { # 가상 서버 설정을 시작합니다.
listen 443 ssl; # 443번 포트에서 SSL 통신을 대기합니다.
server_name example.com; # 서비스할 도메인 주소를 지정합니다.

# SSL 인증서 및 개인키 경로 지정
ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem; # 공개키가 포함된 인증서 경로입니다.
ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem; # 서버의 개인키 파일 경로입니다.

# 보안 프로토콜 및 알고리즘 최적화
ssl_protocols TLSv1.2 TLSv1.3; # 보안이 검증된 최신 TLS 버전만 허용합니다.
ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256...'; # 강력한 암호화 알고리즘 목록을 지정합니다.
ssl_prefer_server_ciphers on; # 클라이언트보다 서버의 암호화 설정 우선순위를 따릅니다.
} # 서버 블록 설정을 종료합니다.

설명: CA (Certificate Authority, 인증 기관)로부터 발급받은 인증서와 개인키를 서버에 설치하여 HTTPS 서비스를 활성화합니다.

Annotate 04. CA (Certificate Authority): 디지털 인증서를 발급하고 신원을 보증하는 공신력 있는 비영리/영리 기관입니다.
Annotate 05. Let's Encrypt: 무료로 SSL 인증서를 발급해주는 대표적인 비영리 인증 기관입니다.

4. HSTS (HTTP Strict Transport Security) 적용

HSTS (HTTP Strict Transport Security, HTTP 엄격한 전송 보안)는 브라우저가 해당 사이트에 접속할 때 항상 HTTPS 프로토콜만 사용하도록 강제하는 보안 헤더입니다.

[도식: Defense against SSL Stripping via HSTS (보안 강제 매커니즘)]

Web Browser
Attack Scenario
HTTP Downshift (Stripping)
HSTS Enabled
Enforce HTTPS Always
Secure Server

HSTS 헤더 설정 (Implementation)

# Nginx 설정에 HSTS 보안 헤더 추가
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always; # 1년간 하위 도메인 포함 모든 접속을 HTTPS로 강제합니다.
Annotate 06. SSL Stripping: HTTPS 연결을 강제로 HTTP로 내려서 중간에 데이터를 가로채는 공격 기법입니다.
Annotate 07. max-age: HSTS 설정이 브라우저에 기억될 기간을 초 단위로 지정합니다. (31536000초 = 1년)
