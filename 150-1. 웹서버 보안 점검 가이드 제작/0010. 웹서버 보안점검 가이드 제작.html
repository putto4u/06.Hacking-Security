Module 01. 인프라 정찰 및 정보 수집 (Infrastructure Reconnaissance)

보안점검의 첫 단계는 공격자의 시선에서 대상의 정보를 파악하는 Reconnaissance (Reconnaissance, 정찰)입니다. 서버가 외부에 노출하는 최소한의 단서를 통해 전체 아키텍처를 유추하는 기법을 학습합니다.

  • Banner Grabbing (Banner Grabbing, 서버 정보 탈취)

    HTTP Response Header 정보를 분석하여 웹서버의 종류와 상세 버전을 식별합니다.

  • OS Fingerprinting (OS Fingerprinting, 운영체제 식별)

    TTL(Time To Live, 패킷 생존 시간) 값을 분석하여 대상 호스트의 OS를 추정합니다.

[그림 1] 정보 수집 및 정찰 흐름도

※ TTL (Time To Live, 패킷 생존 시간): 데이터 패킷이 네트워크 장비를 거칠 때마다 1씩 감소하는 값입니다. Linux는 보통 64, Windows는 128에서 시작합니다.

{/* Module 02: 패치 및 취약점 관리 */}

Module 02. 취약점 관리 및 패치 전략 (Vulnerability & Patch Management)

알려진 취약점을 이용한 공격을 방어하기 위해 CVE (Common Vulnerabilities and Exposures, 알려진 보안 취약점 목록) 기반의 정기적인 패치 프로세스가 필수적입니다.

패치 가용성 점검

웹 서버 엔진(Apache, Nginx)과 라이브러리의 보안 업데이트 현황을 점검하고, 지원 종료(EOL, End of Life) 여부를 확인합니다.

서비스 영향도 분석

보안 패치 적용 시 발생할 수 있는 서비스 중단이나 기능 충돌을 최소화하기 위한 사전 테스트 절차를 수립합니다.

[그림 2] 보안 패치 및 취약점 관리 라이프사이클

※ EOL (End Of Life, 기술 지원 종료): 소프트웨어 개발사가 더 이상 보안 업데이트나 기술 지원을 제공하지 않는 상태로, 취약점에 매우 무방비해집니다.

{/* Module 03: 설정 파일 점검 */}

Module 03. 서버 설정 하드닝 (Server Configuration Hardening)

웹서버의 성능 최적화만큼 중요한 것이 Hardening (Hardening, 시스템 요새화) 설정입니다. 부적절한 기본값을 수정하여 잠재적인 공격 경로를 차단하는 실무 기법을 다룹니다.

Directory Listing 비활성화

디렉토리 목록 노출 취약점은 인덱스 파일이 없는 폴더의 파일 목록을 노출시켜 소스코드나 백업 파일 탈취를 유발합니다.

HTTP Method 제한

GET/POST 외에 PUT, DELETE, TRACE 등 불필요한 HTTP Method를 허용할 경우 정보 유출이 발생할 수 있습니다.

[그림 3] 불필요한 기능 제거를 통한 공격 표면 축소

{/* Module 04: 암호화 통신 점검 */}

Module 04. 암호화 통신 및 인증서 관리 (SSL/TLS Encryption)

데이터 전송 구간의 기밀성을 보장하기 위해 SSL (Secure Sockets Layer, 보안 소켓 계층) / TLS (Transport Layer Security, 전송 계층 보안) 설정의 견고함을 점검합니다.

Cipher Suite (암호 알고리즘 조합) 최적화

취약한 암호화 알고리즘(RC4, DES 등) 사용을 배제하고 강도가 높은 알고리즘만을 허용하도록 설정합니다.

TLS Version (전송 계층 버전) 관리

보안상 결함이 발견된 TLS 1.0, 1.1 버전을 비활성화하고 TLS 1.2 이상 버전 사용을 강제합니다.

[그림 4] SSL/TLS 기반 안전한 데이터 전송 아키텍처

{/* Module 05: HTTP 보안 헤더 */}

Module 05. 보안 헤더 및 브라우저 방어 (HTTP Security Headers)

클라이언트 측 공격(Client-side Attack)을 방지하기 위해 웹 응답에 추가적인 Security Headers (보안 헤더)를 적용하여 브라우저의 보안 정책을 강화합니다.

  • CSP
    Content Security Policy (Content Security Policy, 콘텐츠 보안 정책)

    허용된 출처의 리소스만 실행하도록 제한하여 XSS 공격의 실효성을 제거합니다.

  • HSTS
    HTTP Strict Transport Security (HTTP Strict Transport Security, 강력한 전송 보안)

    모든 요청을 자동으로 HTTPS로 전환하여 암호화되지 않은 접근을 차단합니다.

[보안 헤더 설정 예시]

Strict-Transport-Security: max-age=31536000; includeSubDomains

Content-Security-Policy: default-src 'self'

X-Frame-Options: DENY

X-Content-Type-Options: nosniff

{/* Module 06: 파일 및 디렉토리 권한 */}

Module 06. 파일 시스템 권한 및 무결성 (File System Permission)

웹 서비스 계정이 시스템의 다른 영역에 접근하지 못하도록 Privilege Separation (Privilege Separation, 권한 분리)를 철저히 점검합니다.

Owner Separation: 웹 프로세스(www-data)와 소스코드 소유자(예: admin)를 분리하여 서비스 계정이 파일을 수정할 수 없도록 방어합니다.

Execution Prevention: `/tmp`나 업로드 디렉토리처럼 파일이 생성되는 위치에서 실행 권한(x)을 제거하여 악성 스크립트 실행을 방지합니다.

{/* Module 07: 로그 아키텍처 및 SIEM */}

Module 07. 로그 관리 및 통합 모니터링 (Security Logging & SIEM)

침해 사고 발생 시 원인 규명을 위해 웹 서버의 Log (Log, 활동 기록)를 중앙 집중식으로 보존하고 분석하는 체계를 구축합니다.

통합 로그 저장 (Remote Logging)

웹서버가 탈취되더라도 로그를 삭제할 수 없도록 외부 로그 서버(Syslog, ELK 등)로 실시간 전송합니다.

이상 징후 탐지 (Alerting)

특정 시간 내 반복되는 4xx 에러나 이상 트래픽 발생 시 즉시 보안 담당자에게 경고(Alert)를 발송합니다.

Module 08. 점검 보고서 작성 및 법적 요건 (Audit Reporting & Compliance)

보안점검의 결과물은 Audit Report (Audit Report, 점검 보고서)로 문서화되어야 하며, 국내외 보안 인증 요건을 충족해야 합니다.

  • ISMS-P

  • PCI-DSS
