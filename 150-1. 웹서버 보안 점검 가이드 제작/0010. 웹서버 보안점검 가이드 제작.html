Module 01. 인프라 정찰 및 정보 수집 (Infrastructure Reconnaissance)

보안점검의 첫 단계는 공격자의 시선에서 대상의 정보를 파악하는 Reconnaissance (Reconnaissance, 정찰)입니다. 서버가 외부에 노출하는 최소한의 단서를 통해 전체 아키텍처를 유추하는 기법을 학습합니다.

  • Banner Grabbing (Banner Grabbing, 서버 정보 탈취)

    HTTP Response Header 정보를 분석하여 웹서버의 종류와 상세 버전을 식별합니다.

  • OS Fingerprinting (OS Fingerprinting, 운영체제 식별)

    TTL (Time To Live, 패킷 생존 시간) 값을 분석하여 대상 호스트의 OS를 추정합니다.

Module 02. 취약점 관리 및 패치 전략 (Vulnerability & Patch Management)

알려진 취약점을 이용한 공격을 방어하기 위해 CVE (Common Vulnerabilities and Exposures, 알려진 보안 취약점 목록) 기반의 정기적인 패치 프로세스가 필수적입니다.

가용성 점검 (Availability Check)

웹 서버 엔진(Apache, Nginx)과 라이브러리의 보안 업데이트 현황을 점검하고, 지원 종료(EOL, End of Life) 여부를 확인합니다.

영향도 분석 (Impact Analysis)

보안 패치 적용 시 발생할 수 있는 서비스 중단이나 기능 충돌을 최소화하기 위한 사전 테스트 절차를 수립합니다.

Module 03. 서버 설정 하드닝 (Server Configuration Hardening)

웹서버의 성능 최적화만큼 중요한 것이 Hardening (Hardening, 시스템 요새화) 설정입니다. 부적절한 기본값을 수정하여 잠재적인 공격 경로를 차단하는 실무 기법을 학습합니다.

디렉토리 목록 노출 방지 (Disable Directory Listing)

Directory Listing 취약점은 인덱스 파일이 없는 폴더의 파일 목록을 노출시켜 소스코드나 백업 파일 탈취를 유발합니다.

HTTP 메소드 제한 (Restrict HTTP Methods)

GET/POST 외에 PUT, DELETE, TRACE 등 불필요한 HTTP Method (HTTP Method, 요청 방식)를 허용할 경우 정보 유출이 발생할 수 있습니다.

Module 04. 암호화 통신 및 인증서 관리 (SSL/TLS Encryption)

데이터 전송 구간의 기밀성을 보장하기 위해 SSL (Secure Sockets Layer, 보안 소켓 계층) / TLS (Transport Layer Security, 전송 계층 보안) 설정의 견고함을 점검합니다.

Cipher Suite (암호 알고리즘 조합) 최적화

취약한 암호화 알고리즘(RC4, DES 등) 사용을 배제하고 강도가 높은 알고리즘만을 허용하도록 설정합니다.

TLS Version (전송 계층 버전) 관리

보안상 결함이 발견된 TLS 1.0, 1.1 버전을 비활성화하고 TLS 1.2 이상 버전 사용을 강제합니다.

Module 05. 보안 헤더 및 브라우저 보호 (HTTP Security Headers)

클라이언트 측 공격을 방지하기 위해 웹 응답에 추가적인 Security Headers (보안 헤더)를 적용하여 브라우저의 보안 정책을 강제합니다.

CSP (Content Security Policy, 콘텐츠 보안 정책)

신뢰된 출처의 리소스만 실행하도록 브라우저에 지시하여 XSS 공격의 실효성을 제거합니다.

HSTS (Strict Transport Security, 강력한 전송 보안)

모든 접근을 HTTPS로 강제 전환하여 세션 하이재킹과 중간자 공격을 방어합니다.
