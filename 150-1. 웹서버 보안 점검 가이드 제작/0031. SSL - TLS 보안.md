## 3.1 SSL/TLS (Secure Sockets Layer / Transport Layer Security, 보안 소켓 계층 / 전송 계층 보안) 적용

---

### **1. SSL/TLS의 정의와 보안의 3요소**

**SSL/TLS (Secure Sockets Layer / Transport Layer Security, 보안 소켓 계층 / 전송 계층 보안)**는 웹 브라우저와 웹 서버 간의 데이터를 암호화하여 전송하는 표준 보안 프로토콜입니다. 현대 웹 환경에서는 기존의 보안 취약점이 발견된 SSL을 대신하여 보다 안전한 TLS 프로토콜을 사용하며, 이를 통해 통신의 기밀성, 무결성, 가용성을 보장합니다.



* **Confidentiality (기밀성)**: 스니핑(Sniffing) 공격으로부터 데이터를 보호하기 위해 모든 통신 내용을 암호화합니다.
* **Integrity (무결성)**: 전송 중인 데이터가 변조되지 않았음을 보장합니다.
* **Authentication (인증)**: 접속하려는 서버가 신뢰할 수 있는 기관으로부터 인증받은 실제 서버임을 증명합니다.

> **설명**: **Handshake (핸드셰이크, 악수)** 과정을 통해 클라이언트와 서버는 서로의 신원을 확인하고 통신에 사용할 암호화 키를 안전하게 공유합니다.

---

### **2. SSL/TLS 핸드셰이크 프로세스**

데이터 전송 전, 클라이언트와 서버는 암호화 알고리즘과 키를 결정하는 복잡한 협상 과정을 거칩니다.



1. **Client Hello**: 클라이언트가 지원 가능한 암호화 방식(Cipher Suite) 목록을 보냅니다.
2. **Server Hello & Certificate**: 서버가 암호화 방식을 선택하고 자신의 **Digital Certificate (디지털 인증서, 전자 서명된 보안 인증서)**를 보냅니다.
3. **Key Exchange**: 공개키 암호화 방식을 이용하여 데이터를 암호화할 대칭키(Session Key)를 생성 및 공유합니다.
4. **Cipher Spec Change**: 이후 모든 통신은 공유된 대칭키로 암호화됨을 선언합니다.

> **설명**: **Cipher Suite (사이퍼 스위트, 암호화 알고리즘 세트)**는 프로토콜 버전, 키 교환 방식, 대칭 암호화 방식, 해시 함수 등의 조합을 의미합니다.

---

### **3. 웹 서버 보안 설정 (Nginx 기준)**

취약한 암호화 알고리즘(DES, RC4, MD5)을 제거하고 TLS 1.2 이상 버전만 사용하도록 설정하는 것이 보안의 핵심입니다.

```bash
# Nginx 가상 호스트 설정 파일 수정
sudo vi /etc/nginx/sites-available/default

# SSL 설정 블록 예시
server {
    listen 443 ssl;
    server_name example.com;

    # 인증서 경로 지정
    ssl_certificate /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # 최신 보안 프로토콜만 허용 (TLS 1.2, 1.3)
    ssl_protocols TLSv1.2 TLSv1.3;

    # 취약한 암호화 알고리즘 제외 및 강력한 사이퍼 스위트 설정
    ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';
    ssl_prefer_server_ciphers on;
}
```

> **설명**: **CA (Certificate Authority, 인증 기관)**로부터 발급받은 인증서와 개인키를 서버에 설치하여 HTTPS 서비스를 활성화합니다.

<아파치2의 경우>  /etc/apache2/sites-available/default-ssl.conf
``` 
<VirtualHost *:443>
    ServerName example.com
    DocumentRoot /var/www/html

    # SSL 엔진 활성화
    SSLEngine on

    # 인증서 경로 지정
    SSLCertificateFile    /etc/letsencrypt/live/example.com/fullchain.pem
    SSLCertificateKeyFile /etc/letsencrypt/live/example.com/privkey.pem

    # [보안 핵심 1] 최신 프로토콜만 허용 (TLS 1.2, 1.3)
    # 기존의 SSLv2, SSLv3, TLSv1, TLSv1.1은 명시적으로 제외(-) 처리
    SSLProtocol all -SSLv3 -TLSv1 -TLSv1.1

    # [보안 핵심 2] 강력한 사이퍼 스위트 설정 (취약한 RC4, DES, MD5 제외)
    SSLCipherSuite ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384
    
    # [보안 핵심 3] 서버가 정의한 암호화 순서를 우선함 (Client 무시)
    SSLHonorCipherOrder on

    # 보안 헤더 설정 (HSTS 활성화 권장)
    Header always set Strict-Transport-Security "max-age=63072000"
</VirtualHost>
```
---

### **4. HSTS (HTTP Strict Transport Security) 적용**

**HSTS (HTTP Strict Transport Security, HTTP 엄격한 전송 보안)**는 브라우저가 해당 사이트에 접속할 때 항상 HTTPS 프로토콜만 사용하도록 강제하는 보안 헤더입니다. 이를 통해 HTTP를 HTTPS로 리다이렉트하는 과정에서 발생할 수 있는 **SSL Stripping (SSL 스트리핑, HTTPS를 HTTP로 하향 유도하는 공격)**을 원천 차단합니다.

```bash
# Nginx 설정에 HSTS 헤더 추가
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

## 1. Apache (아파치) HTTPS 설정 절차

**Apache**는 모듈 방식이므로 SSL 모듈을 먼저 활성화해야 합니다.

### [1단계] SSL 모듈 및 가상 호스트 활성화
```bash
# 1. SSL 모듈(Module) 활성화
sudo a2enmod ssl

# 2. SSL 전용 기본 가상 호스트(Virtual Host) 설정 파일 활성화
sudo a2ensite default-ssl
```

### [2단계] 설정 파일 수정
* **파일 경로**: `/etc/apache2/sites-available/default-ssl.conf`
* **주요 수정 내용**:
```apache
<VirtualHost *:443>
    ServerName example.com
    DocumentRoot /var/www/html

    # SSL 엔진 가동
    SSLEngine on

    # 인증서 및 개인키 경로 (실제 발급받은 파일 경로로 수정)
    SSLCertificateFile    /etc/ssl/certs/example.com.crt
    SSLCertificateKeyFile /etc/ssl/private/example.com.key
    
    # 중간 인증서가 있는 경우 주석 해제 후 경로 지정
    # SSLCertificateChainFile /etc/ssl/certs/chain.pem
</VirtualHost>
```

### [3단계] 실행 및 확인
```bash
# 1. 설정 문법 검사 (Syntax Check)
sudo apache2ctl configtest

# 2. 결과가 'Syntax OK'라면 서비스 재시작
sudo systemctl restart apache2

# 3. 443 포트 리슨(Listen) 확인
ss -tulpn | grep :443
```

## 2. Nginx (엔진엑스) HTTPS 설정 절차

**Nginx**는 설정 파일 내에 직접 SSL 지시어를 추가하는 방식입니다.

### [1단계] 설정 파일 수정
* **파일 경로**: `/etc/nginx/sites-available/default` (또는 `/etc/nginx/conf.d/default.conf`)
* **주요 수정 내용**: 기존 `server` 블록을 수정하거나 443용 블록을 추가합니다.
```nginx
server {
    listen 443 ssl;
    server_name example.com;

    # 인증서 및 개인키 경로 지정
    ssl_certificate     /etc/nginx/ssl/example.com.crt;
    ssl_certificate_key /etc/nginx/ssl/example.com.key;

    # 보안 프로토콜 및 사이퍼 스위트(Cipher Suite) 설정
    ssl_protocols       TLSv1.2 TLSv1.3;
    ssl_ciphers         HIGH:!aNULL:!MD5;

    location / {
        root   /var/www/html;
        index  index.html index.htm;
    }
}
```

### [2단계] 실행 및 확인
```bash
# 1. 설정 파일 오류 검사
sudo nginx -t

# 2. 결과가 'successful'이라면 설정 반영 (Restart보다 안전한 Reload 권장)
sudo systemctl reload nginx

# 3. 브라우저 확인 또는 컬(curl) 명령어로 확인
curl -I https://localhost
```

## 4. 백엔드(Backend) 서버에도 HTTPS가 필요한가?

일반적으로 **백엔드 서버**(예: Node.js, Python/Django, Spring Boot 등)는 웹 서버(Nginx/Apache) 뒷단에 위치합니다. 이를 **Reverse Proxy(리버스 프록시)** 구조라고 합니다.

### [보안 아키텍처 관점]
* **외부 구간 (Client ↔ Web Server)**: 반드시 **HTTPS**가 필요합니다. 공인망을 통과하므로 데이터 암호화가 필수입니다.
* **내부 구간 (Web Server ↔ Backend)**: 보통 **HTTP**로 통신합니다. 동일한 서버 내부나 신뢰할 수 있는 사설망(VPC) 내에 있다면 암호화 오버헤드를 줄이기 위해 HTTP를 사용해도 무방합니다.
* **예외 상황**: 금융권이나 고도의 보안이 필요한 환경에서는 **End-to-End Encryption(종단 간 암호화)**을 위해 내부 구간까지 HTTPS를 적용하기도 합니다.



> **기술 보충 설명**
> **Reverse Proxy(리버스 프록시)**는 클라이언트의 요청을 대신 받아 백엔드 서버로 전달하는 대리인 역할을 수행합니다. 이때 웹 서버가 SSL 처리를 전담하는 것을 **SSL Termination(SSL 종단)**이라고 부릅니다.

**상세 설명 주석**
1. **SSL Termination**: 로드 밸런서에서 SSL 복호화를 수행하고, 내부 서버와는 HTTP로 통신하여 서버 자원을 절약하는 방식입니다.
2. **Let's Encrypt**: 무료로 SSL 인증서를 발급해주는 비영리 인증 기관으로, 90일마다 자동 갱신 설정을 통해 주로 사용됩니다.
3. **Forward Secrecy (전방 향후 보안)**: 서버의 개인키가 유출되더라도 과거에 통신했던 암호화된 데이터는 복구할 수 없도록 보장하는 성질입니다.
4. **max-age**: HSTS 설정이 브라우저에 기억될 기간을 초 단위로 지정합니다. (31536000초 = 1년)

다만, 공인 기관에서 **받지(발급받지)** 않고 본인이 직접 만든 **Self-signed Certificate(자가 서명 인증서)**를 사용하여 기술적으로 HTTPS를 가동하는 것은 가능합니다.

## 1. 인증서 없이 HTTPS가 불가능한 이유

**HTTPS**의 핵심은 **TLS(Transport Layer Security, 전송 계층 보안)** 프로토콜을 통한 암호화 통신입니다. 이 암호화 과정에서 서버의 정체를 확인하고 암호화 키를 교환하기 위한 **Digital Certificate(디지털 인증서)**가 반드시 필요합니다.

### 자가 서명 인증서 (Self-signed Certificate)의 특징
외부 기관(CA)을 거치지 않고 서버 관리자가 직접 생성한 인증서입니다.
* **장점**: 비용이 들지 않고 즉시 생성이 가능하여 내부 테스트용으로 적합합니다.
* **단점**: 웹 브라우저가 해당 인증서를 신뢰할 수 없으므로 접속 시 **"주의 요함"** 또는 **"연결이 비공개로 설정되어 있지 않습니다"**라는 경고창을 띄웁니다.

| 구분 | 자가 서명 인증서 (Self-signed) | 공인 인증서 (CA-signed) |
| :--- | :--- | :--- |
| **신뢰도** | 낮음 (브라우저 경고 발생) | 높음 (자물쇠 아이콘 표시) |
| **용도** | 개인 개발, 내부 망 테스트 | 운영 서비스, 쇼핑몰, 기업 웹사이트 |
| **비용** | 무료 | 무료(Let's Encrypt) ~ 유료 |
| **암호화 여부** | **동일하게 암호화됨** | **동일하게 암호화됨** |

## 2. CSR (Certificate Signing Request)의 이해

**CSR(인증서 서명 요청)**은 공인 기관(CA)에 인증서를 신청하기 위해 보내는 **신청서 데이터**입니다. 이 파일에는 서버의 **Public Key(공개키)** 정보와 도메인 정보가 포함되어 있습니다.

### CSR 생성 과정 및 포함 정보
1.  서버에서 **Private Key(개인키)**를 먼저 생성합니다.
2.  이 개인키를 바탕으로 **CSR** 파일을 생성합니다.
3.  **CSR** 파일만 공인 기관(CA)에 전달하고, **Private Key**는 절대로 외부로 유출해서는 안 됩니다.



> **CSR 포함 주요 항목**
> * **Common Name (CN)**: 인증서를 적용할 도메인 주소 (예: `example.com`)
> * **Organization (O)**: 회사 또는 단체명
> * **Country (C)**: 국가 코드 (대한민국은 `KR`)
> * **Public Key**: 암호화에 사용될 공개키 정보

---

## 3. 실전: OpenSSL을 이용한 CSR 생성 명령어

실제 서버에서 인증서를 신청하기 위해 실행하는 표준 명령어입니다.

```bash
# 1. 개인키(Private Key)와 CSR을 동시에 생성
openssl req -new -newkey rsa:2048 -nodes -keyout example.com.key -out example.com.csr
```

* **-newkey rsa:2048**: 2048비트 크기의 **RSA**(Rivest-Shamir-Adleman, 공개키 암호화 알고리즘) 키 쌍을 생성합니다.
* **-nodes**: 개인키를 암호화하지 않고 저장합니다 (서버 재시작 시 비밀번호 입력 생략을 위함).
* **-keyout**: 생성될 개인키 파일명입니다.
* **-out**: 생성될 **CSR** 파일명입니다.

## 4. 인증서 발급 및 적용 논리 흐름

인증서를 준비하여 서비스를 가동하기까지의 단계별 블록 구조입니다.

> **[1. 키 생성]** (서버에서 Private Key 생성)
>       ↓
> **[2. CSR 작성]** (도메인 정보 입력 및 신청서 생성)
>       ↓
> **[3. CA 전송]** (CSR을 기관에 보내고 심사 후 인증서 수령)

웹 사이트 보안을 완성하기 위해서는 사용자가 **HTTP(Hypertext Transfer Protocol, 하이퍼텍스트 전송 프로토콜)**로 접속하더라도 자동으로 암호화된 **HTTPS(Hypertext Transfer Protocol Secure, 보안 하이퍼텍스트 전송 프로토콜)**로 전환해주는 **강제 리다이렉션(Forced Redirection, 통신 경로 재지정)** 설정이 필수적입니다.

---

## 1. Apache (아파치) 리다이렉션 설정

**Apache**에서는 크게 두 가지 방법을 사용합니다. `VirtualHost` 설정을 직접 수정하거나 `.htaccess` 파일을 활용합니다.

### ① VirtualHost (가상 호스트) 설정을 이용한 방법 (권장)
80번 포트(HTTP)로 들어오는 모든 요청을 443번 포트(HTTPS)로 보냅니다.
* **파일 경로**: `/etc/apache2/sites-available/000-default.conf`

```apache
<VirtualHost *:80>
    ServerName example.com
    # [핵심] 모든 요청을 HTTPS로 영구 이동(301) 시킴
    Redirect permanent / https://example.com/
</VirtualHost>
```

### ② mod_rewrite (리라이트 모듈)를 이용한 방법
특정 조건에 따라 더 세밀한 제어가 필요할 때 사용합니다.
```apache
RewriteEngine On
# HTTPS가 꺼져 있는(off) 경우에만 동작
RewriteCond %{HTTPS} off
# 모든 경로(.*)를 HTTPS 주소로 변경하여 전달
RewriteRule ^(.*)$ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
```



---

## 2. Nginx (엔진엑스) 리다이렉션 설정

**Nginx**는 구조가 단순하며, `return` 지시어를 사용하여 매우 빠르게 처리합니다.

### ① Server 블록 설정을 이용한 방법
80번 포트 전용 `server` 블록을 생성하여 리다이렉트합니다.
* **파일 경로**: `/etc/nginx/sites-available/default`

```nginx
server {
    listen 80;
    server_name example.com;

    # [핵심] 301 상태 코드를 반환하며 HTTPS 주소로 연결
    return 301 https://$host$request_uri;
}
```

> **기술 보충 설명**
> **301 Moved Permanently (301 영구 이동)**: 브라우저와 검색 엔진에 해당 페이지의 주소가 영구적으로 변경되었음을 알리는 **HTTP Status Code (HTTP 상태 코드)**입니다. SEO(Search Engine Optimization, 검색 엔진 최적화) 점수를 유지하는 데 유리합니다.

---

## 3. 보안 강화: HSTS (HTTP Strict Transport Security)

리다이렉션만으로는 첫 번째 HTTP 접속 시 발생하는 **Man-in-the-Middle Attack (MITM, 중간자 공격)**을 완벽히 막을 수 없습니다. 이를 보완하기 위해 **HSTS (HTTP Strict Transport Security, HTTP 엄격한 전송 보안)**를 설정합니다.

* **동작 원리**: 브라우저에게 "이 사이트는 앞으로 일정 기간(max-age) 동안 무조건 HTTPS로만 접속하라"는 지시를 내립니다.
* **Apache 설정**: `Header always set Strict-Transport-Security "max-age=31536000; includeSubDomains"`
* **Nginx 설정**: `add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;`



---

## 4. 리다이렉션 동작 구조 도식화

사용자의 요청이 보안 연결로 전환되는 단계별 블록 구조입니다.

> **[1. 클라이언트 접속]** ([http://example.com](http://example.com))
>       ↓
> **[2. 웹 서버 수신]** (80번 포트에서 요청 확인)
>       ↓
> **[3. 리다이렉트 응답]** (HTTP 301 응답 + https:// 주소 전달)
>       ↓
> **[4. 클라이언트 재접속]** ([https://example.com](https://example.com)으로 자동 요청)
>       ↓
> **[5. 보안 연결 확립]** (TLS 핸드쉐이크 후 암호화 통신 시작)

## 5. 설정 확인 방법

명령행 도구인 **curl (컬, 데이터 전송 도구)**을 사용하여 정상적으로 리다이렉트되는지 확인합니다.

```bash
# -I 옵션은 헤더 정보만 출력함
curl -I http://example.com
```
* **확인 사항**: 출력 결과의 첫 줄에 `HTTP/1.1 301 Moved Permanently`가 나타나고, `Location` 항목에 `https://...` 주소가 포함되어야 합니다.
