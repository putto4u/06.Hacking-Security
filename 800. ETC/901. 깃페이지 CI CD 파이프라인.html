GitHub Pages CI/CD 파이프라인 구축

정적 웹사이트 배포 자동화를 위한 엔터프라이즈급 아키텍처 가이드

개요 및 핵심 개념

현대의 웹 개발 환경에서는 소스 코드의 변경 사항을 사용자가 접근 가능한 실제 서버에 즉각적으로 반영하는 것이 필수적입니다. 이를 위해 CI/CD (Continuous Integration/Continuous Deployment, 지속적 통합 및 지속적 배포) 파이프라인(Pipeline, 데이터 처리 단계의 연속적인 흐름)을 구축합니다.

본 문서에서는 깃허브(GitHub)에서 제공하는 워크플로 자동화 도구인 GitHub Actions (깃허브 액션)을 활용하여, 저장소(Repository, 코드와 파일이 저장되는 공간)에 코드가 푸시(Push, 로컬의 변경 사항을 원격 저장소로 업로드하는 작업)될 때마다 웹페이지를 자동으로 빌드하고 배포하는 과정을 설계합니다.

비용 및 과금 안내 (GitHub Actions Pricing)

GitHub Actions는 퍼블릭 저장소(Public Repository)에서는 완전 무료로 무제한 제공됩니다. 단, 프라이빗 저장소(Private Repository)의 경우 Free 티어 기준으로 월 2,000분의 실행 시간(Linux 환경 기준)이 무료로 제공되며, 이를 초과할 경우 분당 과금(약 $0.008/분)이 발생하므로 유의해야 합니다.

배포 아키텍처 및 워크플로 작성

GitHub Pages 배포를 위한 설정은 프로젝트 최상위 경로의 .github/workflows/ 폴더 내에 YAML (YAML Ain't Markup Language, 직관적인 데이터 직렬화 언어) 파일로 정의됩니다. 아래는 범용적으로 사용 가능한 엔터프라이즈 표준 배포 스크립트입니다.

.github/workflows/deploy-pages.yml 
name: Deploy static content to Pages

# 1. 트리거(Trigger) 설정
on:
  push:
    branches: ["main"] # main 브랜치에 푸시될 때 실행

# 2. 권한(Permissions) 명시
permissions:
  contents: read
  pages: write      # GitHub Pages 배포 권한
  id-token: write   # OIDC(OpenID Connect) 인증 토큰 발급 권한

# 3. 동시성(Concurrency) 제어
concurrency:
  group: "pages"
  cancel-in-progress: false # 진행 중인 배포를 취소하지 않음

# 4. 작업(Jobs) 정의
jobs:
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest # 깃허브 호스팅 러너(가상 서버) 사용
    
    steps:
      - name: 저장소 체크아웃 (Checkout)
        uses: actions/checkout@v4

      - name: GitHub Pages 설정 (Setup Pages)
        uses: actions/configure-pages@v4

      - name: 아티팩트 업로드 (Upload artifact)
        uses: actions/upload-pages-artifact@v3
        with:
          path: '.' # 배포할 정적 파일이 있는 디렉토리 (최상위)

      - name: Pages에 배포 (Deploy to GitHub Pages)
        id: deployment
        uses: actions/deploy-pages@v4
[주석 1] 동시성(Concurrency) 제어: 여러 번의 커밋(Commit)이 연속으로 발생할 경우, 이전 배포 작업과 충돌하는 것을 방지하기 위해 하나의 작업 그룹(Group)으로 묶어 순차적으로 처리하게 만드는 고급 제어 기법입니다.

실전 팁 및 자주 발생하는 실수 (Troubleshooting)

1. 403 Forbidden 권한 오류 발생

가장 빈번하게 발생하는 에러입니다. 봇(Bot)이 GitHub Pages 환경에 접근하거나 토큰을 발행할 권한이 없을 때 발생합니다.

해결 방법: 저장소의 Settings > Actions > General에서 최하단 Workflow permissions 항목을 확인하세요. 기본값이 'Read repository contents permission'으로 되어 있다면, 이를 'Read and write permissions'로 변경해야 합니다.

2. GitHub Pages Source 설정 누락

액션은 성공적으로 끝났는데 웹페이지가 갱신되지 않거나 404 에러가 뜨는 경우입니다.

해결 방법: 저장소의 Settings > Pages 메뉴로 이동합니다. Build and deployment 섹션의 Source 드롭다운 메뉴가 'Deploy from a branch'가 아닌 'GitHub Actions'로 선택되어 있는지 반드시 확인해야 합니다.
