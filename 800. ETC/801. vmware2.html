1. 하이퍼바이저 아키텍처 개요 (Hypervisor Architecture Overview)

가상화를 구현하기 위해서는 물리적 하드웨어와 가상의 환경을 연결해주는 매개체가 필요합니다. VMware Workstation Pro는 호스트 OS (Operating System, 운영 체제) 위에서 애플리케이션처럼 동작하는 Type 2 하이퍼바이저 (Hosted Hypervisor, 호스트형 가상화 머신 모니터)에 해당합니다. 이를 통해 물리적 하드웨어 자원을 논리적으로 분할하여 여러 개의 VM (Virtual Machine, 가상 머신)을 독립적으로 구동할 수 있습니다.

[도식: Type 2 하이퍼바이저 아키텍처]

물리적 하드웨어
(Host Hardware)
호스트 OS
(Windows/Linux)

(VMware Workstation)
게스트 OS
(가상 머신 내부)
* OS (Operating System, 운영 체제): 컴퓨터 하드웨어와 소프트웨어 자원을 관리하고 사용자와 컴퓨터 간의 인터페이스를 제공하는 시스템 소프트웨어입니다.
* VM (Virtual Machine, 가상 머신): 물리적 컴퓨터와 동일하게 작동하도록 소프트웨어로 구현된 논리적 컴퓨터 환경입니다.

2. 실습 인프라 구성 요소 (Infrastructure Components)

NODE 01

물리적 호스트 (Host PC)

  • 운영 체제: Windows 10 또는 11 (64-bit 필수)
  • 가상화 지원: BIOS/UEFI에서 Intel VT-x 또는 AMD-V 활성화
  • 여유 공간: 최소 50GB 이상의 SSD (Solid State Drive) 권장
NODE 02

가상 머신 타겟 (Guest VM)

  • 하이퍼바이저: VMware Workstation Pro 17
  • 설치 미디어: 대상 OS의 ISO 이미지 파일 준비
  • 라이선스: Personal Use (개인 목적 무료 사용 정책 적용)

8. 게스트 운영 체제 설치 매체 제어 (Guest OS Installation Media)

가상 머신에 운영 체제(Windows, Linux 등)를 설치할 때 사용할 ISO 이미지 파일의 삽입 시점을 결정하는 매우 중요한 단계입니다. 자동화 기능의 함정을 피하기 위해 정확한 옵션을 선택해야 합니다.

[도식: OS 설치 옵션 3가지 비교]

Installer disc (물리적 CD/DVD 롬 사용 - 현재 거의 안씀)
Installer disc image file (iso) - 선택 금지 (Easy Install 강제 작동)
I will install the operating system later. (운영 체제를 나중에 설치하겠습니다.) - 선택 필수

아키텍트 인사이트: Easy Install(간편 설치) 방지

두 번째 옵션으로 ISO 파일을 지금 바로 지정하게 되면, VMware의 Easy Install 기능이 개입합니다. 이 기능은 사용자 몰래 디스크 파티션을 마음대로 나누고 관리자 비밀번호를 임의로 설정하는 등 IT 엔지니어가 학습해야 할 핵심 OS 설치 과정을 모두 건너뛰어 버립니다. 따라서 반드시 세 번째 옵션(나중에 설치)을 선택하여 빈 깡통 상태의 하드웨어를 먼저 조립하는 것이 원칙입니다.

9. 대상 운영 체제 제품군 선택 (Target OS Family Selection)

앞서 '나중에 설치'를 선택했으므로, 하이퍼바이저에게 "앞으로 이런 운영 체제를 설치할 예정이니 하드웨어 프리셋을 최적화해라"라고 알려주는 단계입니다. Guest Operating System (게스트 운영 체제) 라디오 버튼에서 대분류를 선택하고, Version(버전) 드롭다운에서 소분류를 선택합니다.

Microsoft Windows 계열

  • Windows 10 x64
  • Windows 11 x64 TPM 암호화 요구됨
  • Windows Server 2019 / 2022

※ 대부분의 윈도우는 x64 (64비트) 버전을 선택해야 최신 환경과 호환됩니다.

Linux 계열

  • Ubuntu 64-bit
  • CentOS 7/8 64-bit
  • Red Hat Enterprise Linux (RHEL) 8/9

※ Amazon Linux 2023이나 Rocky Linux의 경우 비슷한 커널인 RHEL 9 64-bit 등을 선택하면 무난하게 호환됩니다.

10. 가상 머신 식별자 및 스토리지 경로 지정 (VM Identifier & Storage Path)

가상 머신의 이름을 부여하고, 이 컴퓨터의 물리적 파일들이 저장될 실제 호스트 PC의 디렉토리 경로(Location)를 설정하는 단계입니다. 인프라 운영에서 용량 관리와 디스크 I/O 최적화를 위해 경로 변경이 필수적입니다.

Path Configuration
Virtual machine name: Win10_LAB_01
Location:
C:\Users\Username\Documents\Virtual Machines\Win10_LAB_01
D:\VM_Data\Win10_LAB_01 권장 변경값

C 드라이브 용량 고갈 경고

Location의 기본값은 C 드라이브의 '내 문서' 폴더입니다. 가상 머신은 한 대당 최소 20GB ~ 60GB 이상의 거대한 단일 파일들을 생성합니다. C 드라이브에 가상 머신을 계속 만들면 운영 체제의 여유 공간이 부족해져 호스트 PC 전체가 다운될 수 있습니다. 반드시 용량이 넉넉한 D 드라이브나 외장 SSD 전용 폴더로 위치를 변경([Browse...] 버튼 클릭)해야 합니다.

11. 시스템 펌웨어 인터페이스 설정 (System Firmware Interface)

가상 머신의 마더보드에 해당하는 Firmware Type (펌웨어 유형)을 선택합니다. 운영 체제가 부팅되기 전에 하드웨어를 초기화하고 OS 커널을 메모리에 로드하는 핵심 인터페이스입니다.

[도식: BIOS vs UEFI 펌웨어 선택 기준]

Legacy BIOS

16비트 레거시 기반의 구형 부팅 방식입니다.

적용 대상: Windows 7 이하, 구형 Linux (CentOS 6 등)
실무 권장
UEFI

64비트 아키텍처, 대용량 디스크, 빠른 부팅을 지원하는 최신 표준입니다.

적용 대상: Windows 10/11, 최신 Linux (Ubuntu 22+, RHEL 9+)
* UEFI (Unified Extensible Firmware Interface, 통일 확장 펌웨어 인터페이스): 기존 BIOS (Basic Input/Output System, 기본 입출력 시스템)의 한계를 극복하기 위해 설계된 최신 컴퓨터 부팅 인터페이스입니다.
* Secure Boot (보안 부팅): UEFI의 기능 중 하나로, 서명되지 않은 악성 부트로더의 실행을 차단합니다. Windows 11 설치 시 필수로 요구됩니다.

12. 컴퓨팅 리소스 할당 아키텍처 (Computing Resource Allocation)

가상 머신이 사용할 프로세서(CPU)와 메모리(RAM) 자원을 물리적 호스트로부터 얼마나 빌려올지 결정합니다. 오버프로비저닝(호스트의 자원을 초과하여 할당하는 것)은 전체 시스템의 붕괴를 초래할 수 있으므로 엄격한 계산이 필요합니다.

Processor Configuration

Number of processors (CPU 물리적 개수): 1 또는 2
Number of cores per processor (코어 수): 2
Total processor cores = 2 또는 4
할당 원칙: 호스트 PC 전체 논리 코어 수의 절반(50%) 이하로 할당합니다. 호스트 OS가 구동될 여유 코어를 반드시 남겨두어야 합니다.

Memory (RAM) Allocation

1GB (CLI Linux)
2GB (GUI Linux)
4GB ~ 8GB (Windows 10/11)

할당 원칙: Windows GUI 환경은 최소 4096MB (4GB) 이상을 할당해야 원활합니다. 마찬가지로 호스트 PC 전체 램의 절반 이하를 사용합니다.

* CPU (Central Processing Unit, 중앙 처리 장치): 컴퓨터의 정중앙에서 모든 연산과 제어를 담당하는 두뇌 역할의 장치입니다.
* RAM (Random Access Memory, 임의 접근 기억 장치): 실행 중인 프로그램과 데이터를 일시적으로 저장하는 주 기억 장치입니다.

13. 가상 네트워크 토폴로지 구성 (Virtual Network Topology)

가상 머신이 외부 인터넷 또는 다른 가상 머신과 통신하기 위한 Network Type (네트워크 연결 방식)을 결정합니다. 네트워크 아키텍처의 목적에 따라 세 가지 대역 중 하나를 선택합니다.

Bridged

물리적 공유기와 직접 연결됩니다. 호스트 PC와 동일한 네트워크 대역의 IP를 공유기(DHCP)로부터 직접 할당받습니다.

외부 접속용 서버 구동 시 적합
기본 권장값

NAT

호스트 PC의 IP 주소를 가상 머신이 빌려 써서(공유하여) 외부 인터넷으로 나갑니다. IP 충돌이 없어 가장 안전합니다.

일반적인 인터넷 실습 시 필수

Host-only

외부 인터넷을 완전히 차단하고, 오직 호스트 PC와 가상 머신들 간의 폐쇄적인 내부 통신망만 구성합니다.

악성코드 분석, 폐쇄망 실습용
* NAT (Network Address Translation, 네트워크 주소 변환): 사설 IP 주소(내부용)를 공인 IP 주소(인터넷용)로 변환하거나 그 반대로 변환하는 통신 기술입니다.

14. 가상 스토리지 I/O 및 디스크 프로비저닝 (Virtual Storage)

가상 머신의 데이터가 저장될 가상 하드 디스크 (.vmdk 파일)의 크기와 파일 구조를 정의합니다. I/O 컨트롤러(LSI Logic 등)와 디스크 타입(NVMe, SCSI 등)은 프로그램이 제안하는 (Recommended) 권장값을 그대로 유지하고 디스크 생성 단계로 진입합니다.

STEP 04

디스크 용량 및 파일 관리 방식 설정 (Disk Capacity)

Maximum disk size (GB)

Windows는 60GB, Linux는 최소 20GB 할당 (Thin Provisioning 적용됨)
퍼포먼스 핵심 옵션
Store virtual disk as a single file
가상 디스크를 하나의 거대한 .vmdk 단일 파일로 저장합니다.
Split virtual disk into multiple files
디스크를 2GB 단위의 수많은 조각 파일로 나눕니다. (이동은 편하나 I/O 속도 저하 발생)

Thin Provisioning (동적 할당) 개념

최대 디스크 크기를 60GB로 설정했다고 해서 호스트 PC의 용량이 즉시 60GB 줄어드는 것이 아닙니다. 초기에는 OS 설치에 필요한 용량(약 15GB)만 차지하며, 데이터를 저장할 때마다 설정한 60GB 한도 내에서 물풍선처럼 유연하게 용량이 점진적으로 늘어나는 방식을 사용합니다. 이를 가상화 스토리지 기술에서 Thin Provisioning 이라고 부릅니다.
