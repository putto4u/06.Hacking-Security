CHAPTER 12. DEVOPS & INFRA AS CODE

자동화된 보안 아카이브 배포 파이프라인(Pipeline) 구축

본 장에서는 작성된 파이썬(Python) 기반의 동적 목차 생성 스크립트(generate_index.py)를 클라우드 환경에서 자동으로 실행하고, 그 결과물을 깃허브 페이지(GitHub Pages, 깃허브 저장소에서 직접 정적 웹사이트를 호스팅하는 서비스)에 배포하는 엔터프라이즈급 CI/CD(Continuous Integration/Continuous Deployment, 지속적 통합 및 배포) 아키텍처를 설계합니다.

학습 목표 및 배경

과거처럼 스크립트를 로컬 PC에서 실행한 후 생성된 index.html을 수동으로 커밋(Commit, 변경 사항을 저장소에 기록하는 작업)하는 방식은 비효율적이며 휴먼 에러를 유발합니다.

최신 데브옵스(DevOps, 개발과 운영의 결합) 표준에 맞추어, 코드가 푸시(Push, 로컬 저장소의 데이터를 원격 저장소로 전송하는 작업)될 때마다 가상 서버가 스크립트를 실행하고 웹사이트를 갱신하도록 워크플로를 구성합니다.

1. 깃허브(GitHub) 저장소 사전 설정 가이드

워크플로(Workflow, 자동화된 작업의 흐름)가 정상적으로 동작하기 위해서는 깃허브 저장소(Repository, 코드와 파일이 저장되는 공간)의 보안 및 배포 권한을 먼저 조정해야 합니다.

1

GitHub Pages 소스 변경

  • 저장소의 [Settings] → [Pages]로 이동합니다.
  • Build and deployment 섹션에서 Source 드롭다운을 찾습니다.
  • Deploy from a branch에서 GitHub Actions로 변경합니다.

이 설정은 깃허브 액션이 직접 빌드한 아티팩트(Artifact, 빌드 프로세스의 결과물)를 웹페이지로 서비스하겠다는 선언입니다.

2

워크플로 권한(Permissions) 부여

  • 저장소의 [Settings] → [Actions] → [General]로 이동합니다.
  • 최하단의 Workflow permissions 섹션을 확인합니다.
  • Read and write permissions 라디오 버튼을 선택하고 저장합니다.

과금 및 요금 주의 사항 (Billing Notice)

GitHub Actions는 퍼블릭 저장소(Public Repository)에서는 무제한 무료로 제공됩니다. 하지만 프라이빗 저장소(Private Repository)의 경우, 무료 계정(Free Tier) 기준으로 매월 2,000분의 리눅스(Linux) 가상 머신 실행 시간만 기본 제공됩니다. 이를 초과할 경우 분당 약 $0.008의 과금이 발생하므로, 무한 루프가 발생하지 않도록 트리거(Trigger, 실행 조건) 설정에 각별히 주의해야 합니다.

2. CI/CD 파이프라인 스크립트 작성

저장소 최상위 디렉터리에 .github/workflows/ 폴더를 생성하고, 내부에 deploy-pages.yml 파일을 작성합니다. 야믈(YAML, 직관적인 데이터 직렬화 언어) 포맷은 들여쓰기(Indentation)가 매우 중요하므로 공백의 개수에 유의해야 합니다.

.github/workflows/deploy-pages.yml 
name: Auto Generate and Deploy Security Archive

# 1. 트리거(Trigger) 설정: 언제 파이프라인을 실행할 것인가
on:
  push:
    branches: ["main"]  # main 브랜치에 코드가 푸시될 때 실행 (master인 경우 변경 필요)
  workflow_dispatch:    # 깃허브 웹 화면에서 수동으로 실행할 수 있는 버튼 활성화

# 2. 권한(Permissions) 설정: 봇(Bot) 계정에 부여할 최소 권한
permissions:
  contents: read      # 저장소 코드 읽기 권한
  pages: write        # GitHub Pages에 배포할 권한
  id-token: write     # OIDC(OpenID Connect) 인증을 위한 토큰 발행 권한

# 3. 동시성(Concurrency) 제어: 중복 실행 방지
concurrency:
  group: "pages"
  cancel-in-progress: true # 새 푸시가 발생하면 진행 중인 구버전 배포를 취소

# 4. 작업(Jobs) 정의
jobs:
  # Job 1: 파이썬 스크립트 실행 및 결과물 패키징
  build:
    runs-on: ubuntu-latest  # 최신 우분투 리눅스 환경 프로비저닝
    steps:
      - name: 저장소 체크아웃 (Checkout Repository)
        uses: actions/checkout@v4
        with:
          fetch-depth: 0    # 전체 커밋 히스토리를 가져옴

      - name: 파이썬 환경 설정 (Setup Python 3.12)
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: 동적 인덱스 생성 스크립트 실행 (Run Script)
        run: python generate_index.py

      - name: GitHub Pages 설정 (Setup Pages)
        uses: actions/configure-pages@v4

      - name: 웹 정적 파일 아티팩트 업로드 (Upload Artifact)
        uses: actions/upload-pages-artifact@v3
        with:
          path: '.'  # 최상위 경로의 모든 파일(생성된 index.html 포함)을 업로드

  # Job 2: 패키징된 아티팩트를 실제 웹 서버에 배포
  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build  # build 작업이 성공적으로 끝난 후에만 실행됨
    steps:
      - name: GitHub Pages 배포 완료 (Deploy to Pages)
        id: deployment
        uses: actions/deploy-pages@v4

3. 실무 아키텍처 해설 및 트러블슈팅(Troubleshooting)

아티팩트(Artifact) 기반 배포의 장점

과거에는 파이썬이 생성한 index.html을 git commit 명령어로 저장소에 강제로 밀어 넣는 방식을 주로 사용했습니다. 그러나 이 방식은 커밋 히스토리를 지저분하게 만들고 충돌(Conflict)을 유발하는 원인이 됩니다.

위 야믈 파일에서 적용한 upload-pages-artifact 방식은 저장소의 원본 소스 코드는 건드리지 않고, 임시 가상 서버의 메모리상에서 생성된 결과물만 패키징하여 웹 서버로 바로 쏘아 올리는(Deploy) 엔터프라이즈 모범 사례(Best Practice)입니다.

파일 수정일(Modification Time) 관련 실전 팁

파이썬 스크립트 내의 os.path.getmtime() 함수는 '파일이 디스크에 기록된 시간'을 반환합니다. 깃허브 액션 가상 환경은 매번 새롭게 우분투(Ubuntu) OS를 설치하고 깃(Git) 저장소를 다운로드(Clone)하기 때문에, 모든 파일의 수정일이 워크플로가 실행된 현재 시간으로 동일하게 설정되는 한계가 있습니다.

아키텍트 조언: 이 현상은 CI/CD 환경의 고유한 특성이며 버그가 아닙니다. 만약 파일이 최초 생성된 과거의 시간이나 마지막으로 커밋된 정확한 시간을 인덱스 페이지에 표시하고 싶다면, 시스템 함수 대신 깃 로그(Git Log)를 추적하여 타임스탬프를 추출하는 방식으로 파이썬 코드를 고도화해야 합니다.

자주 발생하는 에러 (Troubleshooting)

HTTP 403 Not allowed to create a GitHub Pages site

원인: 1장에서 설명한 'Workflow permissions' 권한이 누락되었거나 부족할 때 발생합니다.
해결: 설정의 Actions → General 항목에서 권한이 Read and write permissions로 명확히 체크되어 있는지 재확인합니다.

액션은 성공했는데 페이지 내용이 안 변할 때

원인: 기존의 Legacy(구형) 브랜치 기반 배포가 켜져 있어서 새로운 액션 기반 배포와 충돌하는 경우입니다.
해결: 설정의 Pages 항목에서 Source가 반드시 GitHub Actions로 선택되어 있는지 확인합니다.
