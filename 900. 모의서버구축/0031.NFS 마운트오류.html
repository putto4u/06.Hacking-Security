서비스 가용성 확보와 네트워크 파일 공유 검증

System Architect Pro

1. 아키텍처 개요 (Architecture Overview)

NFS(Network File System, 네트워크 파일 시스템) 환경 설정의 마지막 단계는 구성된 정책을 커널(Kernel, 운영체제 핵심부)에 반영하고 서비스의 가용성(Availability, 서비스가 중단 없이 지속되는 능력)을 확인하는 것입니다. 설정 파일인 /etc/exports의 수정 내용은 단순 저장만으로는 적용되지 않으며, RPC(Remote Procedure Call, 원격 프로시저 호출) 데몬과 통신하여 리엑스포트(Re-export, 재공유 설정 반영) 과정을 거쳐야 합니다.

[도식: NFS 설정 반영 및 마운트 프로세스 흐름]

SERVER CONFIG
/etc/exports
Re-export
Kernel Level
NFS DAEMON
TCP 2049
CLIENT MOUNT
/mnt/nfs_client

2. 실습 구성 요소 (Components)

STORAGE NODE

NFS 서버 (Server)

  • IP Address: 192.168.100.3
  • Export Path: /srv/nfs/shared
CLIENT NODE

NFS 클라이언트 (Client)

  • Network: 192.168.100.0/24 Subnet
  • Mount Point: /mnt/nfs_client

3. 실증 및 구현 (Implementation)

STEP 01

설정 반영 및 서비스 활성화

수정된 /etc/exports 파일의 내용을 커널에 즉시 반영하고 백그라운드 프로세스인 데몬(Daemon)을 재시작하여 정상 동작 여부를 검증합니다.

NFS Server
# 1. 수정된 설정을 커널에 즉시 반영 (Re-export)
$ sudo exportfs -ra
# 2. NFS 서버 서비스 재시작
$ sudo systemctl restart nfs-kernel-server
# 3. 서비스가 정상적으로 동작 중인지 상태 확인
$ sudo systemctl status nfs-kernel-server
STEP 02

마운트 포인트 및 권한 구성

공유하려는 물리적 디렉토리가 실제로 존재해야 하며, 클라이언트가 데이터를 읽고 쓸 수 있도록 디렉토리 소유권 및 퍼미션(Permission, 권한)을 조정합니다.

# 디렉토리가 없다면 생성
$ sudo mkdir -p /srv/nfs/shared
# 테스트를 위해 모든 사용자에게 권한 부여
$ sudo chmod 777 /srv/nfs/shared

※ 실무 환경에서는 보안을 위해 777 대신 특정 시스템 계정 권한 부여 및 ID Squashing(아이디 스쿼싱) 기법 적용을 적극 권장합니다.

STEP 03

최종 유효성 검증 (Verification)

멈춤 현상이 해결되었는지 확인하기 위해 showmount 명령어를 다시 실행하여 서버가 정상적으로 공유 리스트를 출력하는지 확인합니다.

# 서버 자신(192.168.100.3)의 공유 목록 확인
$ showmount -e 192.168.100.3
Export list for 192.168.100.3:
/srv/nfs/shared 192.168.100.0/24
STEP 04

클라이언트 마운트 테스트 (Mount Test)

모든 서버 설정이 완료되었다면 클라이언트 측 마운트 포인트(/mnt/nfs_client)에서 원격 볼륨을 연결합니다.

NFS Client
# 마운트 실행
$ sudo mount -t nfs 192.168.100.3:/srv/nfs/shared /mnt/nfs_client
# 마운트 결과 확인 (Disk Free 사용)
$ df -h | grep nfs
192.168.100.3:/srv/nfs/shared 20G 1.2G 18G 7% /mnt/nfs_client

4. 아키텍트 인사이트 (Expert Insight)

로컬 네트워크 환경 비용 분석

현재 구성 중인 온프레미스(On-Premise, 사내 구축형) 기반 NFS 서비스는 별도의 소프트웨어 구매 비용이 발생하지 않는 오픈소스 기반입니다. 초기 하드웨어 투자비(CapEx) 외에 운영 관리 비용 위주로 산정됩니다.

AWS 확장 시 유료 과금 주의

향후 인프라를 AWS로 확장하여 AWS Storage Gateway (AWS 스토리지 게이트웨이, 유료)를 통해 클라우드 스토리지와 연결할 경우, 게이트웨이 시간당 이용료 및 월별 데이터 전송(Data Transfer) 비용이 발생합니다. 완전 관리형 NFS인 Amazon EFS (Elastic File System) 도입 시에도 데이터 저장량에 따른 사용량 기반 요금이 부과되므로 설계 시 TCO(Total Cost of Ownership, 총 소유 비용)를 반드시 검토해야 합니다.
