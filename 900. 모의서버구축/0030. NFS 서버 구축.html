NFS 서버 구축 및 클라이언트 마운트

본 매뉴얼은 리눅스 및 유닉스(Unix) 환경에서 고성능 네트워크 파일 공유의 표준으로 자리 잡은 NFS(Network File System, 네트워크 파일 시스템)를 최신 우분투(Ubuntu) 환경에 구축하고, 클라이언트 노드에서 안정적으로 연결(Mount) 및 해제하는 실무 아키텍처 가이드를 제공합니다.

Ubuntu 공식 NFS(Network File System) 문서 참조

NFS 서버 패키지 설치 및 스토리지 준비

NFS는 커널(Kernel, 운영체제의 핵심부) 레벨에서 동작하므로 매우 빠르고 가볍습니다. 서버 역할을 수행할 우분투 인스턴스에 코어 데몬(Daemon, 백그라운드 프로세스) 패키지를 설치하고 데이터를 저장할 물리적 경로를 프로비저닝(Provisioning, 자원 할당)합니다.

# 1. 시스템 패키지 인덱스 업데이트 및 NFS 커널 서버 패키지 설치
$ sudo apt update
$ sudo apt install -y nfs-kernel-server

# 2. 클라이언트와 공유할 최상위 디렉토리 생성 (예: /srv/nfs/shared)
$ sudo mkdir -p /srv/nfs/shared

# 3. NFS 접근을 위한 권한 초기화 및 소유권 변경
# 보안상 특정 로컬 계정에 종속되지 않도록 nobody 계정 및 nogroup 그룹으로 매핑합니다.
$ sudo chown nobody:nogroup /srv/nfs/shared
$ sudo chmod -R 0775 /srv/nfs/shared

nobody:nogroup 권한의 의미: 외부 클라이언트가 NFS 서버에 접속할 때, 기본적으로 클라이언트의 root 권한은 서버 측에서 익명 권한(nobody)으로 축소(Squash, 강등)됩니다. 이를 통해 악의적인 클라이언트가 서버의 중요 시스템 파일을 수정하는 것을 방지하는 핵심 보안 메커니즘입니다.

Export(접근 제어) 파일 환경 설정 심층 분석

NFS(Network File System, 네트워크 파일 시스템) 서버의 핵심 보안 및 성능 최적화는 /etc/exports 파일의 정밀한 구성에 달려 있습니다. 이 파일은 특정 IP(Internet Protocol, 인터넷 통신 규약) 대역의 클라이언트에게 부여할 권한과 I/O(Input/Output, 파일 입출력) 방식을 제어하는 ACL(Access Control List, 접근 제어 목록) 역할을 수행합니다.

# 텍스트 에디터로 NFS 설정 파일 열기
$ sudo nano /etc/exports

# 기본 문법 형식: [공유할_경로] [허용대상_IP_또는_도메인]([옵션1],[옵션2],...)
/srv/nfs/shared 192.168.0.0/24(rw,sync,no_subtree_check)

1. 호스트 타겟팅 (Host Targeting) 기법

공유 폴더에 접근할 수 있는 대상을 제한하는 것은 가장 기초적이고 강력한 보안 조치입니다. 다음과 같은 다양한 패턴으로 대상을 지정할 수 있습니다.

  • 단일 IP 지정: 192.168.0.50 (특정 웹/앱 서버 인스턴스 한 대만 허용할 때 사용)
  • CIDR(Classless Inter-Domain Routing, 클래스 없는 도메인 간 라우팅) 대역: 192.168.0.0/24 (해당 서브넷망 내부의 모든 시스템에 접근을 허용할 때 사용. 엔터프라이즈 환경의 표준 방식입니다.)
  • 와일드카드(Wildcard) 도메인: *.example.com (해당 도메인을 사용하는 모든 호스트를 허용합니다.)

NFS 서버의 ACL(Access Control List, 접근 제어 목록) 기반 클라이언트 연결 허용/차단 논리 구조

2. 파일 I/O 및 성능 옵션 (Performance & I/O)

데이터의 무결성(Integrity, 데이터가 손상되지 않고 정확함)을 우선할 것인지, I/O 처리 속도를 우선할 것인지에 따라 동기화 방식을 결정해야 합니다.

  • rw 및 ro (읽기/쓰기 제어) 클라이언트에게 읽기 및 쓰기 권한을 모두 부여하려면 rw(Read/Write)를, 백업 이미지 저장소처럼 데이터가 변조되면 안 되는 경우 읽기 전용인 ro(Read Only)를 사용합니다.
  • sync (데이터 동기화 기록) 클라이언트의 데이터 쓰기 요청 시, NFS 서버의 물리적 디스크(디스크 캐시가 아닌 실제 플래터나 낸드플래시)에 완전히 기록이 끝난 후에만 클라이언트에게 성공 응답(ACK)을 보냅니다. 정전 등 돌발 장애 시 데이터 유실(Data Loss)을 방지하는 엔터프라이즈 필수 옵션입니다.
  • async (데이터 비동기 기록) 서버의 메모리(RAM) 버퍼에만 데이터가 올라가면 즉시 클라이언트에게 성공 응답을 보냅니다. 체감 속도와 성능은 비약적으로 상승하지만, 디스크 기록 전 서버 전원이 차단되면 데이터가 증발하는 치명적 단점이 있습니다.
  • no_subtree_check (서브트리 검사 비활성화) 기본적으로 NFS는 클라이언트가 접근하려는 파일이 해당 공유 볼륨의 올바른 하위 디렉토리(Subtree) 내부에 있는지 매번 추적 검사합니다. 그러나 파일의 이름이 변경되거나 이동할 때 이 추적이 끊어지면 "Stale File Handle(만료된 파일 핸들)" 오류가 발생합니다. 현대 아키텍처에서는 시스템 안정성과 부하 감소를 위해 이 검사를 끄는 no_subtree_check 옵션을 표준으로 권장합니다.

3. 권한 축소 및 매핑 기술 (UID/GID Squashing)

가장 중요한 보안 개념입니다. '스쿼싱(Squashing, 짓누르기/축소)'은 클라이언트 시스템에서 접근하는 사용자의 시스템 권한(UID/GID)을 NFS 서버 단에서 강제로 안전한 익명 권한으로 낮추어 버리는 매핑(Mapping) 기술입니다.

  • root_squash (기본값) 클라이언트 측에서 최고 관리자(root, UID 0) 권한으로 NFS 서버에 파일을 쓰려고 하면, 서버는 이를 즉시 익명 계정인 nobody(또는 nfsnobody) 권한으로 강등시킵니다. 클라이언트 해킹 시 서버 인프라까지 장악되는 것을 막아줍니다.
  • all_squash 클라이언트가 root이든, 일반 사용자(User)이든 상관없이 모든 연결 권한을 nobody로 축소시킵니다. 공용 데이터베이스 덤프 저장소나 불특정 다수가 접근하는 퍼블릭 볼륨(Public Volume) 구성 시 사용됩니다.
  • anonuid=1000, anongid=1000 all_squash 옵션과 함께 사용되며, 축소된 익명 사용자의 권한을 서버에 존재하는 특정 UID/GID(사용자/그룹 식별자) 번호로 지정해 줍니다. 파일 소유권 관리가 매우 깔끔해집니다.

절대 사용 금지 : no_root_squash 옵션

no_root_squash는 스쿼싱(권한 축소)을 수행하지 않는다는 뜻입니다. 즉, 클라이언트 PC의 root 사용자가 NFS 서버 볼륨에서도 동일한 최고 관리자(root) 대우를 받게 됩니다.

※ 리스크: 악의적인 클라이언트가 악성 바이너리(Binary, 실행 파일)에 SetUID(루트 권한 상승 플래그)를 설정하여 NFS 서버에 업로드하는 순간, 전체 스토리지 서버가 랜섬웨어 등 심각한 보안 침해(Breach)에 노출됩니다. OS(운영체제) 디스크리스 부팅 같은 특수 목적 외에는 절대 구성하지 마십시오.

클라이언트 측 UID 0 (root) ➔ NFS Server ➔ UID 65534 (nobody) 변환 매핑 구조

4. 실무 설정 템플릿 예시 및 적용

앞서 설명한 옵션들을 조합하여 인프라 목적에 맞는 /etc/exports 구성을 완성합니다. 설정 완료 후 데몬 재시작 없이 즉시 반영합니다.

# [실무 예제 1] 사내망 백업 저장소 (모든 사용자를 특정 UID 백업 계정으로 통합)
/srv/nfs/backup 10.0.0.0/16(rw,sync,no_subtree_check,all_squash,anonuid=1005,anongid=1005)

# [실무 예제 2] 웹 서버 클러스터 소스코드 공유 (안전한 기본 보안 유지)
/srv/nfs/www 172.16.10.0/24(rw,sync,no_subtree_check,root_squash)

# 설정 테이블 갱신 (-a: 모든 공유 목록 갱신, -r: 기존 목록 제거 후 리로드, -v: 상세 출력)
$ sudo exportfs -arv
exporting 10.0.0.0/16:/srv/nfs/backup
exporting 172.16.10.0/24:/srv/nfs/www

# NFS 서버 커널 서비스 안정화 및 상시 활성화
$ sudo systemctl restart nfs-kernel-server
$ sudo systemctl enable nfs-kernel-server

방화벽 허용 및 아키텍처 도식화

NFS 통신을 허용하기 위해 UFW(Uncomplicated Firewall, 우분투 기본 방화벽)에서 접근 제어 규칙을 추가합니다. 허용할 서브넷(Subnet, 부분 망) 대역을 명시적으로 지정하여 보안을 강화합니다.

# 192.168.0.0/24 대역에서 들어오는 NFS 포트(TCP/UDP 2049) 접근 허용
$ sudo ufw allow from 192.168.0.0/24 to any port nfs
Rule added

# 상태 확인
$ sudo ufw status

Client Node (Web/App)
IP: 192.168.0.50
Mount: /mnt/nfs_client
클라이언트 커널의 VFS(가상 파일 시스템)를 통해 원격 저장소를 로컬 디스크처럼 인식
RPC / TCP 2049
NFS Storage Server
IP: 192.168.0.100
Export: /srv/nfs/shared
nfs-kernel-server 및 rpcbind 데몬이 포트를 수신하고 데이터를 블록(Block) 단위로 처리

클라이언트 접속 확인 및 수동 마운트(Mount)

NFS 공유 자원을 활용할 클라이언트 서버(Ubuntu 기반)로 이동하여, 접속에 필요한 패키지를 설치하고 수동으로 파일 시스템을 연결(Mount)하여 검증합니다.

# 1. 클라이언트 시스템용 NFS 지원 패키지 설치
$ sudo apt update && sudo apt install -y nfs-common

# 2. 원격 NFS 서버(192.168.0.100)의 공유 가능 폴더 목록(Export List) 조회
$ showmount -e 192.168.0.100
Export list for 192.168.0.100:
/srv/nfs/shared 192.168.0.0/24

# 3. 로컬에 마운트(연결)할 빈 디렉토리(마운트 포인트) 생성
$ sudo mkdir -p /mnt/nfs_client

# 4. 수동 마운트 실행 (-t nfs : 파일 시스템 타입을 nfs로 명시)
$ sudo mount -t nfs 192.168.0.100:/srv/nfs/shared /mnt/nfs_client

# 5. 연결 확인 및 파일 시스템 디스크 사용량(Disk Free) 조회
$ df -h -T | grep nfs
192.168.0.100:/srv/nfs/shared nfs4 100G 1.5G 99G 2% /mnt/nfs_client

df -h -T 옵션 해설: -h는 사람이 읽기 쉬운 단위(GB, MB)로 출력하고, -T는 파일 시스템의 타입(Type, 예: nfs4, ext4, xfs)을 함께 출력하여 직관적인 확인을 돕습니다.

영구 마운트(fstab) 및 시스템 연결 해제(umount)

1. 부팅 시 자동 연결을 위한 fstab(File System Table) 등록

수동 마운트는 시스템이 재부팅(Reboot)되면 연결이 해제됩니다. 영구적인 연결을 유지하기 위해 운영체제의 파일 시스템 정보 테이블(/etc/fstab)에 환경을 등록해야 합니다.

# fstab 파일 편집기 열기
$ sudo nano /etc/fstab

# 파일 최하단에 다음 구문을 추가합니다.
# 형식: [서버IP:경로] [로컬_마운트포인트] [파일시스템종류] [마운트옵션] [덤프] [검사순서]
192.168.0.100:/srv/nfs/shared /mnt/nfs_client nfs defaults,_netdev,soft,timeo=10 0 0

실전 팁 (Practical Tips) : 엔터프라이즈급 fstab 옵션

클라우드 환경이나 네트워크가 불안정한 환경에서 단순 defaults 옵션만 사용하면, 부팅 시 네트워크가 아직 올라오지 않았을 때 시스템 부팅이 멈추는 장애(Hang)가 발생할 수 있습니다.

- _netdev: 시스템 부팅 시 네트워크 서브시스템이 완전히 시작된 이후에 마운트를 시도하도록 강제합니다.
- soft: 네트워크 연결이 끊기면 일정 시간 후 재시도를 포기하고 타임아웃(Timeout) 오류를 발생시켜 시스템 전체가 멈추는 것을 방지합니다.
- timeo=10: 응답 대기 시간을 10데시초(Decisecond, 1초)로 짧게 설정하여 빠른 장애 탐지를 돕습니다.

2. 마운트 해제(Unmount) 실무 절차

스토리지 마이그레이션(Migration, 데이터 이관)이나 유지보수를 위해 현재 연결된 NFS 볼륨의 논리적 연결을 안전하게 해제(Umount)하는 방법입니다.

# 1. 대상 디렉토리의 현재 사용 여부 확인 및 안전한 해제
$ sudo umount /mnt/nfs_client

# 2. "device is busy" (장치 사용 중) 오류 발생 시 대처법
# 시스템 내의 다른 프로세스가 해당 폴더를 점유하고 있을 때 강제로 지연 해제(Lazy Unmount)를 수행합니다.
$ sudo umount -l /mnt/nfs_client

# 강제 옵션(-f, Force) (주로 NFS 서버가 응답하지 않는 네트워크 단절 상황에 사용)
$ sudo umount -f /mnt/nfs_client

본 매뉴얼은 Ubuntu 커널 기반 시스템에서 대규모 파일 공유 아키텍처를 구현하기 위한 NFS 표준 실무 가이드입니다. 스케일 아웃(Scale-out, 노드 확장) 스토리지 클러스터를 구성할 경우 CephFS나 GlusterFS와 같은 분산 파일 시스템과 함께 성능 및 가용성(Availability)을 비교하여 설계해야 합니다.
