참고 자료 : 메타스플로잇 프레임워크 로그 시스템의 이해와 활용

메타스플로잇 프레임워크(Metasploit Framework, 오픈소스 모의 해킹 플랫폼)의 로그(Log)는 공격 벡터(Attack Vector, 공격 매개체)의 실행 기록, 시스템 오류, 페이로드(Payload, 공격 코드가 포함된 데이터)의 전송 상태 등을 상세히 기록하는 핵심 감사(Audit) 자원입니다. 성공적인 모의 해킹 수행과 사후 분석(Digital Forensics, 디지털 포렌식)을 위해 로그의 구조와 수집 방식을 명확히 이해해야 합니다.

주요 로그 파일의 구조와 위치

메타스플로잇의 로그는 기본적으로 사용자 홈 디렉터리의 숨김 폴더 내에 저장되며, 프레임워크의 구동 상태와 모듈(Module, 특정 기능을 수행하는 단위 코드)의 실행 내역을 분리하여 관리합니다.

프레임워크 로그 (~/.msf4/logs/framework.log)

프레임워크 코어(Core) 시스템의 동작, 모듈 로드(Load, 메모리에 적재) 상태, 플러그인(Plugin, 확장 프로그램)의 초기화 과정에서 발생하는 이벤트와 에러를 기록합니다. 프레임워크 자체의 트러블슈팅(Troubleshooting, 문제 해결)에 주로 활용됩니다.

콘솔 로그 (~/.msf4/logs/msfconsole.log)

CLI(Command Line Interface, 명령줄 인터페이스) 환경인 msfconsole에서 사용자가 입력한 명령어와 백그라운드 프로세스 기록을 저장합니다. 사용자 행위 추적에 필수적인 파일입니다.

로그 레벨(Log Level) 제어

메타스플로잇은 디버깅(Debugging, 오류 수정 및 추적 과정)과 정보 수집의 깊이를 제어하기 위해 0부터 3까지의 로그 레벨 범위를 제공합니다.

  • 레벨 0 (기본값): 심각한 오류(Error)와 경고(Warning) 메시지만 기록합니다.
  • 레벨 1: 정보성(Info) 메시지를 추가로 기록하여 일반적인 작업 흐름을 파악할 수 있습니다.
  • 레벨 2: 디버그(Debug) 메시지를 포함하여 모듈의 내부 동작 과정을 상세히 기록합니다.
  • 레벨 3: 프레임워크 수준의 매우 상세한 로우 데이터(Raw Data, 가공되지 않은 원본 데이터)를 기록하며, 주로 모듈 개발 및 심층 원인 분석 시 사용됩니다.

설정 명령어 예시

msf6 > setg LogLevel 2

전역 변수(setg)로 로그 레벨을 2로 설정하여 이후 실행되는 모든 세션에 일괄 적용합니다.

세션 스풀링(Session Spooling)을 통한 출력 기록

화면에 출력되는 모든 명령어와 응답 결과를 지정된 텍스트 파일로 실시간 저장(캡처)하는 기능입니다. 모의 해킹 보고서 작성이나 작업 증적(Evidence) 확보에 가장 직관적이고 널리 쓰이는 방법입니다.

스풀링 명령어

msf6 > spool /root/msf_assessment_log.txt

터미널 출력 기록 시작

msf6 > spool off

💡 실전 팁 (Practical Tips)
데이터베이스(Database) 연동의 필수화

텍스트 기반의 로그 파일만으로는 대규모 취약점 진단 시 데이터를 구조화하기 어렵습니다. PostgreSQL(포스트그레스큐엘, 오픈소스 객체-관계형 데이터베이스)을 연동하여 타겟 호스트(Host), 발견된 취약점(Vulnerability), 수집된 자격 증명(Credential)을 DB(Database, 데이터베이스)에 자동 기록되도록 구성하는 것이 실무 아키텍처의 표준입니다. (workspace 명령어를 통해 프로젝트별로 데이터를 안전하게 논리적 분리할 수 있습니다.)

타임스탬프(Timestamp) 동기화

여러 대의 진단 서버를 운영할 경우, 로그 분석 시 시간 오차가 발생하지 않도록 NTP(Network Time Protocol, 네트워크 시간 프로토콜)를 이용하여 시스템 시간을 철저히 동기화해야 합니다. 분산 환경에서 시간 동기화 실패는 치명적인 포렌식 오류를 낳습니다.

⚠️ 자주 오해하거나 실수하는 부분
스풀(Spool) 기능의 맹신

spool 명령어는 현재 콘솔 화면에 나타나는 표준 출력(Standard Output)만을 기록합니다. 백그라운드(Background)에서 실행 중인 잡(Job, 작업 프로세스)이나, 비동기적으로 발생하는 프레임워크 내부 오류는 기록되지 않으므로, 에러 발생 시 반드시 framework.log를 교차 검증해야만 정확한 원인을 파악할 수 있습니다.

로그 용량 관리 방치

높은 LogLevel 설정 상태로 대규모 네트워크 대역에 장기간 스캐닝(Scanning) 작업을 수행하면, 로그 파일의 크기가 급격히 팽창하여 스토리지(Storage, 저장 공간) 고갈 장애를 유발할 수 있습니다. 인프라 운영 관점에서 Linux의 logrotate(로그 로테이트, 로그 파일 자동 교체 및 압축 데몬)와 연동하여 주기적인 로그 관리를 설정하는 것이 중요합니다.
