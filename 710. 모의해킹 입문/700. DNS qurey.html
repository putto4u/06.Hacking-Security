DNS 정보 수집 및 분석: nslookup 심층 활용 가이드

네트워크 인프라의 나침반, 도메인 네임 시스템을 통한 타겟 자산 식별 및 OSINT 정찰 전략

모의 해킹의 첫 단계인 OSINT(Open Source Intelligence, 공개 출처 정보) 수집 과정에서 DNS(Domain Name System, 도메인 이름 시스템)는 타겟 인프라의 논리적 지도를 그리는 가장 확실한 정보의 보고입니다. 본 교재에서는 윈도우와 리눅스 모든 환경에 기본 탑재된 nslookup(Name Server Lookup, 네임 서버 조회) 도구를 활용하여, 단순한 IP 확인을 넘어 이메일 서버 식별, 보안 정책 확인, 내부망 토폴로지 유추까지 수행하는 전문가 수준의 심층 정보 획득 기법을 다룹니다.

1. DNS 아키텍처와 nslookup 동작 원리

nslookup은 운영체제의 DNS 캐시(Cache)를 거치지 않고, 네임 서버(Name Server)에 직접 질의(Query)를 던져 순수한 응답을 받아오는 도구입니다. 이를 통해 해커나 보안 엔지니어는 타겟 도메인의 설정이 전파되는 과정이나 숨겨진 서브 도메인(Sub-domain)의 존재 여부를 정밀하게 추적할 수 있습니다.

[도식 1] nslookup을 통한 DNS 질의 및 응답(Resolution)의 논리적 아키텍처

* 재귀 질의(Recursive Query): 클라이언트(사용자)가 로컬 DNS 서버에게 질의를 던지면, 로컬 DNS 서버가 Root 서버부터 시작하여 최종 권한 있는 서버(Authoritative Server)까지 대신 찾아가서 답을 구해오는 방식입니다.
* 권한 있는 네임서버(Authoritative Nameserver): 특정 도메인(예: example.com)의 실제 IP 주소와 레코드 정보 원본을 관리하고 책임지는 최종 보스 서버입니다.
2. 대화형 모드(Interactive Mode) 진입과 주요 레코드 탐색

터미널에 단순히 nslookup example.com을 치는 것은 초보자의 방식입니다. 실무 정보 수집 단계에서는 터미널에 nslookup만 입력하여 대화형 모드(Interactive Mode)로 진입한 후, set type=레코드종류 명령어를 통해 타겟 도메인의 감춰진 내장 기관들을 하나하나 해부합니다.

레코드 (Record) 해킹 관점에서의 정보 수집 의미 및 활용 가치
A / AAAA 가장 기본적인 IPv4 / IPv6 주소 매핑 레코드입니다. www, dev, test 등 앞부분의 서브 도메인을 무차별 대입하여 숨겨진 개발 서버나 관리자 페이지의 실제 IP를 찾아내는 데 사용됩니다.
NS
(Name Server)		 타겟 도메인을 관리하는 네임 서버 정보입니다. 이를 조회하면 해당 기업이 자체 전산실에서 DNS를 운영하는지, 아니면 AWS Route 53이나 Cloudflare 같은 클라우드 인프라를 사용하는지 아키텍처를 유추할 수 있습니다.
MX
(Mail Exchange)		 메일 수신 서버 레코드입니다. 타겟 기업이 Google Workspace, Microsoft 365, 자체 Exchange 서버 중 무엇을 사용하는지 100% 확인 가능합니다. 이는 차후 스피어 피싱(Spear Phishing) 공격을 기획할 때 메일 서버의 필터링 수준을 가늠하는 핵심 지표가 됩니다.
TXT
(Text)		 텍스트 문자열을 저장하는 레코드입니다. 관리자가 SPF(Sender Policy Framework) 보안 정책을 여기에 기록합니다. SPF 기록을 분석하면 "타겟 기업이 이메일 발송을 허용하는 내부 IP 대역 전체"를 고스란히 파악할 수 있어 네트워크 토폴로지 분석에 매우 유용합니다.
# 1. 대화형 모드 진입
kali@kali:~$ nslookup
>

# 2. 메일 서버(MX) 정보 조회 설정
> set type=mx
> google.com
Server: 192.168.1.1
Address: 192.168.1.1#53

Non-authoritative answer:
google.com mail exchanger = 10 smtp.google.com.
/* 구글이 자체 메일 인프라를 사용함을 확인 */

# 3. 텍스트(TXT) 레코드를 통한 보안 정책 및 IP 대역 정보 수집
> set type=txt
> example.com
example.com text = "v=spf1 ip4:192.0.2.0/24 include:_spf.google.com ~all"
/* 해석: 이 기업은 192.0.2.0/24 대역을 소유하고 있으며, 구글 메일 시스템을 병행 사용함! (엄청난 정보 획득) */

💡 아키텍트 실전 팁: Non-authoritative answer의 의미

명령어 실행 결과에 나타나는 "Non-authoritative answer (권한 없는 응답)"은 오류가 아닙니다. 이는 내가 질의를 던진 로컬 DNS 서버(예: KT DNS)가 타겟의 진짜 서버가 아니며, 단순히 "캐시(Cache)에 저장되어 있거나 남에게 물어봐서 가져온 정보"라는 의미입니다. 보다 정확한 최신 데이터를 얻으려면 타겟의 권한 있는 네임서버로 타겟을 직접 변경해야 합니다.

3. 질의 서버 변경과 DNS Zone Transfer(존 전송) 취약점 타격

일반적인 DNS 조회를 마쳤다면, 이제 질의를 던지는 대상 자체를 타겟 기업의 권한 있는 네임서버(Authoritative Server)로 변경합니다. 그리고 모의 해킹에서 가장 치명적인 인프라 취약점 중 하나인 DNS Zone Transfer (존 전송 - AXFR) 취약점이 존재하는지 테스트합니다.

DNS Zone Transfer (존 전송 - AXFR) 이란?

본래 주 네임서버(Primary NS)가 장애로 죽었을 때를 대비해, 보조 네임서버(Secondary NS)가 주 서버의 전체 도메인 장부(Zone File)를 동기화하여 복사해 가는 합법적인 백업 메커니즘입니다. 하지만 관리자의 설정 실수로 "아무 IP(즉, 해커)나 Zone File의 복사본을 요청해도 허락해 주는 취약점"이 종종 발견됩니다. 이 취약점이 존재하면 해커는 타겟이 보유한 수천 개의 내부 서버 IP와 서브 도메인 목록을 단 한 번의 명령어로 전부 탈취할 수 있습니다.

[도식 2] 관리자 설정 오류를 악용하여 내부의 전체 도메인 지도를 통째로 복사해 가는 Zone Transfer 공격

# 1. 대상 기업의 네임서버(NS) 확인
> set type=ns
> target.com
target.com nameserver = ns1.target.com.

# 2. 질의 대상을 내 로컬 DNS에서 '타겟의 권한 있는 서버(ns1)'로 강제 변경
> server ns1.target.com
Default server: ns1.target.com
Address: 203.0.113.53#53

# 3. 전체 장부(Zone Transfer) 복사 요청 (ls -d 도메인)
# 취약점이 있다면 아래와 같이 수백 개의 비밀스러운 서브 도메인이 와르르 쏟아집니다.
> ls -d target.com

방어자(Blue Team) 관점의 보안 통제 아키텍처

인프라스트럭처 아키텍트는 BIND(리눅스 네임서버 데몬)나 Windows DNS 서버 설정 시 반드시 Zone Transfer 허용 대상을 'Secondary NS의 지정된 IP'로만 엄격하게 제한(ACL 적용)해야 합니다. 클라우드 환경(Route 53 등)은 설계 단계부터 이 취약점이 기본적으로 차단되도록 매니지드 서비스화되어 있습니다.

4. 역방향 DNS 조회 (Reverse Lookup) 및 도구의 한계

역방향 DNS 조회(Reverse DNS Lookup)는 도메인으로 IP를 찾는 정방향의 반대로, IP 주소를 기반으로 거기에 연결된 도메인 네임(PTR 레코드)을 찾아내는 과정입니다. 해킹 과정에서 출처가 불분명한 IP를 발견했을 때, 이 IP가 구글 클라우드 소속인지 특정 기업의 소유인지를 파악하는 데 사용됩니다.

# 대화형 모드에서 특정 IP의 PTR(Pointer) 레코드 타입으로 역방향 조회
> set type=ptr
> 8.8.8.8
8.8.8.8.in-addr.arpa name = dns.google.

💡 보안 전문가의 툴 체인(Tool-chain): nslookup vs dig

nslookup은 Windows와 Linux에 모두 기본 탑재되어 범용성이 높다는 장점이 있습니다. 하지만 출력 포맷이 다소 조잡하고 스크립트 파싱(Parsing)이 어렵기 때문에, 실제 칼리 리눅스를 다루는 고급 해커들은 BIND 유틸리티에 포함된 dig (Domain Information Groper) 명령어를 훨씬 더 선호합니다. dig는 응답 속도가 빠르고 결과값이 명확하여 모의 해킹 자동화 스크립트를 작성할 때 필수적으로 사용됩니다.
