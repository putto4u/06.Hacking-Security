1. 아키텍처 개요 (Architecture Overview)

본 실습은 AES-256 (Advanced Encryption Standard 256-bit, 고급 암호화 표준 256비트)으로 암호화된 마스터 키를 생성하고, 이를 기반으로 HTTPS (HyperText Transfer Protocol Secure, 보안 하이퍼텍스트 전송 프로토콜) 환경을 구축합니다. 보안 사고에 대비하여 마스터 키(Master Key)와 서비스 키(Service Key)를 물리적으로 분리하고, At-Rest (저장 시 암호화) 전략을 통해 서버 내 자산을 보호하는 계층적 방어 체계를 구현합니다.

2. 실습 구성 요소 (Components)

WEB ENGINE

Apache 2.4 (아파치 웹 서버)

  • mod_ssl: 암호화 엔진 활성화 및 TLS 프로토콜 처리
  • mod_headers: HSTS 정책 및 보안 헤더 제어
CRYPTO ENGINE

OpenSSL (오픈 SSL 도구)

  • AES-256: 개인키 자체를 저장 시점에 암호화하는 알고리즘
  • X.509: 공개키 인증서 및 CSR 생성을 위한 표준 규격

3. 실증 및 구현 (Implementation)

STEP 01

마스터 키 생성 및 AES-256 보안 적용

보안 저장을 위해 디렉토리를 생성하고, 2048비트 RSA 키를 생성하되 AES-256 옵션을 통해 키 자체를 패스프라이즈로 보호합니다.

Root Console
# 보안 디렉토리 생성 및 권한 설정
$ sudo mkdir -p /etc/ssl/private /etc/ssl/backup /etc/ssl/archive
$ sudo chmod 700 /etc/ssl/private /etc/ssl/backup /etc/ssl/archive
# AES-256 암호화 마스터 키 생성
$ sudo openssl genrsa -aes256 -out /etc/ssl/private/server.key.master 2048
Enter pass phrase for server.key.master: ********
STEP 02

마스터 키 즉각 백업 (Master Key Backup)

운영 중 발생할 수 있는 키 손상이나 실수에 대비하여 날짜별 백업본을 생성하여 격리 보관합니다.

Backup Process
$ sudo cp /etc/ssl/private/server.key.master /etc/ssl/backup/server.key.master.$(date +%Y%m%d).bak
File copied to /etc/ssl/backup/server.key.master.20260330.bak
STEP 03

CSR 생성 및 식별 정보 입력

인증 기관(CA)에 제출할 CSR (Certificate Signing Request, 인증서 서명 요청서)을 작성합니다. 이 과정에서 공개키가 추출됩니다.

Interactive Shell
$ sudo openssl req -new -key /etc/ssl/private/server.key.master -out /etc/ssl/archive/server.csr
Country Name (2 letter code) [AU]: KR
State or Province Name (full name) [Some-State]: Gyeonggi-do
Locality Name (eg, city) []: Yongin-si
Organization Name (eg, company) [Internet Widgits Pty Ltd]: Megastudy IT
Common Name (e.g. server FQDN or YOUR name) []: www.example.com
STEP 04

서비스용 키(Service Key) 추출 및 권한 최적화

서버 재시작 시 자동화를 위해 마스터 키로부터 비밀번호가 없는 키를 추출하고, ACL(Access Control List, 접근 제어 목록) 권한을 400으로 엄격히 제한합니다.

Security Hardening
# 비밀번호 제거된 구동용 키 추출
$ sudo openssl rsa -in /etc/ssl/private/server.key.master -out /etc/ssl/private/server.key.service
# 소유자만 읽기 가능하도록 권한 설정
$ sudo chown root:root /etc/ssl/private/server.key.service
$ sudo chmod 400 /etc/ssl/private/server.key.service
STEP 05

인증서 발급 및 아파치 무결성 검증 (configtest)

인증서를 발급하고 아파치 설정을 적용합니다. 기존 서비스 중단을 방지하기 위해 configtest를 반드시 먼저 수행합니다.

Final Validation
# 자가 서명 인증서 생성
$ sudo openssl x509 -req -days 365 -in /etc/ssl/archive/server.csr -signkey /etc/ssl/private/server.key.master -out /etc/ssl/certs/server.crt
# 설정 무결성 검사 (Downtime 방지 필수 단계)
$ sudo apache2ctl configtest
Syntax OK
# 안전하게 서비스 적용
$ sudo a2ensite default-ssl && sudo systemctl restart apache2

4. 아키텍트 인사이트 (Expert Insight)

Zero-Downtime(무중단 운영) 전략

사이트 활성화(a2ensite)를 구성 요소 준비 전에 실행하면, 아파치 재시작 시 인증서 경로 오류로 인해 기존 HTTP(80) 서비스까지 중단됩니다. 반드시 인증서 준비 → 설정 수정 → configtest → restart 순서를 엄수해야 합니다.

보안 비용 및 과금 유의사항

실제 클라우드 환경(AWS 등)에서 ALB (Application Load Balancer)를 통해 HTTPS를 운영할 경우, 시간당 약 $0.0225의 비용과 데이터 처리량에 따른 추가 과금이 발생합니다. ACM (AWS Certificate Manager) 공인 인증서 자체는 무료로 제공됩니다.
