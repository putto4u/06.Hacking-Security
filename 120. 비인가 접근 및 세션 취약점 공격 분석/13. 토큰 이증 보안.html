1. JWT 아키텍처 개요 (JWT Architecture Overview)

JWT(JSON Web Token, 자바스크립트 객체 표기법 기반 토큰)는 당사자 간에 정보를 JSON 객체로 안전하게 전송하기 위한 개방형 표준(RFC 7519)입니다. 서버에 상태를 저장하지 않는 Stateless(비상태성, 서버가 클라이언트의 상태를 보존하지 않는 성질) 특성 덕분에 분산 환경 및 MSA(Microservices Architecture, 마이크로서비스 아키텍처)에서 필수적인 인증 수단으로 자리 잡았습니다.

[도식: JWT의 3단계 논리 구조 (Logical Structure of JWT)]

헤더 (Header)

알고리즘 및 토큰 유형
{"alg": "HS256", "typ": "JWT"}

.
페이로드 (Payload)

사용자 권한 및 정보(Claims)
{"sub": "123", "admin": true}

.
서명 (Signature)

무결성 검증용 해시값
HMACSHA256(H+P, secret)

※ 주의: Base64Url로 인코딩된 헤더와 페이로드는 암호화된 것이 아니므로, 누구나 복호화하여 내용을 볼 수 있습니다. 민감한 개인정보를 페이로드에 포함해서는 안 됩니다.

2. 주요 보안 취약점 분석 (Security Vulnerabilities)

HIGH RISK

None 알고리즘 공격

일부 취약한 라이브러리는 헤더의 alg 값을 none으로 변경할 경우 서명 검증 과정을 생략합니다. 공격자는 서명부를 제거하고 페이로드의 권한을 admin: true로 조작하여 인증을 우회할 수 있습니다.

{"alg": "none", "typ": "JWT"}
서명 없이 전송 시 서버가 유효한 토큰으로 오판
MEDIUM RISK

취약한 비밀키 대입 공격

HS256과 같은 대칭키 알고리즘을 사용할 때, Secret Key(비밀키, 서명에 사용되는 공통 키)가 너무 짧거나 예측 가능하다면 공격자가 오프라인 상태에서 무차별 대입(Brute-force) 공격으로 키를 탈취할 수 있습니다.

$ hashcat -m 16500 jwt.txt rockyou.txt
단축키(예: 1234, password) 사용 시 수초 내 탈취 가능

3. 실증 및 시큐어 구현 (Implementation & Secure Coding)

01

알고리즘 명시적 제한 (Explicit Algorithm Restriction)

토큰을 검증할 때 라이브러리에 맡기지 말고, 허용된 알고리즘을 코드로 명시하여 Algorithm Confusion(알고리즘 혼동 공격, 다른 알고리즘으로 위장하여 서명을 우회하는 기법)을 방지합니다.

Python PyJWT
import jwt

# BAD: 알고리즘을 제한하지 않음
decoded = jwt.decode(token, key)

# GOOD: 알고리즘을 HS256으로 고정하여 검증
decoded = jwt.decode(token, key, algorithms=["HS256"])
02

비대칭 암호화 권장 (RS256 vs HS256)

대칭키(HS256)는 모든 서버가 동일한 비밀키를 가져야 하므로 키 노출 시 치명적입니다. RS256(비대칭 암호화, 공개키로 검증하고 비밀키로만 서명하는 방식)을 사용하면 인증 서버만 비밀키를 소유하므로 보안성이 대폭 강화됩니다.

인증 서버 (Auth Server)
Private Key로 서명
API 서버 (Resource Server)
Public Key로 검증만 수행

4. 아키텍트 인사이트 (Expert Insight)

토큰 무효화 메커니즘 (Revocation Strategy)

JWT는 기본적으로 서버에서 상태를 관리하지 않으므로, 토큰이 탈취되었을 때 서버 측에서 강제로 로그아웃(무효화)시키기가 매우 어렵습니다. 이를 해결하기 위해 Access Token(접근 토큰, 수명이 짧은 토큰)과 Refresh Token(갱신 토큰, 접근 토큰을 재발급받기 위한 장기 토큰)을 분리하고, Redis와 같은 인메모리 저장소에 블랙리스트를 관리하는 전략이 필수적입니다.

클라우드 키 관리 서비스 활용 (KMS Integration)

JWT 서명용 키를 소스코드나 환경 변수에 저장하는 행위는 지양해야 합니다. AWS KMS(AWS Key Management Service, 아마존 웹 서비스에서 제공하는 관리형 암호화 키 서비스)를 연동하여 하드웨어 보안 모듈(HSM)에서 키를 안전하게 관리하고, 서명 프로세스를 KMS API 호출로 대체함으로써 키 유출 위험을 원천 차단할 수 있습니다.

보안 서비스 비용 분석 (Cost Notice)

AWS KMS를 사용할 경우 고객 관리형 키(KMS Key)당 월 $1.00의 유지비가 발생하며, 서명 및 검증 시 발생하는 API 호출 10,000건당 $0.03이 과금됩니다. 토큰 검증 횟수가 매우 많은 대규모 서비스에서는 성능과 비용 최적화를 위해 공개키를 로컬 캐싱하는 전략을 병행해야 합니다.
