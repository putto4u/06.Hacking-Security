Identification & Authentication Failures

1. 세션 관리 아키텍처 (Session Management Architecture)

HTTP(Hypertext Transfer Protocol, 하이퍼텍스트 전송 프로토콜)는 본래 **비상태성(Stateless, 상태를 유지하지 않는 성질)**을 가집니다. 이를 보완하여 사용자의 로그인 상태를 유지하기 위해 도입된 개념이 **세션(Session, 서버와 클라이언트 간의 연결 상태)**입니다. 서버는 사용자를 식별하기 위한 고유한 **SID(Session Identifier, 세션 식별자)**를 생성하여 쿠키를 통해 클라이언트에 전달합니다.

[도식: 정상적인 세션 수명 주기 (Session Life-cycle Flow)]

1. 인증 시도

ID/PW 제출 및 검증

2. SID 생성/발급

Set-Cookie: SID=xyz...

3. 상태 유지

Cookie 헤더로 SID 전송

4. 세션 만료

로그아웃/타임아웃 처리

※ 설명: 서버는 메모리나 DB(Database, 데이터베이스), 또는 Redis(레디스, 인메모리 데이터 구조 저장소)와 같은 세션 저장소에 SID를 키로 하여 사용자 정보를 매핑합니다. 만약 SID가 유출되거나 추측 가능하다면, 공격자는 패스워드 없이 사용자로 위장할 수 있습니다.

2. 세션 하이재킹 (Session Hijacking, 세션 가로채기)

세션 하이재킹(Session Hijacking, 세션 가로채기)은 공격자가 활성화된 사용자의 세션을 가로채어 해당 사용자의 권한으로 시스템에 접근하는 공격 기법입니다. 이는 주로 네트워크 스니핑, 교차 사이트 스크립팅, 또는 취약한 세션 식별자 생성 로직을 통해 발생합니다.

[도식: XSS를 이용한 세션 하이재킹 흐름 (XSS-based Hijacking)]

사용자 (Victim)

스크립트 실행: document.cookie

공격자 (Attacker)
주요 공격 벡터 (Attack Vectors)
  • • Sniffing(스니핑, 가로채기): 암호화되지 않은 HTTP 통신에서 패킷을 캡처하여 SID 탈취
  • • XSS(Cross-Site Scripting, 교차 사이트 스크립팅): 악성 스크립트를 삽입하여 document.cookie를 통해 SID를 공격자 서버로 전송
  • • Side-jacking(사이드재킹): HTTP 세션 쿠키를 가로채어 HTTPS로 보호되는 영역 외에서 세션을 탈취
대응 방안 (Countermeasures)
  • • 전체 통신 구간 HTTPS(보안 하이퍼텍스트 전송 프로토콜) 의무화
  • • 쿠키 보안 속성(HttpOnly, Secure) 강제 적용
  • • 사용자의 IP 주소나 User-Agent(사용자 에이전트, 브라우저 식별 정보) 변화 감지 시 세션 종료

3. 세션 고정 (Session Fixation, 세션 식별자 고정 공격)

세션 고정(Session Fixation, 세션 식별자 고정 공격)은 공격자가 미리 생성한 유효한 SID를 사용자에게 강제로 심어놓고, 사용자가 해당 SID로 로그인하도록 유도하여 세션을 공유하는 공격입니다. 하이재킹이 '탈취'라면, 고정은 '미리 설계된 함정'에 가깝습니다.

[도식: 세션 고정 공격 3단계 (Session Fixation Steps)]

01
SID 발급 (Attacker)

공격자가 서버에 접속하여 유효한 SID(예: ID=666)를 발급받음

02
SID 주입 (Fixation)

피싱 메일 등을 통해 사용자에게 해당 SID가 포함된 링크 클릭 유도

03
로그인 및 탈취

사용자가 해당 SID 상태에서 로그인 성공 시, 공격자도 동일 권한 획득

최선의 방어: 로그인 시 세션 재생성 (Session Re-generation)

가장 근본적인 해결책은 사용자가 성공적으로 인증을 마칠 때마다, 기존에 사용하던 SID를 폐기하고 새로운 SID를 발급하는 것입니다.

# Python Flask 기반의 안전한 세션 갱신 예시
from flask import session

def login():
    if authenticate_user(username, password):
        # 기존 세션 데이터를 유지하면서 SID만 교체
        session.regenerate() 
        # 또는 세션을 완전히 비우고 다시 설정
        session.clear()
        session['user_id'] = user.id
        return "Success"

4. 쿠키 보안 속성 마스터 (Cookie Security Attributes)

브라우저가 쿠키를 다루는 방식을 제어하는 보안 속성들을 올바르게 설정하는 것은 세션 보안의 마지막 방어선입니다.

HttpOnly

JavaScript(자바스크립트)의 document.cookie를 통한 쿠키 접근을 원천 차단합니다. XSS 공격을 통한 세션 탈취를 방지하는 가장 핵심적인 속성입니다.

Essential for Session Cookies
Secure

쿠키가 오직 HTTPS 암호화 통신 채널을 통해서만 전송되도록 제한합니다. 네트워크 스니핑을 통한 평문 SID 노출을 방지합니다.

HTTPS Mandatory
SameSite

타사 사이트(Third-party)에서 전송되는 요청에 쿠키를 포함할지 여부를 결정합니다. CSRF(Cross-Site Request Forgery, 교차 사이트 요청 위조) 공격을 효과적으로 방어합니다.

Strict / Lax / None
[HTTP Header Example]
Set-Cookie: SID=abc123xyz789; HttpOnly; Secure; SameSite=Lax

※ 주석: SameSite=Lax는 현대 브라우저의 기본값이지만, 민감한 서비스(뱅킹 등)에서는 Strict 모드를 권장합니다. SameSite=None 사용 시에는 반드시 Secure 속성이 동반되어야 합니다.

5. 아키텍트 인사이트 (Architectural Insights)

세션 타임아웃 및 동시성 제어

세션은 절대 영구적이지 않아야 합니다. **Idle Timeout(유휴 타임아웃, 일정 시간 활동 없을 시 종료)**과 **Absolute Timeout(절대 타임아웃, 활동 여부와 관계없이 특정 시간 후 종료)**을 이중으로 설정하십시오. 또한, 동일 계정의 **중복 로그인 제한** 정책을 통해 공격자가 탈취한 세션을 장기간 유지하는 것을 방지해야 합니다.

클라우드 인프라에서의 세션 보호

AWS(Amazon Web Services, 아마존 웹 서비스) 환경에서는 **ALB(Application Load Balancer, 애플리케이션 부하 분산 장치)**의 **Sticky Sessions(스티키 세션, 세션 유지 기능)** 설정을 통해 부하 분산 환경에서도 세션 일관성을 유지할 수 있습니다.

보안 서비스 비용 분석 (Cost Awareness)

세션 공격(예: 대규모 Brute-force 또는 세션 퍼징)을 차단하기 위해 **AWS WAF(Web Application Firewall, 웹 애플리케이션 방화벽)**를 도입할 경우, **Web ACL(액세스 제어 목록)당 월 $5.00**, **규칙당 월 $1.00**, 그리고 **100만 건의 요청당 $0.60**의 비용이 발생합니다. 대규모 디도스 보호를 위한 **AWS Shield Advanced(쉴드 어드밴스드)**는 **월 $3,000.00**의 고정 비용이 발생하므로 예산 수립 시 반드시 고려해야 합니다.
