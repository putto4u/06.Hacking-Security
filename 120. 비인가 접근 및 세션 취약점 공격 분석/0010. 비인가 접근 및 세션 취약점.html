1. BAC (Broken Access Control, 깨진 접근 제어) 아키텍처

Access Control(접근 제어)는 사용자가 승인된 권한 범위 내에서만 자원에 접근할 수 있도록 강제하는 보안 메커니즘입니다. 이 설계가 실패하면 공격자는 Unauthorized Access(비인가 접근)를 통해 민감한 데이터에 접근하거나 타인의 권한을 도용하게 됩니다.

[도식 1: 접근 제어 판정 메커니즘 (Authorization Decision Mechanism)]

Subject (주체)
Policy Enforcement Point
VULNERABILITY
Resource (자원)
* 주체가 자원에 접근할 때 Enforcement Point(강제 지점)에서 권한 검증이 누락되는 것이 BAC의 핵심입니다.
* Authorization(권한 승인): 인증된 사용자가 특정 리소스에 대해 수행할 수 있는 동작(Read, Write, Delete 등)을 결정하는 과정입니다.
* Policy Enforcement Point(정책 강제 지점): 접근 제어 정책이 실제로 적용되고 검증되는 애플리케이션 내의 논리적 위치입니다.

2. Privilege Escalation (권한 상승) 매트릭스

Horizontal

수평적 권한 상승

Horizontal Privilege Escalation(수평적 권한 상승)은 동일한 권한 수준을 가진 사용자 계정 간에 발생합니다. 공격자가 자신의 자격 증명을 사용하여 타인의 데이터에 접근하는 취약점입니다.

  • IDOR(Insecure Direct Object Reference)를 통한 타인 정보 조회
  • 계정 식별자 조작을 통한 비밀번호 변경 시도
Vertical

수직적 권한 상승

Vertical Privilege Escalation(수직적 권한 상승)은 낮은 권한의 사용자가 관리자(Admin) 권한이나 상위 역할을 획득하는 경우입니다. 시스템 전체를 장악할 수 있는 치명적인 위험이 있습니다.

  • Hidden URL(숨겨진 URL) 직접 접근을 통한 관리 페이지 노출
  • 파라미터 조작을 통한 Role(역할) 값 강제 변경

3. Session Management Vulnerabilities (세션 관리 취약점)

HTTP 프로토콜은 Stateless(상태 비보존) 특성을 가지므로, 통신 상태를 유지하기 위해 Session ID(세션 식별자)를 사용합니다. 이 식별자가 탈취되거나 예측 가능하면 공격자는 적법한 사용자로 위장할 수 있습니다.

Case 01

Session Hijacking

Session Hijacking(세션 가로채기)는 이미 활성화된 사용자의 세션 토큰을 네트워크 스니핑이나 XSS 공격을 통해 탈취하는 기법입니다.

Case 02

Session Fixation

Session Fixation(세션 고정)은 공격자가 미리 발급받은 세션 ID를 희생자에게 주입하고, 희생자가 로그인한 후 그 세션을 그대로 공유하여 권한을 탈취합니다.

Case 03

Weak Session Token

Insufficient Session ID Entropy(불충분한 세션 식별자 엔트로피)로 인해 세션 ID가 순차적이거나 예측 가능한 경우 무작위 대입 공격에 노출됩니다.

[도식 3: 세션 고정 공격 시퀀스 (Session Fixation Attack Sequence)]

Step 1: 공격자가 유효한 세션 ID(SID=666)를 먼저 발급받음
Step 2: 희생자에게 해당 세션 ID가 포함된 링크 유도 (example.com?SID=666)
Step 3: 희생자가 해당 세션으로 로그인 (서버가 세션을 재생성하지 않음)
Final: 공격자가 동일한 SID=666으로 접속하여 희생자 권한 탈취
* Entropy(엔트로피): 정보 이론에서 무작위성이나 불확실성의 척도를 나타내며, 보안에서는 암호나 식별자가 얼마나 예측하기 어려운지를 의미합니다.
* XSS (Cross-Site Scripting, 사이트 간 스크립팅): 공격자가 악의적인 스크립트를 웹 페이지에 삽입하여 다른 사용자의 브라우저에서 실행되게 하는 공격입니다.

4. 실증 및 방어 구현 (Implementation & Defense)

STEP 01

IDOR(Insecure Direct Object Reference) 실증

공격자는 서버로 전송되는 객체 참조 값을 조작하여 권한 검증 누락을 확인합니다.

Proxy History
# Original Request
GET /api/v1/user/profile?uid=1001 HTTP/1.1
# Manipulated Request
GET /api/v1/user/profile?uid=1002 HTTP/1.1
Response: 200 OK (Data of User 1002 Leak!)
STEP 02

보안 쿠키 설정 (Secure Cookie Attributes)

세션 탈취를 방지하기 위해 서버 측에서 쿠키 속성을 엄격하게 제어해야 합니다.

Backend Config
Set-Cookie: SessionID=xyz789;
HttpOnly; // JS 접근 금지
Secure; // HTTPS 필수
SameSite=Strict; // CSRF 방어

5. 아키텍트 인사이트 (Architect Insight)

Defense in Depth (방어 계층화)

접근 제어는 반드시 서버 측에서 검증되어야 합니다. 클라이언트 측 UI에서 버튼을 숨기는 것은 Security by Obscurity(모호함을 통한 보안)일 뿐이며, API 엔드포인트 자체에서 세션 사용자의 역할(Role)과 자원 소유권(Ownership)을 매번 대조해야 합니다.

Zero Trust Architecture (제로 트러스트)

사용자가 일단 인증되었다고 해서 모든 내부 리소스를 신뢰해서는 안 됩니다. "Never Trust, Always Verify" 원칙에 따라, 매 요청 시 마다 토큰의 유효성과 자원에 대한 정당한 권한을 재확인하는 로직을 아키텍처 수준에서 공통 필터(Filter)로 구현하십시오.
