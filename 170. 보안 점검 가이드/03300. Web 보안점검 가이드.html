주요정보통신기반시설 기술적 취약점 분석·평가 기준을 충족하기 위한 전문가 수준의 서버 보안 설정 및 인프라 구축 방안을 다룹니다.

1. 웹 서비스 SSI 사용 제한

개요: 웹 서비스 내 SSI (Server-Side Includes, 서버 사이드 인클루드) 사용 제한 설정 여부를 점검합니다.

점검 목적: 웹 서비스 내 SSI (Server-Side Includes, 서버 사이드 인클루드) 사용을 제한하여 불법적인 데이터 접근을 차단하고 웹 서버의 보안을 강화하기 위함입니다.

보안 위협: 웹 서비스 내 SSI (Server-Side Includes, 서버 사이드 인클루드) 사용을 제한하지 않을 경우, 공격자가 SSI 기능을 이용하여 시스템 명령 실행 및 중요 파일 탈취 등의 공격이 가능하며, 이를 통해 서버 시스템 침해, 데이터 유출 등이 발생할 위험이 존재합니다. 또한, SSI 공격 시 HTML 페이지에 스크립트를 삽입하거나 원격으로 코드를 실행하여 웹 서비스를 악용할 위험이 존재합니다.

점검 대상 및 판단 기준

  • 대상: Apache, Tomcat, Nginx, WebtoB
  • 양호: 웹 서비스 SSI 사용 설정이 비활성화되어 있는 경우
  • 취약: 웹 서비스 SSI 사용 설정이 활성화되어 있는 경우

전문 용어 해설

SSI (Server-Side Includes, 서버 사이드 인클루드): CGI (Common Gateway Interface, 공용 게이트웨이 인터페이스) 프로그램을 작성하거나 혹은 서버 사이드 스크립트를 사용하는 언어로, 웹 서버가 사용자에게 페이지를 제공하기 전에 구문을 해석하도록 지시하는 역할을 합니다.

SSI Injection (Server-Side Includes Injection, 서버 사이드 인클루드 인젝션): HTML 문서 내 입력받은 변수값을 서버 측에서 처리할 때 부적절한 명령문이 포함 및 실행되어 서버의 데이터가 유출되는 취약점입니다.

아키텍트 실전 팁

SSI 모듈은 과거 정적 웹페이지 시절에 동적인 구성을 위해 사용되었으나, 현대적인 프레임워크 환경에서는 거의 사용되지 않는 레거시 기능입니다. 불필요하게 활성화되어 있을 경우 공격 표면만 넓히게 되므로, 초기 인프라 서버 프로비저닝 단계에서부터 원천적으로 비활성화하도록 템플릿화 시키는 것이 아키텍처 관점에서 가장 안전합니다.

2. SSL/TLS 활성화

개요: 서버와 클라이언트 간 통신 시 데이터 암호화 전송 여부를 점검합니다.

점검 목적: 서버와 클라이언트 간 통신 시 데이터의 평문 전송을 사용하지 않고 데이터가 암호화되는 SSL/TLS 인증 암호화 접속을 통해 스니핑을 통한 정보 유출의 위험을 방지하기 위함입니다.

보안 위협: 웹상의 데이터 통신 시 서버와 클라이언트 간에 데이터를 평문 전송하는 경우, 간단한 도청(스니핑)을 통해 정보가 탈취 및 도용될 위험이 존재합니다. 데이터가 암호화되지 않아 공격자가 중간에서 데이터를 가로채거나 도청할 수 있으며, 평문으로 전송되어 중간에서 변경될 우려가 있어 데이터의 정확성이 훼손될 위험이 존재합니다.

점검 대상 및 판단 기준

  • 대상: Apache, Nginx, WebtoB
  • 양호: SSL/TLS 설정이 활성화되어 있는 경우
  • 취약: SSL/TLS 설정이 비활성화되어 있는 경우

전문 용어 해설

Sniffing (스니핑, 도청): 스니퍼(Sniffer)는 "컴퓨터 네트워크상에 흘러 다니는 트래픽을 엿듣는 도청장치"라고 말할 수 있으며 "스니핑"이란 이러한 스니퍼를 이용하여 네트워크상의 데이터를 도청하는 행위를 말합니다.

SSL (Secure Socket Layer, 보안 소켓 계층): 인터넷상에서 정보를 암호화하여 송수신하는 프로토콜입니다. 현재 인터넷에서 널리 쓰이고 있는 www, FTP 등의 데이터를 암호화하여, 프라이버시에 관한 정보나 신용카드 번호, 기업 비밀 등을 안전하게 송수신할 수 있습니다.

TLS (Transport Layer Security, 전송 계층 보안): SSL의 후속 버전으로, 인터넷상에서 통신 보안을 제공하기 위해 설계된 강력한 암호화 규약입니다.

평문 전송(HTTP)과 암호화 전송(HTTPS)의 차이

점검 및 조치 사례

1) Apache

Step 2) SSL 가상 호스트 설정에 SSL 인증서 설정 추가

default-ssl.conf
# 가상 호스트 SSL 설정 파일 편집
# vi /[Apache 설치 디렉터리]/sites-available/default-ssl.conf

<VirtualHost *:443>
  ServerAdmin webmaster@yourdomain.com
  ServerName yourdomain.com
  DocumentRoot /var/www/html

  # SSL 엔진을 활성화합니다.
  SSLEngine on

  # SSL 인증서와 키 파일의 경로를 명시합니다.
  SSLCertificateFile /path/to/your_domain_name.crt
  SSLCertificateKeyFile /path/to/your_domain_name.key
</VirtualHost>

2) Nginx

Step 1) Server 블록 내 SSL 인증서 설정 확인 및 추가

default
# Nginx 가상 호스트 파일 편집
# vi /[Nginx 설치 디렉터리]/sites-available/default

server {
  # 443 포트에 대해 SSL 처리를 지시합니다.
  listen 443 ssl;
  server_name yourdomain.com www.yourdomain.com;

  # 인증서 파일의 절대 경로를 기입합니다.
  ssl_certificate /path/to/your_domain_name.crt;
  ssl_certificate_key /path/to/your_domain_name.key;

  root /var/www/html;
  index index.html index.htm;
}

운영 환경 주의 사항 및 실전 팁

단순히 443 포트를 열고 SSL 인증서를 연결하는 것만으로는 충분하지 않습니다. 취약한 암호화 알고리즘(예: RC4, DES)이나 과거의 프로토콜 버전(SSLv3, TLS 1.0, TLS 1.1)이 허용되어 있다면, BEAST나 POODLE과 같은 세션 하이재킹 공격에 여전히 노출됩니다. 완벽한 보안 인프라를 위해서는 최신 TLS 1.2 또는 1.3만을 사용하도록 제한하고, 강력한 Cipher Suite를 명시적으로 지정하여 암호화 강도를 높여야 합니다.

3. HTTP 리디렉션

개요: 웹 서비스 접근 시 HTTP Redirection (리디렉션, 자동 연결) 활성화 여부를 점검합니다.

점검 목적: HTTP 차단 및 HTTPS로 Redirection 활성화를 통해 평문 전송되는 데이터를 암호화하여 공격자의 데이터 스니핑에 대비하기 위함입니다.

보안 위협: HTTP 통신은 암호화 전송이 아닌 평문 전송을 하므로 공격자가 스니핑을 시도할 경우 관리자의 ID, 비밀번호가 노출되어 악의적 사용자가 관리자 계정을 탈취할 수 있는 위험이 존재합니다.

점검 대상 및 판단 기준

  • 대상: Apache, Nginx, WebtoB
  • 양호: HTTP 접근 시 HTTPS Redirection이 활성화된 경우
  • 취약: HTTP 접근 시 HTTPS Redirection이 비활성화된 경우

전문 용어 해설

HTTP Redirection (리디렉션, 자동 연결): 클라이언트가 웹 서버에 특정 URL을 요청했을 때, 서버가 클라이언트에게 다른 URL로 이동하라고 지시하는 메커니즘입니다. 주로 보안 통신을 강제하기 위해 HTTP(80) 요청을 HTTPS(443) 요청으로 자동 변환하는 데 사용됩니다.

HTTP 요청 시 HTTPS로 강제 리디렉션 (301/307) 통신 흐름도

브라우저
1. GET http://domain.com (Port 80)
웹 서버
2. 301 Moved Permanently
Location: https://domain.com
3. GET https://domain.com (Port 443 / TLS 암호화 터널)

점검 및 조치 사례

1) Apache

Step 4) RewriteRule을 이용한 SSL 가상 호스트 설정 방법

default-ssl.conf
<VirtualHost *:80>
  ServerName yourdomain.com

  # URL 재작성 엔진을 켭니다.
  RewriteEngine On

  # HTTPS가 아닌 경우에 한하여 규칙을 실행합니다.
  RewriteCond %{HTTPS} off

  # 동일 호스트와 URI를 유지한 채 HTTPS로 301 리디렉션 시킵니다.
  RewriteRule ^ https://%{HTTP_HOST}%{REQUEST_URI} [L,R=301]
</VirtualHost>

2) Nginx

Step 1) Server 블록 내 HTTPS Redirection 설정 확인 및 추가

default
server {
  # 기본 HTTP 포트를 수신 대기합니다.
  listen 80;
  server_name yourdomain.com www.yourdomain.com;

  # HTTPS 프로토콜로 301 상태 코드를 반환하며 리턴시킵니다.
  return 301 https://$host$request_uri;
}

고급 아키텍처 방어 전략

단순 301이나 302 리디렉션 설정만으로는 브라우저가 최초 80 포트로 접속하는 그 찰나의 순간에 이루어지는 세션 하이재킹(SSL Stripping 공격 등)을 100% 방어할 수 없습니다. 이를 완벽하게 보완하기 위해서는 웹 서버 응답 헤더에 HSTS (HTTP Strict Transport Security, HTTP 엄격 전송 보안) 정책을 반드시 추가하여 브라우저가 향후 일정 기간 동안 무조건 HTTPS 연결만을 시도하도록 강제해야 합니다.

4. 에러 페이지 관리

개요: 에러 페이지 내 불필요 정보 노출 여부 및 필수 에러 코드에 대해 일원화된 에러 페이지 관리 여부를 점검합니다.

점검 목적: 에러 페이지에서 웹 서버 버전 및 종류, OS 정보 등 웹 서버와 관련된 불필요한 정보 및 에러 코드를 통한 기술적 취약점이 노출되는 것을 최소화하기 위함입니다.

보안 위협: 에러 페이지에서 불필요한 정보가 노출될 경우 공격자에 의해 해당 버전의 알려진 취약점 등을 이용하여 시스템 구조와 특성 노출 및 해당 취약점을 통한 공격의 위험이 존재합니다. 필수 에러 코드에 대해 일원화된 에러 페이지로 관리하지 않는 경우 에러 코드를 통해 각종 정보 유추의 위험이 존재합니다.

점검 대상 및 판단 기준

  • 대상: Apache, Tomcat, Nginx, JEUS, WebtoB
  • 양호: 웹 서비스 에러 페이지가 별도로 지정된 경우
  • 취약: 별도로 지정되지 않거나 중요 정보가 노출되는 경우

전문 용어 해설

HTTP Status Code (HTTP 상태 코드): 클라이언트의 요청에 대한 웹 서버의 처리 결과를 알려주는 세 자리 숫자입니다. 400(Bad Request, 잘못된 요청), 403(Forbidden, 권한 없음), 404(Not Found, 찾을 수 없음), 500(Internal Server Error, 서버 내부 오류) 등이 대표적입니다. 에러 관리가 미흡하면 이 코드를 통해 시스템 내부 정보가 유출될 수 있습니다.

기본 에러 페이지 노출로 인한 서버 정보 유출과 커스텀 에러 페이지 비교

시스템 내부 정보가 완벽하게 은닉됨

점검 및 조치 사례

1) Nginx

Step 1) nginx.conf 파일 내 에러 코드별 에러 페이지 설정 추가

nginx.conf
server {
  # 특정 에러 코드들을 공통된 사용자 지정 HTML 파일로 리다이렉트 처리합니다.
  error_page 400 401 403 404 /error.html;
  error_page 500 502 503 504 /error.html;

  # 에러 페이지 자체에 대한 로케이션 블록을 명시합니다.
  location = /error.html {
    root /var/www/html;
  }
}

2) Tomcat

Step 1) web.xml 파일 내 에러 코드별 에러 페이지 설정 추가

web.xml
<error-page>
  <!-- 에러 코드를 매핑합니다. -->
  <error-code>500</error-code>
  <!-- 커스텀 에러 페이지의 위치를 지정합니다. -->
  <location>/error.jsp</location>
</error-page>

개발 및 인프라 운영 실수 방지

웹 취약점 진단이나 인프라 구축 시, 개발자들이 404(Not Found) 에러 페이지만을 이쁘게 커스터마이징하고 500번대(Internal Server Error) 처리는 프레임워크 기본 설정에 맡겨두는 실수를 빈번하게 범합니다. 애플리케이션에 예외(Exception)가 발생하여 500 에러가 떨어질 때, 화면에 Java Stack Trace 구문이나 민감한 DB Query 문이 그대로 출력되면 해커에게 백엔드 서버의 완벽한 아키텍처 청사진을 제공하게 됩니다. 따라서 java.lang.Exception과 같은 최상위 예외 처리 매핑을 통해 예기치 못한 모든 오류를 잡아내고 공통된 보안 페이지로 우회시켜 내부 로직 노출을 완벽하게 차단해야 합니다.

5. LDAP 알고리즘 적절하게 구성

개요: LDAP (Lightweight Directory Access Protocol, 경량 디렉터리 액세스 프로토콜) 연결 시 취약한 알고리즘 사용 제한 설정 여부를 점검합니다.

점검 목적: LDAP 연결 시 안전한 비밀번호 다이제스트 알고리즘을 사용하여 비밀번호 평문 전송 시 발생할 수 있는 스니핑 등의 공격에 대비하기 위함입니다.

보안 위협: 취약한 다이제스트 알고리즘을 사용하는 경우 공격자의 스니핑, 무차별 공격 등을 통해 인증 정보가 노출될 위험이 존재합니다.

점검 대상 및 판단 기준

  • 대상: Tomcat
  • 양호: 안전한 비밀번호 다이제스트 알고리즘을 사용하는 경우
  • 취약: 안전한 비밀번호 다이제스트 알고리즘을 사용하지 않는 경우

전문 용어 해설

LDAP (Lightweight Directory Access Protocol, 경량 디렉터리 액세스 프로토콜): 네트워크상에서 조직이나 개인정보 혹은 파일이나 디바이스 정보 등을 통합적으로 조회하고 관리하는 중앙 집중형 표준 통신 프로토콜입니다.

Digest Algorithm (다이제스트 알고리즘, 해시 함수): 임의의 길이를 가진 입력 데이터를 고정된 길이의 무작위 해시값(다이제스트)으로 변환하는 일방향 암호화 기술입니다. 역연산(복호화)이 불가능해야 데이터 무결성을 보장할 수 있습니다. (예: SHA-256)

해시 알고리즘 강도 비교 (MD5/SSHA vs SHA-256)

MD5 / SSHA
P@ssw0rd1
5e8848...
레인보우 테이블로 1초 만에 해독 가능
SHA-256 + Salt
P@ssw0rd1
e3b0c4... (64 chars)
현대 컴퓨팅 파워로 역연산(복호화) 불가

점검 및 조치 사례

1) Tomcat

Step 2) 비밀번호 다이제스트 알고리즘 설정 변경

server.xml
# Tomcat 메인 환경 파일을 엽니다.
# vi /[Tomcat 설치 디렉터리]/conf/server.xml

<!-- digest 속성을 안전한 해시 알고리즘인 SHA-256으로 명시적으로 변경하여 기입합니다. -->
<Realm className="org.apache.catalina.realm.JNDIRealm"
       connectionURL="ldap://localhost:389"
       userPattern="uid={0},ou=people,dc=mycompany,dc=com"
       digest="SHA-256" />

레거시 연동 아키텍처 설계 팁

기존 레거시 AD(Active Directory, 액티브 디렉터리)나 사내 LDAP 서버와 연동 구축을 진행할 때, 이전 시스템 환경과의 하위 호환성을 이유로 MD5나 SSHA(Salted SHA-1)를 서버 설정에 그대로 유지하는 사례가 현업에서 자주 발생합니다. 그러나 이러한 해시 방식들은 레인보우 테이블과 현대적인 컴퓨팅 성능을 이용하면 아주 손쉽게 크래킹(Cracking, 비밀번호 해독) 공격에 무너집니다. 연동하는 파트너 서버 측 디렉터리 정책부터 SHA-256 또는 SHA-512 기반으로 필수 업그레이드하도록 아키텍처 수준에서 강제하는 것이 올바른 시스템 보안 설계의 핵심입니다.
