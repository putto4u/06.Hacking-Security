1. 아키텍처 개요 (Architecture Overview)

기업의 IT 인프라스트럭처에서 패치 관리와 로그 관리는 보안의 가장 기본적인 방어선이자 최후의 보루입니다. 운영체제 및 애플리케이션의 취약점을 선제적으로 제거하기 위해 일관된 패치 관리 시스템이 요구되며, 침해 사고 발생 시 역추적 및 법적 증거 자료 확보를 위해 무결성이 보장된 중앙 집중형 로그 아키텍처를 설계해야 합니다. 단일 노드에 로그를 보관하는 것은 공격자에 의한 로그 변조 및 삭제 위험에 노출되므로, 아키텍트 관점에서는 반드시 분리된 로그 수집 구간을 확보해야 합니다.

전문 용어 해설
  • CVE (Common Vulnerabilities and Exposures, 공통 취약점 및 노출): 대중에게 알려진 사이버 보안 취약점의 목록과 고유 식별 번호를 의미합니다. 패치 관리는 이 CVE 리스트에 대응하는 과정입니다.
  • SIEM (Security Information and Event Management, 보안 정보 및 이벤트 관리): 다양한 시스템과 애플리케이션에서 생성되는 보안 로그와 이벤트를 중앙에서 수집, 분석, 상관관계를 도출하여 위협을 탐지하는 엔터프라이즈 보안 솔루션입니다.

2. 실습 구성 요소 (Components)

NODE 01

타겟 운영 서버 (Web/App)

  • 운영체제: RHEL/CentOS 9 Stream 또는 Ubuntu 22.04 LTS
  • 역할: 보안 패치 대상 및 로그 송신지
  • 설치 도구: dnf-automatic (또는 unattended-upgrades), rsyslog, auditd
NODE 02

중앙 집중형 로그 서버

  • 운영체제: RHEL/CentOS 9 Stream 또는 Ubuntu 22.04 LTS
  • 역할: 다수의 노드로부터 로그를 안전하게 수집 및 보관
  • 수신 포트: TCP/UDP 514 포트 개방 및 방화벽 설정 통제
전문 용어 해설
  • LTS (Long Term Support, 장기 지원): 소프트웨어의 일반적인 수명 주기보다 훨씬 긴 기간 동안 보안 패치와 업데이트를 지원하는 릴리스 버전을 의미합니다. 엔터프라이즈 환경에서는 반드시 LTS 버전을 사용하여 시스템 안정성을 확보해야 합니다.

3. 실증 및 구현 (Implementation)

STEP 01

운영 서버의 보안 패치 자동화 설정

수동 패치는 관리자의 누락이나 지연으로 인해 '제로데이 공격(Zero-Day Attack)' 이후의 공격 표면을 제공하게 됩니다. RHEL 계열에서는 dnf-automatic을 사용하여 보안 패치만을 선별하여 자동으로 다운로드하고 설치하도록 구성할 수 있습니다. 운영 환경의 중단을 막기 위해 커널 패치는 재부팅 시점으로 연기하고, 라이브러리와 애플리케이션 보안 패치만 즉시 적용하는 전략이 필요합니다.

Node 01: Bash
$ sudo dnf install -y dnf-automatic
...
Complete!
$ sudo vi /etc/dnf/automatic.conf
# 파일 내 설정 변경 부분
[commands]
upgrade_type = security # 보안 패치만 적용
download_updates = yes
apply_updates = yes # 자동 설치 진행

[emitters]
emit_via = motd # 로그인 시 터미널 화면에 결과 출력
$ sudo systemctl enable --now dnf-automatic.timer
Created symlink /etc/systemd/system/timers.target.wants/dnf-automatic.timer ...
전문 용어 해설
  • Zero-Day Attack (제로데이 공격): 시스템의 보안 취약점이 발견되고 그에 대한 공식적인 패치가 배포되기 전(0일 차)에 이루어지는 해킹 공격입니다. 대응할 수 있는 패치가 없으므로 매우 치명적입니다.
  • MOTD (Message Of The Day, 오늘의 메시지): 관리자가 리눅스 서버에 SSH 등으로 원격 로그인할 때 화면에 가장 먼저 표시되는 안내 메시지입니다. 패치 적용 결과를 관리자에게 알리는 용도로 활용됩니다.
STEP 02

중앙 집중형 로그 서버 (Rsyslog Server) 구축

공격자가 서버에 침입하면 가장 먼저 시도하는 행위가 자신의 흔적을 지우기 위해 /var/log 내의 로그 파일을 변조하거나 삭제하는 것입니다. 이를 방지하기 위해 생성된 로그를 실시간으로 물리적으로 분리된 로그 서버로 전송해야 합니다. Node 02를 로그 전용 서버로 설정합니다. UDP는 속도가 빠르지만 유실 가능성이 있으므로 신뢰성 있는 TCP 프로토콜을 사용합니다.

Node 02: Log Server
$ sudo vi /etc/rsyslog.conf
# 주석 해제하여 TCP 514 포트 수신 활성화
module(load="imtcp")
input(type="imtcp" port="514")
# 클라이언트별로 디렉토리를 분리하여 로그 저장 템플릿 생성
$template RemoteLogs,"/var/log/remote/%HOSTNAME%/%PROGRAMNAME%.log"
*.* ?RemoteLogs
& ~ # 로컬 시스템 로그와 섞이지 않도록 처리 후 중단
$ sudo systemctl restart rsyslog
$ sudo firewall-cmd --permanent --add-port=514/tcp
$ sudo firewall-cmd --reload
success
전문 용어 해설
  • Rsyslog (Rocket-fast System for log processing, 초고속 시스템 로깅 프로세스): 리눅스 시스템에서 로그를 수집, 필터링, 저장, 라우팅하는 기본 서비스(데몬)입니다. 구형 syslog를 대체하여 고성능 및 다양한 프로토콜(TCP, TLS)을 지원합니다.
STEP 03

타겟 운영 서버에서 로그 원격 전송 설정

로그 서버가 준비되었으므로, 타겟 서버(Node 01)에서 발생하는 인증 실패, 시스템 이벤트 등의 중요 로그를 실시간으로 로그 서버(Node 02)로 보내도록 구성합니다. @ 기호 1개는 UDP 전송을, @@ 기호 2개는 TCP 전송을 의미합니다.

Node 01: Bash
$ sudo vi /etc/rsyslog.conf
# 파일 최하단에 다음 줄을 추가 (Node 02의 IP가 192.168.1.100일 경우)
# 보안(authpriv) 및 시스템 주요 로그(*.info)를 전송
*.info;mail.none;authpriv.none;cron.none @@192.168.1.100:514
authpriv.* @@192.168.1.100:514
$ sudo systemctl restart rsyslog

이 설정이 완료된 순간부터 Node 01에서 누군가 로그인을 시도하거나 실패하면, 해당 로그는 즉시 통신망을 타고 Node 02의 /var/log/remote/node01/sshd.log 파일에 안전하게 기록됩니다. 공격자가 타겟 서버를 장악하여 로컬의 /var/log/secure를 지우더라도, 원격 서버의 로그는 건드릴 수 없습니다.

STEP 04

Auditd(감사 데몬)를 활용한 파일 무결성 모니터링

일반적인 시스템 로그(syslog)만으로는 고도화된 해킹 시도를 탐지하기 부족합니다. 리눅스 커널 레벨에서 발생하는 시스템 호출(System Call)과 중요 파일의 권한 변경, 접근 이력을 감시하기 위해 auditd (감사 데몬)를 도입해야 합니다. 이는 시스템 아키텍처의 가시성을 극대화하는 강력한 도구입니다. 예를 들어, 리눅스의 계정 정보가 담긴 /etc/passwd나 /etc/shadow 파일을 누군가 읽거나 수정하려고 할 때 즉각 경고를 발생시킬 수 있습니다.

Node 01: Bash
$ sudo yum install -y audit
$ sudo systemctl enable --now auditd
# /etc/passwd 파일에 쓰기(w) 및 속성 변경(a) 행위가 발생하면 'passwd_change'라는 태그로 기록
$ sudo auditctl -w /etc/passwd -p wa -k passwd_change
# 감시 로그 검색 (aureport 또는 ausearch 활용)
$ sudo ausearch -k passwd_change
전문 용어 해설
  • Auditd (Linux Audit Daemon, 리눅스 감사 데몬): 보안 감사를 목적으로 시스템에서 발생하는 이벤트(프로세스 실행, 파일 접근, 네트워크 연결 등)를 상세하게 커널 단위에서 추적하고 로깅하는 시스템입니다. 미국 국방성(DoD) 보안 요건을 충족하기 위해 개발되었습니다.
  • System Call (시스템 호출): 사용자 공간(User Space)의 응용 프로그램이 운영체제 커널(Kernel)에 특권이 필요한 작업(파일 입출력, 프로세스 생성 등)을 요청하기 위해 사용하는 프로그래밍 인터페이스입니다. 해커의 악의적 행위는 반드시 시스템 호출을 거치므로 이를 모니터링하는 것이 중요합니다.

4. 아키텍트 인사이트 (Expert Insight)

로그 관리의 핵심: 무결성 (Integrity)과 암호화

중앙 로그 서버를 구축했다고 끝나는 것이 아닙니다. 전송 중인 로그를 네트워크 스니핑(Sniffing)으로부터 보호하기 위해 Rsyslog의 TLS 암호화(Transport Layer Security) 기능을 구성해야 합니다. 특히 클라우드 환경이나 퍼블릭 망을 통과해야 하는 아키텍처라면 필수적입니다.

또한 저장된 로그 파일에 chattr +a 명령어를 적용하여 오직 추가(Append)만 가능하도록(수정 및 삭제 불가) 설정하는 것이 아키텍트가 놓치지 말아야 할 실전 보안 강화 팁입니다.

로그 폭탄으로 인한 디스크 고갈 공격 경고

보안 가시성을 높이려고 모든 로그를 수집하다 보면, 공격자가 악의적으로 방대한 트래픽이나 오류를 발생시켜 서버의 /var 파티션 디스크 용량을 100% 꽉 채우게 만드는 서비스 거부 공격(DoS)을 유발할 수 있습니다. 운영 체제에 여유 공간이 없어지면 시스템 데몬이 멈추고 장애가 발생합니다.

반드시 로그 서버 및 운영 서버에 logrotate(로그 순환 도구)를 세심하게 설정하여, 오래된 로그는 압축하거나 아카이빙 스토리지 계층으로 넘기도록(Data Lifecycle Management) 설계해야 합니다.

전문 용어 해설
  • TLS (Transport Layer Security, 전송 계층 보안): 인터넷 상의 통신에 대해 기밀성(데이터 암호화), 데이터 무결성, 인증을 제공하는 암호화 프로토콜입니다. 기존 SSL의 안전한 후속 버전입니다.
  • Sniffing (스니핑): 네트워크 상에 지나다니는 패킷을 엿보거나 도청하여 계정 정보나 중요 데이터를 가로채는 해킹 기법입니다.
