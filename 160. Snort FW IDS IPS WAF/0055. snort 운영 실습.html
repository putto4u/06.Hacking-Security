1. IDS 아키텍처 개요 (Intrusion Detection System Architecture Overview)

Snort는 공개 소스 기반의 NIDS(Network Intrusion Detection System, 네트워크 침입 탐지 시스템)로서, 네트워크 트래픽을 실시간으로 수집하고 분석하여 공격 징후를 포착합니다. Snort의 내부 구조는 Packet Decoder(패킷 디코더), Preprocessors(전처리기), Detection Engine(탐지 엔진), Logging/Alerting(로깅 및 알람) 시스템으로 구성됩니다. 각 패킷은 수집된 즉시 디코딩 과정을 거쳐 프로토콜별로 분류되며, 전처리기에 의해 조각난 패킷의 재조합(Reassembly)이나 이상 징후 정규화 과정을 거친 뒤 탐지 엔진에서 사용자가 정의한 Rule(Rule, 규칙) 세트와 매칭됩니다.

• NIDS(Network Intrusion Detection System, 네트워크 침입 탐지 시스템): 네트워크 전체 트래픽을 감시하여 침입을 탐지하는 시스템.
• Preprocessor(전처리기): 탐지 엔진이 작동하기 전, 패킷 데이터를 분석하기 쉬운 형태로 정규화하거나 조립하는 모듈.
• Normalization(정규화): 공격자가 탐지를 우회하기 위해 변조한 패킷 형식을 표준 형식으로 변환하는 과정.

2. 실무형 탐지 룰(Rule, 규칙) 제작 기초 5선

RULESET 01

Snort 룰은 Header(Header, 헤더)와 Options(Options, 옵션)으로 구성됩니다. 헤더는 처리 방식(Action), 프로토콜, 출발지/목적지 IP 및 포트를 정의하며, 옵션은 탐지할 데이터의 패턴을 명시합니다.

① 기초 ICMP 탐지 룰

모든 소스로부터 내부 망으로 유입되는 ICMP 패킷(Ping 등)을 탐지합니다.

alert icmp any any -> $HOME_NET any (msg:"ICMP Packet Detected"; sid:1000001; rev:1;)

② SSH 무차별 대입 공격 전조 탐지

TCP 22번 포트로 들어오는 연결 요청(SYN Flag)을 감시합니다.

alert tcp any any -> $HOME_NET 22 (msg:"SSH Connection Attempt"; flags:S; sid:1000002; rev:1;)

③ HTTP 웹 공격 문자열 탐지 (SQL Injection)

웹 요청 중 데이터베이스 조회 쿼리인 'SELECT' 문구가 포함된 패킷을 탐지합니다.

alert tcp any any -> $HOME_NET 80 (msg:"SQLi Attempt - SELECT"; content:"SELECT"; nocase; sid:1000003; rev:1;)

④ FTP 권한 남용(Root Login) 탐지

파일 전송 프로토콜에서 관리자 계정으로의 직접 접근 시도를 확인합니다.

alert tcp any any -> $HOME_NET 21 (msg:"FTP Root Login"; content:"USER root"; sid:1000004; rev:1;)

⑤ 대용량 패킷(DDoS 전조) 탐지

패킷의 데이터 크기가 1000바이트를 초과하는 비정상적인 ICMP 패킷을 필터링합니다.

alert icmp any any -> $HOME_NET any (msg:"Large ICMP Packet"; dsize:>1000; sid:1000005; rev:1;)
• Header(Header, 헤더): 규칙의 행동, 프로토콜, 주소 및 포트 정보가 포함된 앞부분.
• SID(Snort ID, 스노트 식별자): 각 룰을 고유하게 식별하는 번호. 사용자 정의 룰은 보통 1,000,000번부터 시작.
• nocase(No Case, 대소문자 미구분): 문자열 패턴 매칭 시 대소문자를 구별하지 않도록 설정하는 옵션.

3. Snort 가동, 중지 및 실시간 제어 (Implementation & Control)

STEP 01

서비스 제어 (systemctl)

백그라운드에서 Snort 데몬을 관리하기 위한 표준 명령입니다.

$ sudo systemctl start snort # 가동
$ sudo systemctl stop snort # 중지
$ sudo systemctl restart snort # 재가동
STEP 02

실시간 감시 및 콘솔 알람 출력

터미널 화면에 즉각적으로 탐지 결과를 출력하여 룰의 정확성을 테스트합니다.

$ sudo snort -A console -q -c /etc/snort/snort.conf -i ens33
[**] [1:1000001:1] ICMP Packet Detected [**] [Priority: 0] {ICMP} 192.168.1.5 -> 192.168.1.10
STEP 03

로그 관리 및 백업 (Logging & Backup)

로그 데이터를 영구적으로 보존하거나 가동 중에 로그 파일을 교체하는 방법입니다.

Hot Backup Script
# 1. 기존 로그 파일 이름 변경 (백업)
$ mv /var/log/snort/alert /var/log/snort/alert_$(date +%Y%m%d)
# 2. Snort 프로세스에 HUP 신호를 보내 새 로그 파일 생성 유도
$ kill -HUP $(pgrep snort)
• Daemon(데몬): 사용자가 직접 제어하지 않고 백그라운드에서 실행되는 프로세스.
• SIGHUP(Signal Hang Up, 가동 중 재시작 신호): 프로세스를 종료하지 않고 설정 파일을 다시 읽거나 로그를 초기화하도록 보내는 신호.
• Quiet Mode(Quiet Mode, 조용한 모드): 실행 시 배너 정보나 통계 정보를 생략하고 알람 정보만 표시하는 모드(-q).

4. 아키텍트 인사이트 (Expert Insight)

성능 최적화 (Performance Tuning)

운영 환경에서는 로그를 ASCII 형태보다는 Unified2(통합 바이너리 포맷)로 기록하는 것이 유리합니다. 이는 Snort의 오버헤드를 줄여 패킷 유실(Packet Loss)을 최소화하며, 추후 Barnyard2와 같은 도구를 통해 데이터베이스나 SIEM으로 전달하기 용이합니다.

클라우드 과금 경고 (Cloud Billing)

AWS(Amazon Web Services) 환경에서 Traffic Mirroring(트래픽 미러링) 기능을 사용하여 IDS를 운영할 경우, 미러링되는 데이터 양에 따라 VPC Traffic Mirroring 시간당 요금 및 데이터 전송 비용이 추가로 발생합니다. 대량의 트래픽이 발생하는 서비스에서는 과금 계획을 반드시 검토해야 합니다.
