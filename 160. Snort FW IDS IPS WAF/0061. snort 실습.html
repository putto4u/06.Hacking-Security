1. 아키텍처 개요 (Architecture Overview)

현대의 보안 위협은 지능형 지속 위협(APT, Advanced Persistent Threat, 특정 대상을 목표로 장기간 공격)의 형태로 진화하고 있습니다. 본 실습에서는 업계 표준인 Snort 3(스노트 3, 오픈소스 침입 탐지 시스템)와 Splunk(스플렁크, 데이터 분석 플랫폼)를 통합하여 실시간 탐지 파이프라인(Pipeline, 데이터 흐름 구조)을 구축합니다. Snort 3는 전통적인 텍스트 기반 규칙(Rule, 탐지 정책)과 유연한 Lua(루아, 경량 스크립팅 언어)를 결합하여 탐지 로직을 처리하며, 이를 JSON(JavaScript Object Notation, 데이터 교환 표준 형식)으로 출력하여 Splunk SIEM(Security Information and Event Management, 보안 정보 및 이벤트 관리)으로 전송함으로써 강력한 가시성(Visibility, 데이터 시각화 능력)을 확보합니다.

APT (Advanced Persistent Threat, 지능형 지속 위협): 특정한 대상을 목표로 하여 장기간에 걸쳐 다양한 수단을 동원해 지속적으로 공격하는 형태입니다.
SIEM (Security Information and Event Management, 보안 정보 및 이벤트 관리): 기업 내 산재한 보안 장비의 로그를 중앙에서 수집하여 분석하는 통합 보안 관제 플랫폼입니다.
JSON (JavaScript Object Notation, 자바스크립트 객체 표기법): 사람이 읽기 쉽고 기계가 파싱하기 용이한 데이터 전송 표준 형식입니다.

2. 실습 구성 요소 (Components)

NODE 01

Snort 3 IDS Sensor

  • OS: Ubuntu 22.04 LTS (롱텀 서포트 버전)
  • Snort 3: Multi-threading (멀티스레딩 지원 엔진)
  • Promiscuous Mode (무차별 모드 인터페이스)
Multi-threading (멀티스레딩): 하나의 프로세스 내에서 여러 작업을 동시에 수행하여 성능을 높이는 기술입니다.
Promiscuous Mode (무차별 모드): 네트워크 카드가 자신에게 할당되지 않은 패킷까지 모두 수신하도록 설정하는 모드입니다.
NODE 02

Splunk Enterprise

  • Splunk Enterprise (유료 라이선스 및 프리 한도 확인 필요)
  • HTTP Event Collector (데이터 수신 프로토콜)
  • SPL (Search Processing Language, 전용 검색 언어)
SPL (Search Processing Language, 검색 처리 언어): 스플렁크에서 데이터를 검색하고 시각화하기 위해 사용하는 전용 명령형 언어입니다.
주의: Splunk Enterprise는 일일 무료 수집 한도(500MB) 초과 시 유료 라이선스가 필요할 수 있습니다.

3. 실증 및 구현 (Implementation)

STEP 01

Splunk Enterprise 인프라 구축

분석 서버인 Splunk를 설치하고 시스템 데몬(Daemon, 백그라운드 서비스)으로 등록합니다.

Shell Console
# 패키지 설치 및 초기화
$ sudo dpkg -i splunk_package.deb
$ sudo /opt/splunk/bin/splunk enable boot-start --accept-license
$ sudo service splunk start
STEP 02

ICMP 정찰 공격 탐지 실습 (ICMP Echo Request)

네트워크 생존 여부를 확인하는 핑(Ping) 패킷을 탐지하고 분석합니다.

Rule Config (/etc/snort/rules/local.rules)
alert icmp any any -> $HOME_NET any (msg:"ICMP Ping Detected"; sid:1000001; rev:1;)
Victim: Snort Execution
$ sudo snort -c /etc/snort/snort.conf -R /etc/snort/rules/local.rules -A fast
Attacker: Kali Linux
$ ping -c 4 192.168.1.10
[로그 파일: /var/log/snort/alert_fast.txt]
03/31-08:35:12.450123 [**] [1:1000001:1] "ICMP Ping Detected" [**] [Priority: 0] {ICMP} 192.168.1.5 -> 192.168.1.10
분석: 출발지 1.5(공격자)에서 목적지 1.10(센서)으로 ICMP 패킷이 유입되어 sid 1000001 규칙이 정상 발동됨.
STEP 03

TCP SYN 포트 스캔 탐지 (Nmap Stealth Scan)

로그를 남기지 않으려는 SYN 스텔스(Stealth, 잠행) 스캔을 임계값(Threshold) 설정을 통해 탐지합니다.

Rule Config
alert tcp any any -> $HOME_NET any (msg:"Nmap SYN Scan Detected"; flags:S; threshold:type both, track by_src, count 10, seconds 1; sid:1000002; rev:1;)
Victim: Snort
$ sudo snort -c /etc/snort/snort.conf -A fast
Attacker: Nmap Scan
$ sudo nmap -sS 192.168.1.10
[로그 파일: /var/log/snort/alert_fast.txt]
03/31-08:37:05.112 [**] [1:1000002:1] "Nmap SYN Scan Detected" [**] {TCP} 192.168.1.5:45231 -> 192.168.1.10:80
분석: 1초 내에 10개 이상의 SYN 패킷이 유입됨을 감지하여 단순 접속이 아닌 자동화 스캔 행위로 판별함.
STEP 04

로컬 파일 포함 공격(LFI) 탐지

웹 취약점을 통해 시스템 민감 파일(/etc/passwd)에 접근하려는 시도를 차단합니다.

Rule Config
alert tcp any any -> $HOME_NET 80 (msg:"Possible LFI - /etc/passwd access"; content:"/etc/passwd"; http_uri; sid:1000003; rev:1;)
Victim Node
$ sudo snort -c /etc/snort/snort.conf -A fast
Attacker: cURL
$ curl "http://192.168.1.10/index.php?file=/etc/passwd"
[로그 파일: /var/log/snort/alert_fast.txt]
03/31-08:40:22.776 [**] [1:1000003:1] "Possible LFI - /etc/passwd access" [**] {TCP} 192.168.1.5:54322 -> 192.168.1.10:80
분석: HTTP URI 페이로드(Payload, 데이터 본체) 내에 시스템 핵심 경로가 포함되어 정책 위반으로 탐지됨.
STEP 05

SQL 인젝션 인증 우회 탐지

공격자가 데이터베이스 쿼리를 조작하여 관리자 권한을 탈취하려는 행위를 감시합니다.

Rule Config
alert tcp any any -> $HOME_NET 80 (msg:"SQL Injection Attempt - OR 1=1"; content:"or 1=1"; nocase; http_uri; sid:1000005; rev:1;)
Victim Node
$ sudo snort -c /etc/snort/snort.conf -A fast
Attacker: SQLi
$ curl "http://192.168.1.10/login.php?user=' or 1=1 --"
[로그 파일: /var/log/snort/alert_fast.txt]
03/31-08:45:01.332 [**] [1:1000005:1] "SQL Injection Attempt - OR 1=1" [**] {TCP} 192.168.1.5 -> 192.168.1.10:80
분석: nocase(대소문자 무시) 옵션을 사용하여 'OR 1=1' 구문의 변형된 시도까지 모두 탐지 성공.
STEP 06

Splunk 실시간 대시보드 시각화

수집된 JSON 데이터를 SPL(검색 처리 언어)을 통해 통계 분석하고 시각화합니다.

Splunk SPL Editor
index="snort_logs" sourcetype="_json"
| stats count by msg
| rename msg as "Threat Type", count as "Frequency"
| sort - Frequency
시각화 팁:

우측 'Visualization' 탭에서 'Pie Chart(원형 차트)'를 선택하면 전체 위협 대비 공격 유형별 비중을 직관적으로 파악할 수 있습니다.

4. 아키텍트 인사이트 (Expert Insight)

탐지 성능 및 오탐 관리

대규모 트래픽 환경에서는 무분별한 Rules 적용이 패킷 손실(Packet Drop)을 유발할 수 있습니다. 이를 방지하기 위해 Snort 3의 Inspector(인스펙터, 프로토콜 검사기) 설정을 최적화하고, 오탐(False Positive, 정상 행위를 공격으로 오인) 발생 시 제외 규칙(Exclusion Rule)을 적용하여 SIEM의 분석 품질을 높여야 합니다.

보안 운영 체계의 보안(Security for SecOps)

보안 장비 자체에 대한 공격도 고려해야 합니다. Snort 센서가 설치된 노드는 반드시 최소 권한(Least Privilege) 원칙에 따라 루트 권한 실행을 지양하고, Splunk와 통신 시 전용 암호화 프로토콜(TLS, Transport Layer Security)을 적용하여 로그 무결성(Integrity)을 보호하십시오.

라이선스 및 규정 준수 알림 (Licensing & Compliance)

Splunk Enterprise Free 라이선스는 하루 수집 한도(500MB) 초과 시 색인이 중단됩니다. 대량의 로그 발생이 예상되는 실습 환경에서는 Snort의 로그 수준을 최적화하거나, 한도가 없는 오픈소스 대안(ELK Stack 등)을 검토할 수도 있습니다. 단, 엔터프라이즈 환경에서는 Splunk의 강력한 상관 분석(Correlation Analysis) 기능이 더 선호됩니다.
