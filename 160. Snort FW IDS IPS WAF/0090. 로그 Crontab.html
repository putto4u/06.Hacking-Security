ARCHITECT LAB INFRA-SCHEDULING | CRON & LOGROTATE

시스템 크론탭 분석과 logrotate 실행 메커니즘

MAP: /etc/crontab

Lead Security Engineer

`/etc/crontab` 설정 내용을 바탕으로, Snort 로그가 어떻게 자동으로 관리되는지 그 실행 흐름을 정밀하게 분석해 드리겠습니다. 이 설정 파일은 시스템 전체의 예약 작업을 관리하는 핵심 지도 역할을 하며, 로그 순환의 주기를 결정짓는 결정적인 통제 장치입니다.

1. 로그 순환의 자동 실행 지점 분석 (Automated Execution Point)

보내주신 내용 중 다음 줄이 바로 Snort 로그 순환(logrotate)이 실행되는 통로입니다. 이 명령줄은 시스템 가용성과 데이터 연속성을 보장하는 핵심 구문입니다.

/etc/crontab : Line Analysis
25 6   * * *   root   test -x /usr/sbin/anacron || ( cd / && run-parts --report /etc/cron.daily )
[TIME CONFIG]
- 25 6 * * *: 매일 오전 6시 25분에 실행됩니다.
- root: 시스템 최고 관리자 권한으로 명령을 수행합니다.
[LOGIC GATE]
- test -x: anacron 설치 여부 및 실행 권한을 확인합니다.
- || (OR): anacron이 없을 경우에만 뒤의 명령을 수행합니다.

• 실행 시간 (25 6 * * *): 매일 오전 6시 25분에 실행됩니다.

• 실행 권한 (root): 관리자 권한으로 명령을 수행합니다.

• 실행 조건 (test -x /usr/sbin/anacron || ...): 시스템에 anacron (서버가 꺼져 있을 때 누락된 작업을 보충 실행하는 도구)이 설치되어 있는지 확인합니다. 만약 anacron이 없다면(`||`), 직접 `/etc/cron.daily` 디렉터리에 있는 스크립트들을 실행합니다.

• 핵심 명령 (run-parts /etc/cron.daily): `/etc/cron.daily` 폴더 안에 있는 모든 실행 파일을 순차적으로 실행하라는 의미입니다. 보통 이 폴더 안에 logrotate 스크립트가 들어 있습니다.

2. 전체적인 시스템 연동 흐름 (System Integration Workflow)

`/etc/logrotate.d/snort` 파일이 실제로 동작하기까지의 연쇄 반응은 다음과 같습니다. 각 단계는 이전 단계의 성공을 전제로 실행되며, 최종적으로 보안 로그의 무결성을 유지합니다.

[도식 2-1: Snort 로그 순환 자동화 파이프라인]

STEP 01
매일 오전 6:25
System Crontab 가동
STEP 02
logrotate 구동
/etc/cron.daily
STEP 03
Snort 설정 로드
/etc/logrotate.d/snort
STEP 04
로그 순환 및 점검
Rotate & Compress

1. 매일 오전 6:25: 시스템 크론탭에 의해 `/etc/cron.daily/logrotate` 스크립트가 실행됩니다.

2. logrotate 실행: 해당 스크립트가 메인 설정인 `/etc/logrotate.conf`를 불러옵니다.

3. 개별 설정 로드: `logrotate.conf` 내의 `include /etc/logrotate.d` 지시어에 따라 /etc/logrotate.d/snort 파일을 읽습니다.

4. 로그 점검 및 순환: Snort 설정 파일에 적힌 규칙(예: 파일 크기가 일정 수준 이상이거나 날짜가 경과함)을 검토하여 실제 로그 파일을 압축하거나 교체합니다.

3. 주요 구성 요소 상세 설명 (Key Components Breakdown)

crontab

리눅스에서 주기적인 작업을 예약하기 위한 설정 파일입니다. 분, 시, 일, 월, 요일 순으로 구성되며 시스템 및 사용자 레벨에서 독립적으로 운영 가능합니다.

run-parts

특정 디렉터리에 포함된 모든 실행 가능한 스크립트를 찾아 순서대로 실행해주는 유틸리티입니다. cron.daily, cron.weekly 등의 폴더 관리에 사용됩니다.

anacron

서버가 24시간 켜져 있지 않은 환경에서, 꺼진 시간 동안 수행되지 못한 cron 작업을 시스템 부팅 후에 보충 실행해주는 특수 도구입니다.

exit status

명령어 실행 후 커널에 반환되는 값입니다. `0`은 성공을 의미하며, `||` 연산자는 앞의 명령이 실패(상태 코드 0이 아님)했을 때 실행되도록 제어합니다.

4. 전문 용어 해설 (Technical Glossary)

• system-wide crontab (시스템 전역 크론탭): 특정 사용자가 아닌 시스템 전체에 영향을 미치는 예약 작업 설정 파일입니다. `/etc/crontab` 및 `/etc/cron.d/` 디렉터리에 위치합니다.

• SHELL (쉘 환경 변수): 크론이 명령어를 실행할 때 사용할 쉘 인터프리터를 지정합니다. 예시에서는 `/bin/sh`를 사용하여 최소한의 환경에서 실행합니다.

• PATH (경로 환경 변수): 실행 파일을 찾을 경로들의 목록입니다. 명령어 실행 시 전체 경로를 적지 않아도 이 경로들에서 파일을 자동으로 찾습니다.

• run-parts (런-파츠): 지정한 디렉터리 내의 모든 스크립트를 일괄적으로 실행해주는 리눅스용 보조 프로그램입니다.

실행 시간의 분산 설계

크론탭에서 오전 6시 25분과 같이 '정시'가 아닌 시간을 선택하는 이유는, 정시(00분)에 몰리는 다른 시스템 작업들과의 충돌을 방지하고 I/O 부하를 분산시키기 위한 아키텍처적 고려입니다.

Cloud Infrastructure Warning

AWS CloudWatch Logs Agent나 Kinesis Firehose를 연동하여 로그를 클라우드로 전송할 경우, logrotate가 파일을 이동하거나 압축하는 순간 파일 설명자(FD)가 유실되어 전송이 중단될 수 있습니다. `copytruncate` 옵션 활용을 반드시 검토하십시오.
