특정 두 호스트 사이의 트래픽 필터링 및 분석 실무

System Architect Pro

1. 아키텍처 개요 (Architecture Overview)

네트워크 보안 및 분석 관점에서 특정 두 **Host (호스트, 네트워크 연결 장치)** 간의 모든 통신인 **Inbound (인바운드, 유입)** 및 **Outbound (아웃바운드, 유출)** 트래픽을 모니터링하는 것은 전체적인 데이터 흐름을 파악하고 침해 사고를 분석하는 데 필수적인 작업입니다. 본 실습에서는 **Snort (Open Source Network Intrusion Detection System, 오픈 소스 네트워크 침입 탐지 시스템)**를 활용하여 출발지와 목적지의 방향성에 관계없이 특정 IP 쌍 사이의 트래픽만을 정밀하게 필터링하는 아키텍처를 구축합니다. 이를 위해 **BPF (Berkeley Packet Filter, 버클리 패킷 필터)** 기술을 사용하여 커널 수준에서 불필요한 데이터를 배제하고 분석의 가용성을 높입니다.

[도식: 호스트 간 양방향 모니터링 메커니즘]

Host A (192.168.1.10)
BIDIRECTIONAL
Snort Inspection
Host B (192.168.1.20)
용어 해설 (Glossary)
  • • **BPF (Berkeley Packet Filter, 버클리 패킷 필터)**: 네트워크 인터페이스 카드로 유입되는 전체 패킷 중 응용 프로그램이 정의한 특정 조건에 맞는 데이터만을 커널 수준에서 효율적으로 선별하는 고성능 필터링 기술입니다.
  • • **Payload (페이로드, 순수 데이터 본문)**: 네트워크 패킷의 구성 요소 중 전송 경로 제어를 위한 헤더 정보를 제외하고, 사용자가 실제로 전송하고자 하는 응용 계층의 본문 데이터를 의미합니다.
  • • **Inbound & Outbound (인바운드 & 아웃바운드)**: 인바운드는 외부 네트워크에서 내부 자원으로 유입되는 트래픽을, 아웃바운드는 내부 시스템에서 외부망으로 나가는 트래픽을 지칭합니다.

2. 실습 구성 요소 (Components)

NODE 01

소스 타겟 (Source A)

  • IP 주소: 192.168.1.10
  • 역할: 트래픽 생성 및 요청지
NODE 02

목적지 타겟 (Destination B)

  • IP 주소: 192.168.1.20
  • 역할: 서비스 제공 및 응답지
용어 해설 (Glossary)
  • • **MAC Address (Media Access Control Address, 맥 주소)**: 네트워크 인터페이스 카드(NIC)에 부여된 전 세계적으로 유일한 물리적 하드웨어 식별 주소로, 데이터 링크 계층(Layer 2) 통신의 핵심입니다.
  • • **IDS (Intrusion Detection System, 침입 탐지 시스템)**: 네트워크 트래픽을 실시간으로 감시하여 비정상적인 활동이나 이미 알려진 공격 시그니처와 일치하는 패턴을 탐지하고 관리자에게 경고를 전달하는 보안 솔루션입니다.

3. 실증 및 구현 (Implementation)

STEP 01

스니퍼 모드 기반의 호스트 간 통신 확인

가장 직관적인 방법은 스노트를 실행할 때 BPF 문법을 사용하여 필터를 적용하는 것입니다. `host` 키워드는 해당 IP가 소스이든 데스티네이션이든 관계없이 모두 식별합니다.

Sniffer Console
# 두 호스트(192.168.1.10과 192.168.1.20) 사이의 모든 통신 실시간 확인
# snort -vde host 192.168.1.10 and host 192.168.1.20
STEP 02

특정 인터페이스를 지정한 정밀 모니터링

서버에 다수의 네트워크 인터페이스가 존재하는 경우, 대상 트래픽이 실질적으로 통과하는 인터페이스를 정확히 명시해야 합니다.

Interface Binding
# eth0 인터페이스를 통해 통신하는 두 호스트의 패킷 상세 분석
# snort -vde -i eth0 host 10.0.0.5 and host 10.0.0.10
STEP 03

사용자 정의 룰(Custom Rule)을 통한 영구 탐지

단순한 실시간 출력이 아니라 통신 발생 시 로그를 남기거나 알람을 발생시키려면 `snort.conf`에 사용자 정의 규칙을 추가해야 합니다. 양방향 연산자인 `<>`를 사용하여 소스와 목적지를 통합 지정합니다.

Local Rules Configuration
# /etc/snort/rules/local.rules 파일에 다음 규칙 추가
alert tcp 192.168.1.10 any <> 192.168.1.20 any (msg:"Two Hosts Communication Detected"; sid:1000001;)
STEP 04

실전 분석 팁: PCAP 바이너리 결과 저장

실시간 화면 출력은 트래픽 양이 많을 경우 시각적 한계가 있으므로, 바이너리 형태로 저장하여 추후 정밀 분석을 수행하는 것이 효율적입니다.

Binary Logging
# 두 호스트 간의 통신을 pcap(패킷 캡처 파일) 형식으로 저장
# snort -i eth0 host 1.1.1.1 and host 2.2.2.2 -b -L ./capture.log

4. 아키텍트 인사이트 (Expert Insight)

필터링 논리(and vs or)의 중요성

스노트에서 두 호스트를 지정할 때 `and` 조건이 아닌 `or`를 사용하면 두 호스트 중 하나라도 연관된 모든 통신(외부와의 통신 포함)을 보게 되므로, 반드시 `and`를 사용하여 두 지점 간의 '상호 폐쇄적 통신'만 필터링하시기 바랍니다.

리소스 소모 주의 사항

운영 중인 실제 프로덕션 환경의 트래픽 양에 따라 `-v (Verbose)` 옵션은 CPU 및 I/O 리소스를 급격히 소모하여 패킷 유실을 초래할 수 있으니 대량 트래픽 분석 시에는 가급적 바이너리 로깅 모드를 우선적으로 사용하십시오.
