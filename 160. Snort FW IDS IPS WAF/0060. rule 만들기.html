1. 운영 환경 및 권한 아키텍처 (Environment & Permissions)

Snort 3는 차세대 침입 방지 시스템(IPS: Intrusion Prevention System)으로, 고속 패킷 처리를 위해 멀티스레딩 아키텍처를 채택하고 있습니다. 안정적인 운영을 위해서는 최소 권한 원칙(Principle of Least Privilege)에 따른 디렉토리 및 실행 권한 설정이 필수적입니다.

경로 (Path)

  • /usr/local/etc/snort/ (설정)
  • /usr/local/etc/snort/rules/ (룰)
  • /usr/local/var/log/snort/ (로그)

계정 (User)

  • 파일 소유자: root
  • 서비스 계정: snort (1001)
  • 실행 주체: sudo -u snort

모드 (Mode)

  • 룰 파일: 644 (-rw-r--r--)
  • 설정 파일: 640 (-rw-r-----)
  • 로그 폴더: 755 (drwxr-xr-x)

[도식: Snort 3 탐지 및 로깅 워크플로우]

인터페이스 감시
(eth0)
룰 매칭 엔진
(Detection Engine)
로그 파일 기록
(alert_fast)

2. 룰 실증 및 구현 (Implementation & Step-by-Step)

STEP 01

사용자 정의 룰(Local Rules) 작성

다음은 네트워크 위협 및 활동을 탐지하기 위한 기초 5대 룰입니다. root 권한으로 파일을 편집합니다.

/usr/local/etc/snort/rules/local.rules
$ sudo vi local.rules
# 1. ICMP 탐지 (Ping 체크)
alert icmp any any -> any any ( msg:"ICMP Packet Detected"; sid:1000001; rev:1; )
# 2. HTTP GET 요청 탐지
alert tcp any any -> any 80 ( msg:"HTTP GET Request Detected"; content:"GET"; http_method; sid:1000002; rev:1; )
# 3. SSH 접속 시도 (Syn 패킷)
alert tcp any any -> any 22 ( msg:"SSH Connection Attempt"; flags:S; sid:1000003; rev:1; )
# 4. 악성 문자열 탐지
alert tcp any any -> any any ( msg:"Malicious String Found"; content:"malware_test"; nocase; sid:1000004; rev:1; )
# 5. DNS 질의 탐지
alert udp any any -> any 53 ( msg:"DNS Query Detected"; sid:1000005; rev:1; )
STEP 02

파일 권한 설정 및 엔진 기동

파일 모드를 변경하고 snort 유저 권한으로 엔진을 실행합니다.

# 권한 부여 (Read for Others)
$ sudo chmod 644 /usr/local/etc/snort/rules/local.rules
# 탐지 엔진 기동 (User: snort)
$ sudo -u snort snort -c /usr/local/etc/snort/snort.lua -R /usr/local/etc/snort/rules/local.rules -i eth0 -A alert_fast

3. 로그 데이터 정밀 분석 (Alert Log Deep Dive)

실시간 탐지 로그 출력 화면 (Console Output)

03/30-09:15:22.112 [**] [1:1000001:1] "ICMP Packet Detected" [**] {ICMP} 10.0.2.15 -> 192.168.1.1
03/30-09:16:45.991 [**] [1:1000004:1] "Malicious String Found" [**] {TCP} 10.0.2.5:44321 -> 192.168.1.10:80
Field 01: Timestamp
03/30-09:15:22.112

패킷이 탐지 엔진에 의해 분석된 시각(월/일-시:분:초.밀리초)을 의미합니다. 침해 사고 대응(Incident Response) 시 타임라인을 구축하는 결정적 근거가 됩니다.

Field 02: SID (Signature ID)
[1:1000001:1]

룰의 고유 식별자입니다. [GID:SID:REV] 형식으로 구성되며, 여기서는 1,000,001번 룰의 첫 번째 버전이 매칭되었음을 나타냅니다.

Field 03: Message
"Malicious String Found"

룰 정의 시 msg 옵션에 기록한 내용입니다. 탐지된 트래픽의 성격(공격 유형, 정책 위반 등)을 관리자가 직관적으로 파악하게 합니다.

Field 04: Flow & Protocol
10.0.2.5:44321 -> 192.168.1.10:80

패킷의 출발지 IP:포트에서 목적지 IP:포트로의 흐름을 보여줍니다. 서비스(80 포트 등)에 대한 공격 여부와 공격자의 위치를 파악할 수 있습니다.

4. 아키텍트 인사이트 (Expert Insight)

실시간 로그 모니터링 기법

로그가 파일로 저장될 경우 tail -f /usr/local/var/log/snort/alert_fast 명령어를 활용하십시오. 특정 공격 IP만 실시간으로 모니터링하려면 grep "10.0.2.5"와 파이프를 조합하는 것이 효율적입니다.

권한 에러(Permission Denied) 대응

snort 유저로 기동 시 로그 파일 작성이 실패한다면 디렉토리 소유권을 확인하십시오. sudo chown -R snort:snort /usr/local/var/log/snort 명령을 통해 로그 저장 권한을 명확히 부여해야 합니다.
