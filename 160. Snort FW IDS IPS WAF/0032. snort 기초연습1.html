1. 아키텍처 개요 (Architecture Overview)

Snort(스노트)는 소스파이어(Sourcefire, 현재 Cisco가 인수)에서 개발한 오픈소스 기반의 네트워크 침입 탐지 시스템(NIDS, Network Intrusion Detection System)입니다. Snort는 단순히 패킷을 수집하는 스니퍼(Sniffer) 역할을 넘어, 시그니처(Signature) 기반 탐지 엔진을 통해 실시간으로 네트워크 트래픽을 감시하고 비정상적인 행위나 공격 시도를 식별합니다. 본 실습에서는 Snort의 세 가지 운영 모드와 더불어, 실제 보안 관제 현장에서 즉시 활용 가능한 20가지 실습 시나리오를 통해 시스템의 동작 원리를 완벽히 파악합니다.

[도식: SNORT 3단계 분석 파이프라인]

패킷 디코더
(Packet Decoder)
탐지 엔진
(Detection Engine)

(Alert/Logging)
Key Terminology
NIDS (Network Intrusion Detection System): 네트워크 전체의 트래픽을 감시하여 침입을 탐지하는 시스템.
Signature (시그니처): 이미 알려진 공격 패턴을 정의해 놓은 규칙 데이터베이스.
Promiscuous Mode (무차별 모드): 자신의 MAC 주소가 아닌 패킷도 모두 수신하도록 설정된 네트워크 인터페이스 모드.
IPS (Intrusion Prevention System): 탐지를 넘어 실시간으로 공격을 차단하는 능동형 보안 시스템.

2. 실습 환경 구성 (Lab Components)

SENSOR

Snort 센서 (Linux)

  • OS: Ubuntu 22.04 LTS 또는 Kali Linux
  • 환경: Snort v2.9.x 설치 완료
  • 설정 파일: /etc/snort/snort.conf
NETWORK

트래픽 소스 및 로그

  • 인터페이스: eth0 (또는 실습 환경의 NIC)
  • 로그 경로: /var/log/snort/
  • 도구: nmap, hping3, curl (트래픽 생성용)

3. Snort 기본 실행 및 모드 연습 (01~05)

Part A: Basic Modes
01

스니퍼 모드(Sniffer Mode) 실행

네트워크 인터페이스로 유입되는 패킷의 IP/TCP/UDP/ICMP 헤더 정보를 실시간으로 터미널에 출력합니다.

$ sudo snort -v

Detailed Analysis

  • sudo: 패킷 캡처를 위해 시스템 루트 권한을 획득합니다.
  • snort: Snort 바이너리를 실행합니다.
  • -v: Verbose 모드. 패킷의 헤더 정보를 화면에 상세히 표시합니다.
02

데이터 링크 계층 포함 출력

패킷의 페이로드뿐만 아니라 데이터 링크(Ethernet) 헤더의 MAC 주소 정보까지 출력합니다.

$ sudo snort -vd

Detailed Analysis

  • -d: Application 계층의 데이터를 덤프(Dump)하여 보여줍니다.
  • 활용: 네트워크 상의 MAC 주소 위조(ARP Spoofing) 등을 의심할 때 유용합니다.
03

애플리케이션 계층 포함 상세 출력

$ sudo snort -vde

-e 옵션을 추가하여 데이터 링크 계층의 상세 정보까지 포함하는 가장 상세한 분석 모드입니다.

04

패킷 로그 저장 (Binary Logging)

$ sudo snort -vde -l ./log

-l [directory] 옵션은 캡처한 패킷을 화면에 출력하는 동시에 지정된 디렉터리에 tcpdump 형식의 바이너리 파일(pcap)로 저장합니다.

Note

실행 전 mkdir log 명령어로 디렉터리가 생성되어 있어야 합니다.

05

저장된 로그 읽기 및 사후 분석

$ sudo snort -r ./log/snort.log.123456789

-r 옵션은 오프라인 분석 모드입니다. 과거에 저장된 pcap 파일을 불러와 다시 Snort 규칙을 적용하거나 상세 내용을 검토할 때 사용합니다.

4. Snort 규칙(Rule) 작성 기초 (06~10)

Part B: Rule Implementation

Rule Structure Guide

alert icmp any any -> any any
(msg:"..."; sid:1000001; rev:1;)

Action: alert, log, pass, drop (차단) 등 수행 동작

Protocol: tcp, udp, icmp, ip 등 대상 프로토콜

Src IP/Port: 출발지 정보 (any는 모두 허용)

Dst IP/Port: 목적지 정보

Options: 탐지 로그 메시지(msg), 고유 식별자(sid) 등

Example 06

ICMP(Ping) 탐지

alert icmp any any -> any any (msg:"ICMP Packet Detected"; sid:1000001; rev:1;)

설명: 외부의 모든 ICMP 트래픽을 감시하여 로그에 남깁니다.

Example 07

특정 서버 80포트 탐지

alert tcp any any -> 192.168.1.50 80 (msg:"HTTP Access Attempt"; sid:1000002;)

설명: 내부 웹 서버(1.50)로의 모든 HTTP 연결 시도를 식별합니다.

Example 08

SSH 접속 시도 탐지

alert tcp any any -> any 22 (msg:"SSH Connection Attempt"; sid:1000003;)

설명: 무차별 대입 공격(Brute-force)의 전조 증상인 SSH 접근을 감시합니다.

Example 09

전체 UDP 패킷 탐지

alert udp any any -> any any (msg:"UDP Traffic Detected"; sid:1000004;)

설명: 비연결 지향성 프로토콜인 UDP의 흐름을 전반적으로 분석합니다.

Example 10

특정 IP 대역(C-Class) 경고

alert ip 211.0.0.0/8 any -> any any (msg:"Malicious IP Block Warning"; sid:1000005;)

설명: 211.x.x.x 대역에서 들어오는 모든 종류의 IP 패킷에 대해 경고를 발생시킵니다.

5. 콘텐츠 매칭 및 고급 탐지 (11~15)

Part C: Deep Packet Inspection
11. HTTP GET 메서드 매칭

페이로드 내 문자열을 직접 매칭(Content Matching)하여 애플리케이션 행위를 정확히 식별합니다.

alert tcp any any -> any 80 (msg:"HTTP GET Method Detected"; content:"GET"; sid:1000006;)
12. 민감한 파일(/etc/passwd)

서버의 계정 정보 파일 탈취 시도를 탐지하는 핵심 보안 규칙입니다.

alert tcp any any -> any 80 (msg:"Password File Access Attempt"; content:"/etc/passwd"; sid:1000007;)
13. 포트 스캔(Well-known)

1번부터 1024번까지의 포트 범위를 지정하여 광범위한 스캐닝 행위를 탐지합니다.

alert tcp any any -> any 1:1024 (msg:"Well-known Port Scan"; sid:1000008;)
14. 페이로드 크기(dsize) 기반

정상 범위를 벗어난 거대 패킷을 이용한 DoS(서비스 거부 공격)나 터널링을 차단합니다.

alert icmp any any -> any any (msg:"Large ICMP Packet"; dsize:>1000; sid:1000009;)
15. TCP SYN 플래그 탐지

3-way Handshake의 시작인 SYN 패킷만 필터링하여 연결 시도 횟수를 파악합니다.

alert tcp any any -> any any (msg:"TCP SYN Detected"; flags:S; sid:1000010;)

6. 실전 대응 및 설정 확인 (16~20)

16

설정 파일 문법 검사

$ sudo snort -T -c /etc/snort/snort.conf
# -T: Test mode (설정 파일의 무결성 검사)
17

IDS 모드 콘솔 실행

$ sudo snort -A console -q -c /etc/snort/snort.conf
# -q: Quiet mode (배너 출력 생략)
Case 18

익명 FTP 로그인 시도

alert tcp any any -> any 21 (msg:"FTP Anonymous Login Attempt"; content:"anonymous"; sid:1000011;)
Case 19

SQL Injection 기초 탐지

alert tcp any any -> any 80 (msg:"SQL Injection Attempt"; content:"' or '1'='1"; nocase; sid:1000012;)
# nocase: 대소문자 구분 없이 매칭
Case 20

상태 기반 탐지 (Stateful)

alert tcp any any -> any any (msg:"Established TCP Session"; flow:established,to_server; sid:1000013;)
# flow: 이미 수립된 세션의 서버향 트래픽만 분석

7. 아키텍트 인사이트 (Architect Insight)

효율적인 Rule 설계 전략

규칙을 설계할 때는 공통 헤더 옵션을 먼저 정의하여 탐지 속도를 높여야 합니다. 특히 content 매칭은 시스템 자원을 많이 소모하므로, 가능한 범위가 좁은 포트나 특정 IP 대역을 선결 조건으로 거는 것이 아키텍처 관점에서 유리합니다.

오탐(False Positive)의 최소화

sid가 중복되지 않도록 주의하십시오. 또한, 광범위한 문자열 매칭은 정상적인 웹 요청(예: 검색 쿼리)을 공격으로 오인할 수 있습니다. http_uri나 http_client_body 같은 수정자를 사용하여 매칭 범위를 제한하는 습관을 들여야 합니다.
