CNAPP (Cloud Native Application Protection Platform, 클라우드 네이티브 애플리케이션 보호 플랫폼)

현대적 클라우드 보안의 핵심은 파편화된 도구의 통합입니다. 본 교재는 개발부터 운영까지의 전 과정을 아우르는 통합 보안 가시성 확보를 목표로 합니다.

01 CNAPP의 핵심 정의와 등장 배경

**CNAPP**은 클라우드 환경에서 애플리케이션을 보호하기 위해 기존의 독립적이었던 보안 솔루션들을 하나의 단일 플랫폼으로 통합한 개념입니다. 과거에는 설정 오류를 잡는 도구와 실행 중인 서버를 지키는 도구가 따로 놀았으나, 이제는 컨텍스트(Context, 문맥/상황 정보)를 공유하여 위협의 우선순위를 결정합니다.

과거의 방식 (Siloed)

  • 도구 간 데이터 공유 불가
  • 보안 경고(Alert) 과부하 발생
  • 런타임 위협과 설정 오류의 연관성 파악 불가

현대의 CNAPP 방식

  • 통합된 데이터 레이크 활용
  • 실제 공격 경로(Attack Path) 가시화
  • 개발 단계(Shift-Left)부터 보안 적용

03 CNAPP의 4대 핵심 구성 요소

1. CSPM (Cloud Security Posture Management, 클라우드 보안 태세 관리)

클라우드 인프라의 설정 오류를 지속적으로 탐지하고 규정 준수 여부를 확인합니다.

  • S3 버킷의 공용 읽기 권한 노출 탐지
  • 비활성화된 로깅 및 모니터링 기능 확인
  • ISMS-P, NIST, CIS 벤치마크 준수성 자동 점검

2. CWPP (Cloud Workload Protection Platform, 클라우드 워크로드 보호 플랫폼)

가상 머신(VM), 컨테이너, 서버리스 등 실행 중인 서비스의 내부 보안을 담당합니다.

  • 애플리케이션 취약점 스캐닝 및 EDR 기능 수행
  • 컨테이너 이미지의 악성코드 및 하드코딩된 비밀번호 탐지
  • 이상 징후(Anomaly Detection) 및 파일 무결성 모니터링

3. CIEM (Cloud Infrastructure Entitlement Management, 클라우드 인프라 권한 관리)

과도하게 부여된 권한을 식별하고 '최소 권한 원칙'을 적용합니다.

  • 사용하지 않는 IAM 계정 및 과다 권한 역할(Role) 탐지
  • 사람뿐만 아니라 서비스/머신 계정의 권한 남용 분석
  • 권한의 유효 범위를 시각화하여 잠재적 위협 제거

4. IaC Security (Infrastructure as Code Security, 코드형 인프라 보안)

Terraform, CloudFormation 등 코드로 인프라를 구축할 때 미리 취약점을 잡습니다.

  • 보안 사고 발생 전(Pre-deployment) 설정 오류 차단
  • 데브섹옵스(DevSecOps) 파이프라인의 자동화된 관문 역할

04 전문가 인사이트: 위협 경로(Attack Path) 분석

단순히 "S3가 열려 있다"는 경고는 무의미합니다. CNAPP 전문가라면 다음과 같은 연쇄 관계를 파악해야 합니다.

인터넷 노출 (CSPM)
→
중요 취약점 존재 (CWPP)
→
과도한 관리자 권한 (CIEM)

* 위 세 가지가 동시에 일어나는 지점이 바로 '가장 먼저 해결해야 할 치명적 위험(Toxic Combination)'입니다.

05 클라우드 보안 과금 체계 가이드

💡 비용 관리 실무 팁 (Billing Management)

대부분의 CNAPP 솔루션은 보호 대상 자산의 수(Asset Count) 또는 워크로드의 규모(Instance/Node count)에 따라 비용이 발생합니다.

  • AWS GuardDuty/Inspector: 스캔한 데이터 양과 인스턴스 수에 따라 종량제(Pay-as-you-go)로 과금되므로, 대규모 환경에서는 자동 스캔 주기 설정이 중요합니다.
  • Third-party CNAPP (SaaS): 일반적으로 연간 구독 모델이며, 에이전트 설치 여부에 따라 성능 오버헤드와 비용이 비례할 수 있습니다.
  • 주의사항: 개발 환경(Dev)과 테스트 환경(Staging)에 무분별하게 유료 스캐닝을 활성화할 경우 예산을 초과할 수 있으니 태깅(Tagging)을 통해 관리 범위를 지정하십시오.
