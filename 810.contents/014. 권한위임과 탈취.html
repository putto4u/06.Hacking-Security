1. OAuth 2.0 및 OIDC 아키텍처 (Architectural Overview)

OAuth 2.0(Open Authorization 2.0, 개방형 권한 위임 프로토콜)는 애플리케이션이 사용자의 비밀번호를 직접 공유하지 않고도 타 서비스의 데이터에 접근할 수 있도록 권한을 위임받는 표준입니다. 여기에 인증 계층을 추가한 OIDC(OpenID Connect, 오픈아이디 커넥트)는 신원 확인 기능을 제공하여 현대적인 SSO(Single Sign-On, 단일 로그인) 환경의 근간을 이룹니다.

[도식: 인가 코드 승인 방식 (Authorization Code Grant Flow)]

자원 소유자
(사용자)
클라이언트
(애플리케이션)
인가 서버
(ID Provider)
1. 서비스 로그인 요청
2. 인가 코드 발급 및 리다이렉트
3. 코드를 토큰으로 교환 (Back-channel)

※ 설명: 클라이언트는 사용자의 브라우저를 통해 인가 코드(Authorization Code)를 전달받고, 이를 서버 간 통신(Back-channel)을 통해 액세스 토큰(Access Token)으로 교환합니다. 이 과정의 허점을 찌르는 것이 주요 공격 포인트입니다.

2. 주요 보안 취약점 분석 (Security Vulnerabilities)

Critical Path

리다이렉트 URI 검증 미흡

인가 서버가 Redirect URI(리다이렉트 유알아이, 인증 후 코드를 전달받을 주소)를 엄격하게 검증하지 않을 때 발생합니다. 부분 일치나 정규 표현식 오설정(Misconfiguration)을 이용해 공격자의 도메인으로 인가 코드를 가로챌 수 있습니다.

# 공격자가 삽입한 파라미터 예시
redirect_uri=https://victim.com.attacker.com/callback

※ 설명: 서버가 서브도메인이나 하위 경로를 허용할 경우, 공격자는 오픈 리다이렉터(Open Redirector) 취약점이 있는 페이지를 활용해 코드를 탈취합니다.

Account Takeover

OAuth CSRF (State 누락)

인증 요청 시 생성되는 State(스테이트, 클라이언트 상태를 확인하기 위한 임의의 값) 파라미터를 사용하지 않거나 검증하지 않을 때 발생합니다. 공격자는 자신의 인가 코드를 피해자의 브라우저에 강제로 주입하여 피해자의 계정을 공격자의 신원과 연결(Account Linking)할 수 있습니다.

# State 값이 없는 취약한 요청
/authorize?client_id=123&response_type=code... (state missing)

※ 설명: 이 공격에 성공하면 피해자가 결제하거나 정보를 입력할 때 공격자의 계정으로 데이터가 쌓이게 되는 기묘한 상황이 연출됩니다.

3. PKCE(Proof Key for Code Exchange)와 현대적 보안

모바일 앱이나 SPA(Single Page Application, 단일 페이지 애플리케이션)는 Client Secret(클라이언트 시크릿, 애플리케이션 비밀번호)을 안전하게 보관할 수 없습니다. 이를 보완하기 위해 도입된 기술이 PKCE(Proof Key for Code Exchange, 코드 교환을 위한 증명 키)입니다.

[도식: PKCE 보안 메커니즘 (PKCE Proof Mechanism)]

1. 검증자 생성
Code Verifier

랜덤 문자열 생성 후 SHA-256 해싱하여 챌린지 생성

2. 검증 수행
Code Challenge

인가 서버는 챌린지 값을 저장하고 나중에 검증자와 비교

시큐어 구현: PKCE 적용 가이드

"PKCE는 이제 모바일뿐만 아니라 모든 웹 클라이언트에서도 표준 권장 사항(BCP, Best Current Practice)입니다."

# OAuth 2.0 인가 요청 시 PKCE 파라미터 포함

  response_type=code&

  code_challenge_method=S256

4. 아키텍트 인사이트 (Architectural Insight)

OIDC 신뢰 체인과 SSRF 위협

OIDC는 request_uri라는 파라미터를 통해 외부에서 인증 설정을 가져올 수 있습니다. 이를 악용하여 인가 서버가 내부 인프라에 접근하도록 유도하는 SSRF(Server-Side Request Forgery, 서버 측 요청 위조) 공격이 가능합니다. 인가 서버는 신뢰할 수 있는 도메인의 요청만 수락하도록 화이트리스트(Whitelist)를 운영해야 합니다.

Referer 헤더를 통한 정보 유출

인가 코드가 포함된 페이지에서 외부 리소스를 불러올 때, Referer(리퍼러, 이전 페이지 주소 정보) 헤더를 통해 인가 코드가 외부 서버로 전송될 수 있습니다. Referrer-Policy를 no-referrer로 설정하거나, 코드를 한 번만 사용할 수 있도록 보장해야 합니다.

클라우드 인증 비용 분석 (AWS Cognito Pricing)

AWS Cognito(코그니토, 클라우드 사용자 신원 관리 서비스)는 월간 활성 사용자(MAU) 50,000명까지는 무료 등급을 제공합니다. 하지만 SAML(Security Assertion Markup Language, 보안 인증 마크업 언어)이나 OIDC를 통한 외부 연동 시 활성 사용자당 약 $0.015의 추가 비용이 발생합니다. 또한, 고급 보안 기능(MFA, 이상 징후 탐지 등) 활성화 시 추가 과금이 발생하므로 설계 시 주의가 필요합니다.
