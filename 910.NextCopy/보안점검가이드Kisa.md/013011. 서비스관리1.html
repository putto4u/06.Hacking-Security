1. Finger 서비스 비활성화

🚨 보안 위협: Finger 서비스가 활성화되어 있을 경우, 비인가자가 Finger 서비스를 사용하여 사용자 정보를 조회한 후 비밀번호 공격을 통해 계정을 탈취할 위험이 존재합니다. ⚙️ 조치 시 영향: 일반적인 경우 영향 없음.

점검 및 조치 사례

[ SOLARIS(5.9 이하 버전) ]

Step 1) /etc/inetd.conf 파일 내 Finger 서비스 활성화 여부 확인 및 비활성화
Finger 서비스 항목 주석 처리
예시) #finger stream tcp nowait bin /usr/lbin/fingered fingerd

Step 2) inetd 서비스 재시작

[ SOLARIS (5.10 이상 버전) ]

Step 1) Finger 서비스 활성화 여부 확인
# inetadm | grep finger

Step 2) Finger 서비스 데몬 중지
# inetadm -d svc:/network/finger:default

[ LINUX ]

[inetd]
Step 1) /etc/inetd.conf 파일 내 Finger 서비스 활성화 여부 확인 및 비활성화
Finger 서비스 항목 주석 처리
예시) #finger stream tcp nowait bin /usr/lbin/fingered fingerd

Step 2) inetd 서비스 재시작

[xinetd]
Step 1) /etc/xinetd.d/finger 파일 내 Finger 서비스 활성화 여부 확인 및 비활성화
finger의 disable 옵션을 yes로 수정

Step 2) 설정 적용 및 xinetd 서비스 재시작
# systemctl restart xinetd

[ AIX, HP-UX ]

Step 1) /etc/inetd.conf 파일 내 Finger 서비스 활성화 여부 확인 및 비활성화
Finger 서비스 항목 주석 처리
예시) #finger stream tcp nowait bin /usr/lbin/fingered fingerd

Step 2) inetd 서비스 재시작

📖 전문 용어 해설 * Finger(사용자 정보 확인 서비스): who 명령어가 현재 사용 중인 사용자들에 대한 간단한 정보만을 보여주는 데 반해, Finger 명령은 옵션에 따라 시스템에 등록된 사용자뿐만 아니라 네트워크를 통하여 연결되어 있는 다른 시스템에 등록된 사용자들에 대한 자세한 정보를 보여주는 프로토콜입니다.




2. 공유 서비스에 대한 익명 접근 제한 설정

🚨 보안 위협: 공유 서비스의 익명 접근을 허용할 경우, 비인가자의 무단 접근으로 인한 중요 정보 탈취 또는 변조, 악성 코드 유포 등의 위험이 존재합니다. ⚙️ 조치 시 영향: 응용프로그램에서 익명 연결을 사용할 경우를 제외하고, 일반적인 경우 영향 없음.

# cat /etc/passwd | grep anonymous

Step 2) FTP 계정 제거

# userdel anonymous

# vi /etc/vsftpd/vsftpd.conf

3. r계열 서비스 비활성화

🚨 보안 위협: rlogin, rsh, rexec 등의 r-command를 이용하여 원격에서 인증 절차 없이 터미널 접속, 쉘 명령어를 실행할 수 있는 위험이 존재합니다. ⚙️ 조치 시 영향: 일반적인 경우 영향 없음.

점검 및 조치 사례

[ SOLARIS(5.9 이하 버전) ]

Step 1) /etc/inetd.conf 파일 내 불필요한 계열 서비스 활성화 여부 확인
# vi /etc/inetd.conf

Step 2) 불필요한 계열 서비스 관련 필드 주석처리
예시) #shell stream tcp nowait root /usr/sbin/in.rshd in.rshd

Step 3) 서비스 재시작
# kill -HUP [inetd PID]

[ SOLARIS (5.10 이상 버전) ]

Step 1) 불필요한 계열 서비스 활성화 여부 확인
# inetadm | egrep "shell|rlogin|rexec"

Step 2) 불필요한 계열 서비스 데몬 중지
# inetadm -d <rcommand 데몬>

[ LINUX ]

[inetd]
Step 1) /etc/inetd.conf 파일 내 불필요한 계열 서비스 활성화 여부 확인

Step 2) 불필요한 계열 서비스 관련 필드 주석 처리
# vi /etc/inetd.conf
예시) #rlogin stream tcp nowait root /usr/sbin/in.rlogind in.rlogind

Step 3) inetd 서비스 재시작

[xinetd]
Step 1) /etc/xinetd.d/<파일 이름> 파일 내 불필요한 계열 서비스 활성화 여부 확인

Step 2) 불필요한 계열 서비스 비활성화
disable 값을 yes로 수정

Step 3) 설정 적용 및 서비스 재시작
# systemctl restart xinetd

[systemd]
Step 1) 불필요한 계열 서비스 활성화 여부 확인
# systemctl list-units --type service | grep -E "rlogin|rsh|rexec"

Step 2) 현재 가동되고 있는 계열 서비스 중지
# systemctl stop <서비스 이름>

Step 3) 불필요한 계열 서비스 비활성화
# systemctl disable <서비스 이름>

[ AIX, HP-UX ]

[inetd]
Step 1) /etc/inetd.conf 파일 내 불필요한 계열 서비스 활성화 여부 확인

Step 2) 불필요한 계열 서비스 관련 필드 주석 처리
# vi /etc/inetd.conf
예시) #login stream tcp6 nowait root /usr/sbin/rlogind rlogind

Step 3) 변경된 설정 적용
- AIX: # refresh -s inetd
- HP-UX: # inetd -c

[r-command]
Step 1) /etc/inetd.conf 파일 내 불필요한 계열 서비스 활성화 여부 확인

Step 2) /etc/hosts.equiv, $HOME/.rhosts 파일에 접근을 허용할 사용자 이름 또는 IP주소 설정

Step 3) /etc/hosts.equiv, $HOME/.rhosts 파일 권한을 600 이하로 설정

📖 전문 용어 해설 * r-command(알 커맨드): 원격 인증 없이 시스템 관리자의 원격 접속을 가능하게 하는 명령어 모음으로 rsh(remsh), rlogin, rexec, rsync 등이 있습니다. 클러스터링 구성이나 주기적인 백업 용도로 종종 사용됩니다.




4. crontab 설정파일 권한 설정 미흡

🚨 보안 위협: 일반 사용자가 crontab 및 at 서비스를 사용할 수 있을 경우, 고의 또는 실수로 불법적인 예약 파일 실행으로 시스템 피해를 일으킬 수 있는 위험이 존재합니다. ⚙️ 조치 시 영향: 일반적인 경우 영향 없음.

점검 및 조치 사례

[ SOLARIS ]

Step 1) crontab, cron 작업 목록 파일, cron 관련 파일 소유자 및 권한 확인
# ls -l /usr/bin/crontab
# ls -l /var/spool/cron/crontabs/<cron 작업 목록 파일>
# ls -l /etc/cron.d/<cron 관련 파일>

Step 2) at, at 작업 목록 파일 소유자 및 권한 확인
# ls -l /usr/bin/at
# ls -l /var/spool/cron/atjobs/<at 작업 목록 파일>

Step 3) crontab 파일 및 at 파일 소유자를 root로, 파일 권한을 750으로 변경

Step 4) cron 작업 목록 파일, cron 관련 파일 및 at 작업 목록 파일 소유자를 root로, 파일 권한을 640으로 변경

5. DoS 공격에 취약한 서비스 비활성화

🚨 보안 위협: 해당 서비스가 활성화된 경우, 시스템 정보 유출 및 시스템의 자원을 부족하게 만드는 DoS 공격의 직접적인 타겟이 될 위험이 존재합니다. ⚙️ 조치 시 영향: 일반적인 경우 영향 없음.

Step 2) /etc/inetd.conf 파일 수정(주석 처리)

SMTP (25) 인터넷에서 메일을 보내기 위해 사용되는 서비스

14. 스팸 메일 릴레이 제한

🚨 보안 위협: SMTP 서버에 타인의 메일을 대신 전달해주는 릴레이(Relay) 기능을 제한 없이 방치할 경우, 전 세계의 악의적인 해커들이 대량의 광고성 스팸 메일 발송용 중계 기지(서버)로 몰래 악용하거나, 수많은 트래픽을 집중시켜 서버 자원을 고갈시키는 서비스 거부(DoS) 공격으로 서버가 마비될 극심한 위험이 존재합니다. ⚙️ 조치 시 영향: 일반적인 경우 영향 없음.

점검 및 조치 사례

[ SOLARIS, LINUX, AIX, HP-UX ]

[Sendmail 8.9 이상 버전]
Step 1) /etc/mail/sendmail.cf 파일 내 릴레이 허용 설정 여부 확인

Step 2) /etc/mail/sendmail.mc 파일 수정
FEATURE(`promiscuous_relay')dnl <해당 설정 제거>

Step 3) sendmail.cf 설정 파일 재생성
# m4 /etc/mail/sendmail.mc > /etc/mail/sendmail.cf

Step 4) /etc/mail/access 파일에 특정 IP, Domain, Email 주소, 네트워크에 대한 접근제한 설정

localhost RELAY
127.0.0.1 RELAY
spam.com REJECT

15. expn, vrfy 명령어 제한

🚨 보안 위협: 정보 조회를 목적으로 하는 expn, vrfy 명령어를 차단하지 않으면 외부 공격자가 이 명령어를 무차별로 던져보며 특정 시스템 내부에 해당 사용자 계정이 실제로 존재하는지 손쉽게 알아낼 수 있고, 이를 통해 획득한 아이디 리스트를 바탕으로 해킹의 초기 발판(무차별 비밀번호 대입 공격 등)으로 악용할 심각한 위험이 존재합니다. ⚙️ 조치 시 영향: 일반적인 경우 영향 없음.

점검 및 조치 사례

[ SOLARIS, LINUX, AIX, HP-UX ]

[Sendmail]
Step 1) /etc/mail/sendmail.cf 파일 내 PrivacyOptions 설정 확인

Step 2) PrivacyOptions 옵션 수정
PrivacyOptions = authwarnings, novrfy, noexpn, restrictqrun
또는
PrivacyOptions = restrictqrun, goaway

Step 3) Sendmail 서비스 재시작
※ goaway : authwarnings, noexpn, novrfy, noveb, needmailhelo, needexpnhelo, needvrfyhelo, nobodyreturn 기능이 통합된 단축 옵션

[Postfix]
Step 1) /etc/postfix/main.cf 파일 내 vrfy 설정 확인

Step 2) disable_vrfy_command 옵션을 yes로 수정
disable_vrfy_command = yes

Step 3) postfix 설정 적용 및 재시작
# postfix reload
※ Postfix는 기본적으로 expn 기능 및 설정을 허용하지 않음

[Exim]
Step 1) /etc/exim/exim.conf 또는 /etc/exim4/exim4.conf 파일 내 expn, vrfy 설정 확인

Step 2) 해당 옵션이 허용된 경우 설정 제거
acl_smtp_vrfy = accept
acl_smtp_expn = accept
주석 처리 혹은 명령어 줄 삭제

Step 3) Exim 서비스 재시작

📖 전문 용어 해설 * expn(메일링 리스트 확장 명령): 메일 전송 시 해당 메일 주소가 속한 포워딩 경로나 메일링 리스트에 포함된 실제 개별 사용자의 상세 목록을 확인하기 위해 사용하는 명령어입니다. * vrfy(사용자 검증 명령): SMTP 클라이언트가 SMTP 서버에 접속해 특정 아이디를 전송하며 해당 아이디에 대한 올바른 수신 메일함(계정)이 실제로 서버에 존재하는지 검증받기 위해 보내는 명령어입니다.
