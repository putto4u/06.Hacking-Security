아파치 톰캣(Apache Tomcat) 설치 및 웹 서버 연동 실무


아파치 톰캣은 자바(Java) 기반의 웹 애플리케이션을 구동하기 위한 핵심 인프라스트럭처입니다. 앞서 아파치(Apache) 웹 서버를 성공적으로 구축하셨다면, 이제 정적 콘텐츠 처리의 한계를 넘어 동적 비즈니스 로직을 처리할 수 있는 WAS (Web Application Server, 웹 애플리케이션 서버) 환경을 완성할 차례입니다. 본 장에서는 기초적인 설치부터 전문가 수준의 시스템 서비스 등록, 그리고 기존 아파치 웹 서버와의 연동 아키텍처까지 깊이 있게 다룹니다.



전문 용어 해설 * WAS (Web Application Server, 웹 애플리케이션 서버): 데이터베이스 조회나 다양한 로직 처리를 요구하는 동적인 콘텐츠를 제공하기 위해 만들어진 애플리케이션 미들웨어입니다. * 인프라스트럭처 (Infrastructure, 기반 시설): IT 환경을 구성하는 하드웨어, 소프트웨어, 네트워크 리소스 등의 뼈대를 의미합니다.





사전 준비: JDK (Java Development Kit, 자바 개발 도구) 설치


톰캣은 자바 환경에서 동작하므로, 시스템에 반드시 JDK (Java Development Kit, 자바 개발 도구)가 설치되어 있어야 합니다. JRE (Java Runtime Environment, 자바 실행 환경)만으로도 실행은 가능하지만, JSP (JavaServer Pages, 자바 서버 페이지) 소스 코드를 컴파일하고 오류를 분석하는 등의 서버 운영 및 개발을 위해서는 JDK를 설치하는 것이 엔터프라이즈 환경의 표준입니다.


sudo apt update # 우분투 패키지 관리자의 원격 저장소 목록을 최신 상태로 갱신합니다.
sudo apt install openjdk-17-jdk -y # 오픈 소스 기반의 자바 17 개발 도구를 사용자 프롬프트 없이 즉시 설치합니다.
java -version # 자바 실행 파일의 버전을 출력하여 정상적으로 설치되었는지 시스템에 질의합니다.
echo $JAVA_HOME # 자바 환경 변수가 설정되어 있는지 콘솔에 출력하여 확인합니다.


💡 실전 팁 (Practical Tip)

실무에서 자바 버전을 선택할 때는 반드시 개발팀의 애플리케이션 호환성을 먼저 확인해야 합니다. 현재는 LTS (Long Term Support, 장기 지원) 버전인 Java 11, Java 17, Java 21이 주로 사용되며, 본 과정에서는 클라우드 및 컨테이너 환경에서 널리 쓰이는 Java 17을 기준으로 합니다.


전문 용어 해설 * JDK (Java Development Kit, 자바 개발 도구): 자바 애플리케이션을 개발하고 실행하기 위한 필수 도구 모음입니다. * JRE (Java Runtime Environment, 자바 실행 환경): 컴파일된 자바 프로그램을 실행시키기 위한 핵심 환경을 제공합니다. * LTS (Long Term Support, 장기 지원): 일반적인 소프트웨어 릴리스보다 훨씬 긴 기간 동안 보안 패치와 업데이트를 지원하는 안정적인 버전입니다.





아파치 톰캣 코어 설치 및 보안 권한 설정


공식 문서에 따르면 보안 및 시스템 격리를 위해 톰캣은 root (루트, 최고 관리자) 계정이 아닌 전용 사용자 계정으로 실행하는 것을 강력히 권고합니다. 아키텍트 관점에서 보안 취약점을 최소화하기 위해 전용 그룹과 사용자를 생성하고, 바이너리 파일을 안전한 시스템 경로에 배치해야 합니다.


sudo groupadd tomcat # 톰캣 전용 시스템 그룹을 생성하여 접근 권한 관리를 분리합니다.
sudo useradd -s /bin/false -g tomcat -d /opt/tomcat tomcat # 쉘 로그인이 불가능하고(/bin/false) 홈 디렉토리를 /opt/tomcat으로 지정한 톰캣 전용 사용자를 생성합니다.
cd /tmp # 인터넷에서 임시 파일을 다운로드하기 위해 시스템의 임시 디렉토리로 이동합니다.
wget https://dlcdn.apache.org/tomcat/tomcat-10/v10.1.19/bin/apache-tomcat-10.1.19.tar.gz # 아파치 공식 미러 사이트에서 톰캣 10.1.x 버전의 압축 파일을 다운로드합니다.
sudo mkdir /opt/tomcat # 톰캣 서버 프로그램이 설치될 대상 디렉토리를 시스템의 추가 패키지 경로(/opt)에 생성합니다.
sudo tar xzvf apache-tomcat-10*tar.gz -C /opt/tomcat --strip-components=1 # 다운로드한 파일의 압축을 풀되, 최상위 폴더 1단계를 제거하여 /opt/tomcat 아래에 깔끔하게 배치합니다.
sudo chgrp -R tomcat /opt/tomcat # /opt/tomcat 경로 내의 모든 파일과 하위 디렉토리의 소유 그룹을 tomcat 그룹으로 재귀적으로(-R) 변경합니다.
sudo chmod -R g+r /opt/tomcat/conf # 환경설정(conf) 디렉토리의 그룹(g) 구성원에게 읽기(r) 권한을 재귀적으로 부여합니다.
sudo chmod g+x /opt/tomcat/conf # 환경설정(conf) 디렉토리 자체에 그룹 실행(x, 디렉토리 진입 권한) 권한을 부여합니다.
sudo chown -R tomcat /opt/tomcat/webapps/ /opt/tomcat/work/ /opt/tomcat/temp/ /opt/tomcat/logs/ # 톰캣 실행 중 실제로 쓰기 작업이 발생하는 주요 디렉토리들의 소유자를 tomcat 사용자로 확정합니다.


⚠️ 자주 오해하거나 실수하는 부분 (Common Pitfalls)

초기 인프라 구축 시 설정이 번거롭다는 이유로 톰캣을 root 권한으로 구동하는 경우가 종종 있습니다. 이는 웹 애플리케이션에 RCE (Remote Code Execution, 원격 코드 실행) 취약점이 발생했을 때, 해커가 서버 전체의 제어권을 단숨에 탈취할 수 있게 만드는 치명적인 설계 결함입니다. 반드시 제한된 권한의 전용 계정으로 구동하십시오.


전문 용어 해설 * RCE (Remote Code Execution, 원격 코드 실행): 공격자가 원격지에서 대상 시스템의 시스템 명령이나 악성 코드를 무단으로 실행할 수 있는 치명적인 보안 취약점입니다. * Binary (바이너리, 이진 파일): 컴퓨터가 직접 이해하고 메모리에 올려 실행할 수 있는 기계어 코드로 이루어진 컴파일된 실행 파일입니다.





Systemd (시스템 데몬) 서비스 등록 및 최적화


상용 엔터프라이즈 환경에서는 서버가 재부팅되거나 예기치 않은 오류로 프로세스가 종료되었을 때, 관리자의 개입 없이 자동으로 서비스를 복구할 수 있어야 합니다. 리눅스 표준 시스템 관리자인 Systemd를 사용하여 톰캣을 안정적인 백그라운드 서비스로 등록합니다.


sudo nano /etc/systemd/system/tomcat.service # systemd 데몬이 인식할 수 있는 시스템 경로에 톰캣 서비스 설정 파일을 생성하고 편집기를 엽니다.


열린 편집기 화면에 아래의 구성 값들을 세밀하게 작성합니다.


[Unit] # 해당 시스템 서비스의 메타데이터와 부팅 의존성을 정의하는 섹션입니다.
Description=Apache Tomcat Web Application Container # 시스템 로그 및 상태 확인 시 표시될 해당 서비스의 명확한 설명을 작성합니다.
After=network.target # 네트워크 서브시스템이 완전히 초기화되고 IP가 할당된 이후에 이 서비스를 실행하도록 순서를 보장합니다.

[Service] # 서비스의 실제 실행 방법, 메모리, 환경 변수를 정의하는 핵심 제어 섹션입니다.
Type=forking # 프로세스가 자체적으로 백그라운드로 분기(fork)되어 실행되는 데몬 형태임을 systemd에 알려줍니다.

Environment=JAVA_HOME=/usr/lib/jvm/java-17-openjdk-amd64 # 톰캣이 자바 컴파일러와 런타임을 찾을 수 있도록 정확한 자바 설치 경로를 시스템 환경 변수로 명시합니다.
Environment=CATALINA_PID=/opt/tomcat/temp/tomcat.pid # 톰캣 메인 프로세스의 고유 ID 번호가 저장될 파일 경로를 지정하여 systemd의 프로세스 추적을 돕습니다.
Environment=CATALINA_HOME=/opt/tomcat # 톰캣 실행 바이너리와 라이브러리가 위치한 기본 홈 디렉토리를 지정합니다.
Environment=CATALINA_BASE=/opt/tomcat # 톰캣의 설정 파일 및 실제 배포된 애플리케이션 데이터가 위치한 디렉토리를 지정합니다.
Environment='CATALINA_OPTS=-Xms512M -Xmx1024M -server -XX:+UseParallelGC' # 자바 가상 머신의 초기/최대 힙 메모리 크기를 할당하고 성능 중심의 서버 모드 가비지 컬렉터를 지정하여 튜닝합니다.
Environment='JAVA_OPTS=-Djava.awt.headless=true -Djava.security.egd=file:/dev/./urandom' # 모니터(GUI)가 없는 서버 환경임을 명시하고, 암호화 작업 시 필요한 난수 생성 병목을 해결하기 위해 urandom을 사용합니다.

ExecStart=/opt/tomcat/bin/startup.sh # systemctl start 명령을 내릴 때 실행할 톰캣의 공식 시작 쉘 스크립트의 절대 경로입니다.
ExecStop=/opt/tomcat/bin/shutdown.sh # systemctl stop 명령을 내릴 때 캐시를 지우고 안전하게 톰캣을 종료하는 스크립트 경로입니다.

User=tomcat # 앞서 생성한 제한된 권한의 tomcat 시스템 사용자로 프로세스를 실행하도록 강제합니다.
Group=tomcat # 프로세스의 소유 그룹 역시 tomcat 그룹으로 한정합니다.
UMask=0007 # 톰캣이 실행 중에 생성하는 임시 파일이나 로그에 대해 그룹 외 타인(Others)의 접근을 원천 차단하는 보안 마스크입니다.
RestartSec=10 # 프로세스가 비정상적인 이유로 종료되었을 때, 충돌 방지를 위해 10초 대기 후 재시작합니다.
Restart=always # 프로세스 종료 코드를 불문하고 무조건적으로 서비스를 자동 재시작하여 24시간 가용성을 확보합니다.

[Install] # 서비스 자동 실행 활성화(enable) 시 운영체제의 런레벨과 어떻게 결합할지 정의합니다.
WantedBy=multi-user.target # 그래픽 인터페이스가 없는 일반적인 다중 사용자 서버 모드로 부팅될 때 이 서비스가 자동으로 시작되도록 연결 고리를 만듭니다.


설정 파일을 저장한 후 시스템에 즉시 반영하고 기동합니다.


sudo systemctl daemon-reload # 새로 생성하거나 수정된 systemd 서비스 파일들을 커널이 다시 읽어들여 캐시를 갱신하도록 지시합니다.
sudo systemctl start tomcat # 설정된 정보를 바탕으로 톰캣 서비스를 즉시 시작하여 자바 프로세스를 메모리에 올립니다.
sudo systemctl enable tomcat # 서버 장비가 재부팅될 때마다 톰캣 서비스가 자동으로 시작되도록 부팅 스크립트에 정식 등록합니다.
sudo systemctl status tomcat # 톰캣 서비스의 현재 실행 상태(Active)와 최근 발생한 로그 라인들을 확인하여 정상 작동 여부를 검증합니다.


전문 용어 해설 * Systemd (System Daemon, 시스템 데몬): 최신 리눅스 배포판(Ubuntu, CentOS 등)에서 시스템 부팅 시 가장 먼저 실행되어 다른 프로세스들을 초기화하고 수명주기를 관리하는 핵심 백그라운드 프로세스입니다. * GC (Garbage Collection, 가비지 컬렉션): 자바 메모리 관리 기법의 하나로, 애플리케이션에서 더 이상 사용되지 않는 객체를 식별하고 자동으로 메모리를 회수하여 메모리 누수를 원천 방지하는 기능입니다. * Heap Memory (힙 메모리): 프로그램이 실행되는 동안 동적으로 할당되는 데이터(사용자 세션, 객체 변수 등)가 보관되는 자바 가상 머신 내부의 주요 메모리 영역입니다.





기존 아파치 웹 서버와의 연동 (Integration)


아키텍처 관점에서 아파치(정적 이미지, HTML 콘텐츠 처리 탁월)와 톰캣(DB 연동 등 동적 콘텐츠 처리 탁월)을 연동하는 것은 매우 표준적인 실무 패턴입니다. 이를 통해 사용자 부하 분산(Load Balancing)과 아키텍처 다중화를 통한 보안 강화를 달성할 수 있습니다.

연동 방식에는 전통적인 mod_jk (AJP 프로토콜 사용) 모듈과 현대적인 mod_proxy (HTTP 또는 AJP 프로토콜 사용) 모듈을 사용하는 방법이 있습니다. 최근 클라우드 네이티브 환경에서는 별도의 컴파일 과정 없이 아파치에 기본 내장되어 있으며, 직관적인 라우팅 설정이 가능한 mod_proxy를 기반으로 리버스 프록시를 구성하는 것이 글로벌 트렌드입니다.



1. 아파치 프록시 모듈 활성화


sudo a2enmod proxy # 아파치 코어 프록시 모듈을 시스템에 활성화하여 중계 서버 역할을 준비합니다.
sudo a2enmod proxy_http # HTTP 프로토콜 규약을 이용해 백엔드 서버(톰캣의 8080 포트)로 클라이언트의 요청을 포워딩하는 확장 모듈을 활성화합니다.
sudo systemctl restart apache2 # 추가된 모듈 사항을 메모리에 반영하기 위해 아파치 웹 서버 데몬을 재시작합니다.


2. 아파치 가상 호스트 (Virtual Host) 설정 적용


기존에 구동 중인 아파치의 웹 서비스 설정 파일(예: /etc/apache2/sites-available/000-default.conf)을 열어 리버스 프록시 라우팅 규칙을 추가합니다.


sudo nano /etc/apache2/sites-available/000-default.conf # 아파치의 기본 가상 호스트 설정 파일을 최고 관리자 권한으로 텍스트 편집기에서 엽니다.



<VirtualHost *:80> # 80번 포트(표준 웹 HTTP)로 들어오는 모든 IP 주소의 요청을 처리하기 위한 가상 호스트 블록의 시작점입니다.
    ServerName www.example.com # 웹 서버가 응답할 공식 도메인 이름을 지정합니다. (실제 환경에서는 구매한 도메인이나 공인 IP로 변경해야 합니다.)

    # Proxy 통합 라우팅 설정 시작
    ProxyRequests Off # 서버가 외부 사용자의 접근을 우회시켜주는 오픈 프록시(해커의 IP 세탁 중계지)로 악용되는 것을 막기 위해 Forward Proxy 기능을 명시적으로 차단합니다.
    ProxyPreserveHost On # 클라이언트 웹 브라우저가 최초 요청 시 보낸 원본 Host 헤더 정보를 변조하지 않고 백엔드인 톰캣 컨테이너까지 그대로 유지하여 전달합니다.

    ProxyPass / http://localhost:8080/ # 사용자 접속 경로인 루트 경로(/) 하위로 들어오는 모든 트래픽 요청을 서버 내부망(localhost)의 8080 포트에서 대기 중인 톰캣으로 포워딩합니다.
    ProxyPassReverse / http://localhost:8080/ # 톰캣 내부에서 생성되어 응답되는 HTTP 헤더의 URL(예: 리다이렉트 주소)을 클라이언트가 인식할 수 있는 원래 아파치 서버의 공인 주소로 자동 치환하여 보안을 유지합니다.
    # Proxy 통합 라우팅 설정 종료

    ErrorLog ${APACHE_LOG_DIR}/error.log # 서버 운영 중 발생하는 다양한 오류 및 에러 메시지를 기록할 로그 파일의 절대 경로를 환경 변수를 사용하여 지정합니다.
    CustomLog ${APACHE_LOG_DIR}/access.log combined # 클라이언트의 IP, 접속 시간, 브라우저 정보 등 상세 접속 기록을 산업 표준인 combined 포맷으로 파일에 남깁니다.
</VirtualHost> # 선언된 가상 호스트 설정 블록의 끝을 시스템에 알립니다.


설정을 저장하고 빠져나온 후, 적용 전 아파치 문법을 안전하게 검사합니다.


sudo apache2ctl configtest # 작성한 아파치 설정 파일에 오타, 누락, 또는 심각한 문법적 오류(Syntax Error)가 없는지 데몬 실행 전에 미리 안전하게 검증합니다.
sudo systemctl restart apache2 # Syntax OK(문법 이상 없음) 판정을 확인한 후, 검증이 완료된 프록시 설정을 라이브 서비스에 적용하기 위해 아파치를 재시작합니다.


💡 실전 팁 (Practical Tip)

아키텍트가 아파치 설정 시 ProxyPreserveHost On 구문을 누락하는 실수가 현업에서 빈번하게 발생합니다. 이 설정이 누락되면 웹 애플리케이션(톰캣 내 자바 소스)에서 로그인 성공 후 메인 페이지로 리다이렉트(Redirect)를 수행할 때, 사용자 브라우저의 주소창이 공인 도메인에서 localhost:8080 이라는 사설망 주소로 갑자기 변경되어 사이트 접속이 끊어지는 치명적인 서비스 장애가 발생합니다. 필수적으로 적용해야 하는 옵션입니다.


전문 용어 해설 * Proxy (프록시, 대리 서버): 사용자인 클라이언트와 원격의 서버 사이에서 데이터를 안전하게 중계하는 서버입니다. 캐싱을 통한 성능 향상이나 네트워크 로드 밸런싱 등의 목적으로 활용됩니다. * Reverse Proxy (리버스 프록시, 역방향 프록시): 외부 인터넷의 사용자가 내부망에 숨겨진 실제 애플리케이션 서버(톰캣)에 직접 접근하지 못하도록, 앞단(DMZ 구간)에서 요청을 대신 받아 백엔드로 라우팅해 주는 강력한 보안 아키텍처 구조입니다. 본 실습에서 아파치가 바로 이 역할을 수행합니다. * Virtual Host (버추얼 호스트, 가상 호스트): 물리적으로는 단 한 대의 웹 서버 장비이지만, 소프트웨어 설정을 통해 마치 여러 대의 서버가 존재하는 것처럼 다수의 도메인(예: a.com, b.com)을 동시에 서비스할 수 있게 해주는 핵심 웹 호스팅 기술입니다.
