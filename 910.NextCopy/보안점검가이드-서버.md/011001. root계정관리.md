## 1. root 계정 원격 접속 제한

**🚨 보안 위협:** root 계정 탈취 시 외부에서 원격을 이용해 시스템을 장악하거나 무차별/사전 대입 공격 등에 악용되어 루트 계정을 사용할 수 없게 될 위험이 존재합니다.
**⚙️ 조치 시 영향:** 일반적인 경우 영향 없음.

### 점검 및 조치 사례

**[ SOLARIS ]**
```bash
[Telnet]
Step 1) /etc/default/login 파일 내에 CONSOLE 설정값 수정
CONSOLE=/dev/console

[SSH]
Step 1) /etc/ssh/sshd_config 파일 내의 PermitRootLogin 설정값 수정
PermitRootLogin No
```

**[ LINUX ]**
```bash
[Telnet]
Step 1) /etc/pam.d/login 파일 내에 auth required /lib/security/pam_securetty.so 입력
Step 2) /etc/securetty 파일 내에 pts/설정값 주석 처리 및 제거
#pts/0
#pts/1
#pts/2
Step 3) /etc/pam.d/login 파일 내에 모듈 추가
auth required /lib/security/pam_securetty.so
# ※ /etc/securetty 파일 내 pts/x 관련 설정이 존재하는 경우 PAM 모듈 설정과 관계없이 root 계정 접속을 허용하므로 반드시 제거 필요
# ※ CentOS 8, Ubuntu 20.04 이상부터 /etc/securetty 파일이 존재하지 않으며 기본적으로 Telnet 서비스가 비활성화됨

[SSH]
Step 1) /etc/ssh/sshd_config 파일에 PermitRootLogin 값 수정
PermitRootLogin No
```

**[ AIX ]**
```bash
[Telnet]
Step 1) /etc/security/user 파일에 rlogin 설정값 수정
rlogin = false

[SSH]
Step 1) /etc/ssh/sshd_config 파일에 PermitRootLogin 값 수정
PermitRootLogin No
```

**[ HP-UX ]**
```bash
[Telnet]
Step 1) /etc/securetty 파일 내에 console 값 수정
console
# ☑ /etc/securetty 파일은 기본적으로 존재하지 않으므로 해당 파일이 존재하지 않는 경우 생성 후 설정할 것

[SSH]
Step 1) /opt/ssh/etc/sshd_config 파일 내에 PermitRootLogin 값 수정
PermitRootLogin No
```

> **💡 실전 팁 & 자주 하는 실수**
> 현장에서 `sshd_config` 수정 후 `systemctl restart sshd` 등 서비스 데몬을 재시작하지 않아 정책이 즉시 반영되지 않는 실수를 가장 많이 합니다. 설정 변경 후에는 반드시 관련된 서비스를 재시작하여 적용 여부를 테스트해야 합니다.

> **📖 전문 용어 해설**
> * **root 계정:** 운영체제의 모든 기능을 설정 및 변경할 수 있는 최고 관리자 계정입니다.
> * **Telnet(Teletype Network, 텔넷):** 인터넷이나 로컬 네트워크를 통해 원격으로 컴퓨터에 접속할 수 있게 해주는 평문 기반의 네트워크 프로토콜입니다.
> * **SSH(Secure Shell, 보안 셸):** 트래픽이 강력하게 암호화되어 원격 시스템에서 안전하게 명령을 실행할 수 있도록 해주는 네트워크 프로토콜입니다.
> * **PAM(Pluggable Authentication Modules, 장착형 인증 모듈):** 다양한 인증 메커니즘을 응용 프로그램의 소스코드 수정 없이 플러그인 방식으로 유연하게 통합 제공하는 라이브러리입니다.

## 2. 비밀번호 관리정책 설정

**🚨 보안 위협:** 비밀번호 관련 정책이 없을 경우 비인가자의 다양한 암호 해독 공격(무차별 대입 공격 등)에 의해 비밀번호가 노출될 위험이 큽니다.
**⚙️ 조치 시 영향:** 비밀번호 변경 시 연동된 다른 구간(Web, WAS, DB 등)에서 인증 문제가 발생할 수 있으므로, 연동 시스템에 미칠 영향을 사전 고려해야 합니다.

### 점검 및 조치 사례

**[ SOLARIS ]**
```bash
Step 1) /etc/default/passwd 파일 내에 비밀번호 설정값 수정
HISTORY=4
PASSLENGTH=8
MINDIGIT=1
MINUPPER=1
MINLOWER=1
MINSPECIAL=1
WHITESPACE=NO
```

| 권고값 | 기능 | 설명 |
| --- | --- | --- |
| HISTORY= 10 | 이전 비밀번호 기억 개수 | 이전 10개의 암호를 기억함 |
| MINDIFF= 4 | 이전 암호와 차이 | 이전 암호와 4자 이상 차이 요구 |
| MINALPHA= 1 | 최소 문자 요구 | 최소 1자 이상 문자 요구 |
| MINNONALPHA= 1 | 최소 숫자 또는 특수문자 요구 | 숫자 또는 특수문자 1자 이상 요구 |
| MINUPPER=1 | 최소 대문자 요구 | 최소 1자 이상 대문자 요구 |
| MINLOWER= 1 | 최소 소문자 요구 | 최소 1자 이상 소문자 요구 |
| MAXREPEATS= 0 | 연속 문자 사용 허용 | 0일 경우 문자 연속 사용이 불가 |
| MINSPECIAL= 1 | 최소 특수문자 요구 | 최소 1자 이상 특수문자 요구 |
| MINDIGIT= 1 | 최소 숫자 요구 | 최소 1자 이상 숫자 요구 |
| NAMECHECK= YES | 아이디와 비밀번호 동일 검증 | 아이디와 동일한 비밀번호 사용 불가 |
| MAXDAYS= 90 | 비밀번호 최대 유효일 수 | 최대 90일 비밀번호가 유효 |
| MINDAYS= 1 | 비밀번호 변경 최소일 수 | 비밀번호 최소 1일 후 변경 가능 |
| MAXWEEKS= 12 | 비밀번호 최대 유효 주 수 | 최대 12주 비밀번호가 유효 |
| MINWEEKS=1 | 비밀번호 변경 최소일 수 | 비밀번호 최소 일주일 후 변경 가능 |
| WARNWEEKS= 1 | 비밀번호 만료 전 알림 주 수 | 비밀번호 만료 일주일 전 알림 |
| PASSLENGTH= 8 | 비밀번호 최소 길이 | 비밀번호 최소 길이 8 |
| WHITESPACE= NO | 비밀번호 공백문자 사용 여부 | 비밀번호에 공백문자 사용 금지 설정 |

> ※ MINDIGIT, MINSPECIAL 설정이 적용되어 있는 경우 MINNONALPHA 설정은 적용되지 않음
> ※ MINDAYS 설정이 적용되어야 MAXDAYS 설정이 적용됨
> ※ 비밀번호 유효일 설정과 비밀번호 유효 주 설정은 중복 설정 불가함
> ※ WHITESPACE 기본값: YES

**[ LINUX - Redhat ]**
```bash
Step 1) /etc/login.defs 파일에 PASS_MAX_DAYS/PASS_MIN_DAYS 값 수정
PASS_MAX_DAYS 90
PASS_MIN_DAYS 0

Step 2) /etc/security/pwquality.conf 파일에 정책 값 수정
minlen = 8
dcredit = -1
ucredit = -1
lcredit = -1
ocredit = -1
enforce_for_root

Step 3) /etc/security/pwhistory.conf 파일에 값 추가 및 수정
enforce_for_root
remember=4
file=/etc/security/opasswd

Step 4) /etc/pam.d/system-auth 파일에 값 수정
Step 5) /etc/login.defs 파일에 PASS_MAX_DAYS/PASS_MIN_DAYS 값 수정
PASS_MAX_DAYS 90
PASS_MIN_DAYS 1
```

**[ LINUX - Debian ]**
```bash
Step 1) /etc/security/pwquality.conf 파일에 정책 값 수정
minlen = 8
dcredit = -1
ucredit = -1
lcredit = -1
ocredit = -1
enforce_for_root

Step 2) /etc/pam.d/common-password 파일에 정책 값 수정
# pam_pwquality.so, pam_pwhistory.so 모듈은 pam_unix.so 모듈 위에 위치해야 적용됨

Step 3) /etc/login.defs 파일에 값 수정
PASS_MAX_DAYS 90
PASS_MIN_DAYS 1
```

| 권고값 | 기능 | 설명 |
| --- | --- | --- |
| difok=N | 기존 비밀번호와 비교 | 기존 비밀번호에 포함되지 않는 문자를 최소 N개 이상 포함하도록 설정 |
| minlen=8 | 최소 비밀번호 길이 설정 | 최소 8자리 이상 설정 |
| dcredit=-1 | 최소 숫자 요구 | 최소 숫자 1자 이상 요구 |
| ucredit=-1 | 최소 대문자 요구 | 최소 대문자 1자 이상 요구 |
| lcredit=-1 | 최소 소문자 요구 | 최소 소문자 1자 이상 요구 |
| ocredit=-1 | 최소 특수문자 요구 | 최소 특수문자 1자 이상 요구 |
| remember=N | 최근 비밀번호 기억 | 최근 변경한 비밀번호를 N개 이상 기억하여 동일한 비밀번호로 변경하지 못하도록 설정 |
| PASS_MIN_DAYS=1 | 비밀번호 최소 사용 기간 설정 | 비밀번호 최소 사용 기간 설정 (단위 : 일) |
| PASS_MAX_DAYS=90 | 비밀번호 최대 사용 기간 설정 | 비밀번호 최대 사용 기간 설정 (단위: 일) |

> ※ /etc/security/pwquality.conf 파일과 /etc/pam.d/common-password(/etc/pam.d/system-auth) 파일 중 어느 하나라도 비밀번호 관리 정책이 설정되어 있으면 양호
> ※ 비밀번호 복잡성 설정에서 최소 요구 항목의 값은 반드시 -1로 설정되어야 함
> ※ /etc/pam.d/system-auth 파일에 enforce_for_root 추가

**[ AIX ]**
```bash
Step 1) /etc/security/user 파일에 정책 값 수정
default:
minage = 1
maxage = 12
minalpha = 2
minother = 2
minspecialchar = 1

```

| 권고값 | 기능 | 설명 |

| minalpha=2 | 최소 알파벳 문자 포함 | 비밀번호에 필요한 최소 영문자 수 |
| minother=2 | 최소 알파벳 문자 이외의 문자 수 | 비밀번호에 필요한 최소 알파벳을 제외한 문자 수 |

**[부적절한 비밀번호 유형]**
* 사전에 나오는 단어나 이들의 조합
* 길이가 너무 짧거나 NULL(공백)인 비밀번호
* 키보드 자판의 일련의 나열(예시: abcd, qwert 등)
* 사용자 계정 정보에서 유추 가능한 단어들(예시: 지역명, 부서명, 계정명, 사용자 이름 이니셜, root, admin 등)

**[비밀번호 관리 방법]**
영문, 숫자, 특수문자를 조합하여 계정명과 다른 8자 이상의 비밀번호 설정
1. 다음 각 항목의 문자 종류 중 2종류 이상을 조합하여 최소 10자리 이상 또는 3종류 이상을 조합하여 최소 8자리 이상의 길이로 구성
   가. 영문 대문자(26개)
   나. 영문 소문자(26개)
   다. 숫자(10개)
   라. 특수문자(32개)
2. 시스템마다 다른 비밀번호 사용
3. 비밀번호를 기록해 놓는 경우 변형하여 기록

> **💡 실전 팁 & 자주 하는 실수**
> `dcredit`이나 `ucredit` 같은 복잡성 설정에서 요구 항목 값을 `-1`로 설정해야 함을 자주 잊습니다. 양수를 쓰면 해당 종류의 문자로 주어지는 '크레딧(가점)'이 되어 다른 문자를 줄일 수 있게 허용하는 의미가 되므로, 최소 1자 이상의 '강제 요구'를 하려면 반드시 음수(`-1`)를 설정해야 합니다. (Red Hat 계열: 이 설정은 유료 구독(서브스크립션)을 통해 운영되는 Enterprise Linux 환경에서도 매우 중요하게 다뤄지는 정책이므로 정확히 숙지해야 합니다.)

> **📖 전문 용어 해설**
> * **WAS(Web Application Server, 웹 애플리케이션 서버):** 동적인 웹 사이트나 웹 애플리케이션의 비즈니스 로직을 처리하기 위해 구동되는 미들웨어입니다.
> * **DB(Database, 데이터베이스):** 체계적으로 구성되어 관리되는 데이터의 집합체를 의미합니다.

## 3. 계정 잠금 임계값 설정

**🚨 보안 위협:** 계정 잠금 임계값이 미설정될 경우, 공격자의 인증 요청에 시스템이 무한히 응답하게 되어 무차별 대입이나 사전 대입 공격을 통해 비밀번호가 노출될 위험이 존재합니다.
**⚙️ 조치 시 영향:** PAM 모듈 설정 시 라이브러리 경로가 틀리면 로그인 장애가 유발되며, HP-UX는 Trusted Mode로 전환될 때 파일 시스템 구조가 변경되어 운영 서비스 문제를 일으킬 수 있으므로 충분한 사전 테스트가 요구됩니다.

### 점검 및 조치 사례

**[ SOLARIS ]**
```bash
[5.9 미만 버전]
Step 1) /etc/default/login 파일에 RETRIES 값 수정
RETRIES=10

[5.9 이상 버전]
Step 1) /etc/security/policy.conf 파일에 LOCK_AFTER_RETRIES 값 수정
LOCK_AFTER_RETRIES=YES
UNLOCK_AFTER=2m
```

| 옵션 | 설명 |
| --- | --- |
| RETRIES | 로그인 시도 횟수 |
| LOCK_AFTER_RETRIES | 로그인 시도 횟수와 같거나 초과 되면 잠금 여부 |
| UNLOCK_AFTER | 잠금시간 분(m), 시(h), 일(d), w(주) 단위로 설정 가능 |

**[ LINUX ]**
```bash
[Redhat 계열의 pam_tally.so 또는 pam_tally2.so]
Step 1) /etc/pam.d/system-auth 파일에 deny 값 수정
auth required /lib/security/pam_tally.so 또는 /lib/security/pam_tally2.so deny=10 unlock_time=120 no_magic_root
account required /lib/security/pam_tally.so 또는 /lib/security/pam_tally2.so no_magic_root reset
# ※ /etc/pam.d/system-auth 파일 수정 시 모듈이 해당 경로에 존재하지 않을 경우, 모든 계정의 로그인이 되지 않는 등 예기치 못한 상황이 발생할 수 있으므로 반드시 올바른 경로를 작성해야 함

[Redhat 계열의 authselect]
Step 1) # authselect enable-feature with-faillock 입력하여 faillock 적용
# authselect current
Profile ID: sssd
Enabled features:
- with-fingerprint
- with-silent-lastlog
- with-faillock

Step 2) /etc/security/faillock.conf 파일에 정책 값 수정
silent
deny = 10
unlock_time = 120

[Redhat 계열의 pam_faillock.so]
Step 1) /etc/pam.d/system-auth 파일에 모듈 값 수정
auth required pam_faillock.so preauth silent audit deny=10 unlock_time=120

Step 2) /etc/pam.d/password-auth 파일에 모듈 값 수정
auth required pam_faillock.so preauth silent audit deny=10 unlock_time=120
# ※ RHEL 8 이상부터 authselect 명령어를 이용하여 설정하는 것을 권장함

[Debian 계열의 pam_tally.so 또는 pam_tally2.so]
Step 1) /etc/pam.d/common-auth 파일에 모듈 값 수정
auth required /lib/security/pam_tally.so 또는 /lib/security/pam_tally2.so deny=10 unlock_time=120 no_magic_root
account required /lib/security/pam_tally.so 또는 /lib/security/pam_tally2.so no_magic_root reset

[Debian 계열의 pam_faillock.so]
Step 1) /etc/pam.d/common-auth 파일에 pam_faillock.so 모듈 값 수정
# here are the per-package modules (the "Primary" block)
auth required pam_faillock.so preauth audit deny=10 unlock_time=120
auth [success=1 default=ignore] pam_unix.so nullok
auth [success=1 default=ignore] pam_sss.so use_first_pass
# here's the fallback if no module succeeds
auth [default=die] pam_faillock.so authfail audit deny=10 unlock_time=120
auth sufficient pam_faillock.so authsucc audit deny=10 unlock_time=120
auth requisite pam_deny.so
# prime the stack with a positive return value if there isn't one already;
# this avoids us returning an error just because nothing sets a success code
# since the modules above will each just jump around
auth required pam_permit.so
# [/etc/pam.d/common-auth]

Step 2) /etc/pam.d/common-account 파일에 pam_faillock.so 모듈 값 수정
account required pam_permit.so
# and here are more per-package modules (the "Additional" block)
account sufficient pam_localuser.so
account sufficient pam_sss.so
account [default=bad success=ok user_unknown=ignore] pam_faillock.so
# end of pam-auth-update config
account required pam_faillock.so
# [/etc/pam.d/common-account]
```

| 옵션 | 설명 |
| --- | --- |
| no_magic_root | root 계정은 비밀번호 잠금 설정을 적용하지 않음 |
| deny=N | N회 입력 실패 시 계정 잠금 |
| unlock_time | 계정이 잠긴 경우, 마지막 계정 실패 시간부터 설정된 시간이 지나면 자동으로 계정 잠금 해제 (단위: 초) |
| reset | 접속 시도 성공 시 실패한 횟수 초기화 |

**[ AIX ]**
```bash
Step 1) /etc/security/user 파일에 loginretries 값 수정
loginretries = 3
```

**[ HP-UX ]**
```bash
[11.v2 이하 버전]
Step 1) /tcb/files/auth/system/default 파일에 u_maxtries 값 수정
u_maxtries=3
# ※ HP-UX 서버에 계정 잠금 정책 설정을 위해서는 HP-UX 서버가 Trusted Mode로 동작하고 있어야 하므로 Trusted Mode로 전환 후 잠금 정책 적용

[11.v3 이상 버전]
Step 1) /etc/default/security 파일에 AUTH_MAXTRIES 값 수정
AUTH_MAXTRIES=3
# ※ Standard 모드와 Shadow 모드만 적용 가능
```

> **💡 실전 팁 & 자주 하는 실수**
> `pam_tally2`나 `pam_faillock` 모듈을 설정할 때 설정 파일의 맨 위나 중간 등 아무 곳에나 라인을 넣는 실수를 자주 합니다. PAM 모듈은 위에서 아래로 순차적으로 파싱되므로, 순서가 잘못되면 모든 로그인이 차단되어버릴 수 있습니다. 수정 전 반드시 백업을 해두세요.

> **📖 전문 용어 해설**
> * **임계값(Threshold):** 시스템이 특정한 반응이나 조치를 취하기 시작하는 경계가 되는 수치를 뜻합니다.
> * **PAM(Pluggable Authentication Modules, 장착형 인증 모듈):** 모듈화된 설계를 통해 인증 요구사항을 시스템 전반에 일관되게 적용할 수 있는 라이브러리입니다.

## 4. 비밀번호 파일 보호

**🚨 보안 위협:** 사용자 비밀번호가 담긴 파일이 평문으로 관리되어 유출되거나 탈취될 시, 공격자에게 모든 비밀번호 정보가 그대로 노출되는 위험이 있습니다.
**⚙️ 조치 시 영향:** HP-UX의 경우 Trusted Mode로 전환 시 기존 파일 시스템 구조가 변하여 운영 중인 서비스에 오류를 유발할 수 있으므로 치밀한 사전 테스트가 요구됩니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX ]**
```bash
Step 1) /etc/passwd 입력 후 파일 내 두 번째 필드가 x 표시되는지 확인
root:x:0:0:root:/root:/bin/bash

Step 2) # pwconv 명령으로 쉐도우 비밀번호 적용
# ※ SOLARIS 11은 pwunconv 명령어가 존재하지 않음
```

**[ AIX ]**
```bash
Step 1) /etc/security/passwd 파일에 암호화 여부 확인
# ※ AIX는 기본적으로 /etc/security/passwd 파일에 비밀번호를 암호화하여 저장·관리함
```

**[ HP-UX ]**
```bash
Step 1) /etc/passwd 파일에 암호화 확인
Step 2) # pwconv 명령으로 쉐도우 비밀번호 적용
# ※ HP-UX 서버는 Trusted Mode로 전환할 경우 비밀번호를 암호화하여 /tcb/files/auth 디렉터리에 계정 이니셜과 계정 이름에 따라 파일로 저장·관리할 수 있으므로 Trusted Mode인지 확인 후 UnTrusted Mode인 경우 모드를 전환함
# ※ Trusted mode 전환 방법: root계정으로 아래 명령어 실행
# /etc/tsconvert
# ※ UnTrusted mode 전환 방법: root계정으로 아래 명령어 실행
# /etc/tsconvert -r
# ※ HP-UX 11.11의 경우 Shadow Password Bundle을 설치하여야 /etc/shadow 파일 생성됨
```

> **💡 실전 팁 & 자주 하는 실수**
> `pwconv`를 실행한 후 제대로 `/etc/shadow`에 암호가 옮겨졌는지 직접 열어서 육안 확인을 안 하는 경우가 많습니다. 반드시 설정 후 `cat /etc/shadow`를 통해 암호화된 해시 스트링이 제대로 기록되었는지 검증해야 합니다.

> **📖 전문 용어 해설**
> * **쉐도우 비밀번호(Shadow Password):** 누구나 읽을 수 있는 `/etc/passwd`에 암호를 두는 대신, 관리자(root) 권한으로만 접근 가능한 별도의 공간(예: `/etc/shadow`)에 암호를 저장해 보안을 높인 메커니즘입니다.

## 5. root 이외의 UID가 '0' 금지

**🚨 보안 위협:** 일반 사용자 계정에 root와 동일한 UID가 부여될 경우, 사실상 해당 계정이 최고 관리자 권한을 갖게 되어 모든 시스템 통제권을 남용할 수 있으며 시스템 감사 추적 시 사용자를 명확히 식별할 수 없게 됩니다.
**⚙️ 조치 시 영향:** 해당 계정에 관리자 권한이 원래 불필요했던 것이라면 권한 회수 후에도 일반적인 시스템 운영에 영향을 미치지 않습니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX, AIX, HP-UX ]**
```bash
Step 1) usermod 명령어를 이용하여 # usermod -u <변경할 UID> <사용자 이름> 명령으로 0이외의 중복되지 않는 UID로 변경
```

| 계정명 | 패스워드 | UID | GID | 설명(comment) | 홈디렉터리 위치 | 지정된 쉘 |
| --- | --- | --- | --- | --- | --- | --- |
| Test | x | 500 | 500 | Gen-User | /home/test | /usr/bin/bash |

> ※ `[/etc/passwd 파일 구조]` ":"(콜론)을 사용하여 필드를 구분함
> ※ 세 번째 필드(UID)가 0인 경우 슈퍼 유저 권한을 가지며, 0 이외의 계정은 일반, 시스템 계정으로 볼 수 있음

> **💡 실전 팁 & 자주 하는 실수**
> 계정의 UID를 변경(`usermod -u`)하더라도, 이전에 해당 계정이 생성했던 파일이나 디렉터리의 소유자 정보가 자동으로 변경되지 않는 경우가 생깁니다. UID 변경 후에는 반드시 `find / -user [이전UID]` 명령을 통해 기존에 소유하고 있던 파일들을 찾아 소유권(chown)을 다시 매핑해 주어야 합니다.

> **📖 전문 용어 해설**
> * **UID(User Identification, 사용자 식별자):** UNIX 시스템 내부에서 각 사용자 계정을 고유하게 구분하기 위해 부여되는 고유 식별 번호입니다.
> * **GID(Group Identification, 그룹 식별자):** 사용자들이 속한 그룹을 고유하게 식별하기 위한 번호입니다.

<br><br><br>

## 6. 사용자 계정 su 기능 제한

**🚨 보안 위협:** 모든 사용자에게 `su` 명령어를 허용할 경우, 타인의 계정이나 최고 관리자(root) 계정으로 권한 상승을 시도하여 시스템을 임의로 제어할 수 있는 통로가 활짝 열리게 됩니다.
**⚙️ 조치 시 영향:** 특권 그룹(예: wheel)에 추가된 계정은 바로 적용되지 않을 수 있으며, 반드시 모든 세션 종료 후 재로그인해야 수정한 권한으로 `su` 명령어를 사용할 수 있습니다.

### 점검 및 조치 사례

**[ SOLARIS, AIX, HP-UX ]**
```bash
Step 1) /etc/group 파일 내 wheel 그룹 (su 명령어 사용 그룹) 및 그룹 내 구성원 존재 여부 확인
Step 2) ls -l /usr/bin/su 입력 후 wheel 그룹이 su 명령어를 사용할 수 있는지 설정 여부 확인
Step 3) wheel group 생성 (wheel 그룹이 존재하지 않는 경우)
# groupadd wheel

su 명령 그룹 변경
# chgrp wheel /usr/bin/su

su 명령어 권한 변경
# chmod 4750 /usr/bin/su

wheel 그룹에 su 명령 허용 계정 등록
# usermod -G wheel <username>
또는 직접 /etc/group 파일을 수정하여 필요한 계정 등록
wheel:x:10:root,admin
```

**[ LINUX ]**
```bash
[PAM 모듈 이용 중이지 않을 경우]
Step 1) /etc/group 파일 내 wheel 그룹(su 명령어 사용 그룹) 확인
Step 2) ls 명령어를 이용하여 # ls -l /usr/bin/su 입력 후 su 명령어 그룹과 권한 확인
Step 3) wheel group 생성 (wheel 그룹이 존재하지 않는 경우)
# groupadd wheel

su 명령 그룹 변경
# chgrp wheel /usr/bin/su

su 명령어 권한 변경
# chmod 4750 /usr/bin/su

wheel 그룹에 su 명령 허용 계정 등록
# usermod -G wheel <username>
# ※ /etc/group 파일에서 기본 그룹의 경우 사용자 이름은 생략되며 자동으로 포함됨

[PAM 모듈 이용 중인 경우]
Step 1) /etc/group 입력 후 wheel 그룹(su 명령어 사용 그룹) 확인
예시) wheel:x:1002:

Step 2) /etc/pam.d/su 파일 내 su 명령어 허용 그룹 확인
Step 3) /usr/bin/su 파일 내 su 명령어 그룹과 권한 확인
Step 4) /etc/pam.d/su 파일에 모듈 값 수정
auth required pam_wheel.so use_uid
또는
auth required pam_wheel.so group=wheel
```

> **💡 실전 팁 & 자주 하는 실수**
> 리눅스에서 PAM 설정 시, `pam_wheel.so` 라인의 주석을 해제할 때 단순히 파일만 수정하고 실제 `wheel` 그룹에 사용자를 넣지 않아 관리자 본인조차 `su -` 접근이 원천 차단되는 사고가 매우 빈번하게 일어납니다. 

> **📖 전문 용어 해설**
> * **su(Substitute User, 대체 사용자):** 현재 로그아웃하지 않은 상태에서 잠시 다른 사용자의 권한을 얻기 위해 사용하는 터미널 명령어입니다.

## 7. 불필요한 계정 제거

**🚨 보안 위협:** 퇴직, 휴직, 프로젝트 종료 후에도 삭제되지 않은 계정은 관리가 소홀해지며 공격자에게 손쉬운 패스워드 타겟이 되어 결국 시스템 침입 경로로 악용될 위험이 존재합니다.
**⚙️ 조치 시 영향:** 불필요한 계정을 삭제하는 것이므로 일반적인 시스템 운영에 미치는 영향은 없습니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX, AIX, HP-UX ]**
```bash
[/etc/passwd 파일을 이용하여 점검]
Step 1) /etc/passwd 파일 내 계정을 확인 후 "# userdel <사용자 이름>" 명령으로 불필요한 사용자 계정 제거
# ※ AIX 경우 rmuser 명령어 사용
# ※ /etc/passwd 파일에서 계정 앞에 #을 삽입하여도 주석으로 처리되지 않으므로 조치 시에는 반드시 계정을 제거하도록 권고함

[log를 이용하여 점검]
Step 1) last 명령어로 불필요한 계정 확인 후 제거
```

> **💡 실전 팁 & 자주 하는 실수**
> 많은 엔지니어가 설정 파일(`/etc/passwd`)에서 계정 라인 앞에 `#`을 붙이면 주석 처리가 되어 계정이 비활성화된다고 오해합니다. 패스워드 파일은 주석 기호를 정상적으로 인식하지 않으므로 보안을 위해서는 무조건 `userdel` 명령으로 완전히 지우거나 쉘을 `/bin/false`로 막아야 합니다.

> **📖 전문 용어 해설**
> * **log(로그):** 운영체제나 각종 애플리케이션 소프트웨어가 동작하며 생성하는 주요 이벤트 및 이력 메시지가 기록된 파일입니다.

<br><br><br>

## 8. 관리자 그룹에 최소한의 계정 포함

**🚨 보안 위협:** 관리자 그룹 내에 꼭 필요하지 않은 잉여 계정들이 포함되면, 해당 계정 정보가 유출될 시 곧바로 관리자 권한을 통한 시스템 장악 및 환경설정 파일 변조 등의 심각한 위험을 초래합니다.
**⚙️ 조치 시 영향:** 과도한 권한을 제거하는 것이기 때문에, 실제 권한이 불필요했던 사용자의 작업에 별다른 영향을 미치지 않습니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX, AIX, HP-UX ]**
```bash
Step 1) /etc/group 파일에 root 그룹에 포함된 계정 확인
Step 2) 불필요한 계정을 그룹원에서 제거
# gpasswd -d <사용자 이름> root
# ※ AIX의 경우 chgrpmem -m -<사용자 이름> root 명령어 사용
```

> **💡 실전 팁 & 자주 하는 실수**
> `usermod -G` 옵션 사용 시 소문자 `-g`와 대문자 `-G`를 혼동하거나 부가적인 그룹을 추가하는 옵션(`-aG`)을 쓰지 않아 해당 사용자가 속해 있던 다른 권한 그룹에서 몽땅 탈퇴되는 치명적인 실수가 잦습니다. 

> **📖 전문 용어 해설**
> * **그룹원(Group Member):** UNIX 시스템 안에서 특정 파일이나 디렉터리에 대한 접근 권한을 공유받기 위해 논리적으로 묶인 사용자 집합을 말합니다.

<br><br><br>

## 9. 계정이 존재하지 않는 GID 금지

**🚨 보안 위협:** 더 이상 사용자가 없는 유령 그룹이 남아있을 경우, 해당 그룹의 소유권으로 설정된 파일에 접근할 때 권한 관리에 허점이 생겨 의도치 않은 권한 부여나 보안 감사상의 심각한 사각지대를 만들 위험이 있습니다.
**⚙️ 조치 시 영향:** 불필요한 그룹을 제거하더라도 일반적인 시스템 운영에는 영향이 없습니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX, AIX, HP-UX ]**
```bash
Step 1) /etc/group, /etc/gshadow 파일에 계정이 존재하지 않거나, 불필요한 그룹 확인
Step 2) 불필요한 그룹 제거
# groupdel <그룹 이름>
# ※ 해당 그룹 제거 시 그룹 권한으로 존재하는 파일이 존재하는지 확인이 필요하며, 사용자가 없는 그룹이더라도 추후 권한 할당을 위해 그룹을 먼저 생성하였을 가능성도 존재하므로 확인 필요
```

> **💡 실전 팁 & 자주 하는 실수**
> `groupdel`을 실행하기 전에 반드시 `find / -group [그룹명]` 명령을 통해 이 그룹을 소유자로 갖고 있는 파일이 시스템 내에 남아 있는지 파악해야 합니다. 남아 있다면 다른 정상적인 그룹으로 권한을 재할당한 후 제거하는 것이 안전합니다.

> **📖 전문 용어 해설**
> * **GID(Group Identification, 그룹 식별자):** 다수의 사용자가 파일이나 자원을 안전하게 공유할 수 있게 시스템 차원에서 그룹에 부여하는 특정 식별 번호입니다.

<br><br><br>

## 10. 동일한 UID 금지

**🚨 보안 위협:** 여러 계정이 같은 UID를 가질 경우 시스템은 이들을 기술적으로 동일 사용자로 인식하게 되어 과도한 권한 중복이 생길 뿐 아니라, 시스템 감사 로그에서도 어느 계정 사용자가 작업을 수행했는지 구분이 불가능해지는 위험이 존재합니다.
**⚙️ 조치 시 영향:** 서비스 운영 목적상 억지로 동일한 UID를 할당했던 경우라면, 권한(UID)을 분리할 시 기존에 접근하던 파일/디렉터리에 접근 불가가 발생할 수 있으므로 영향도 사전 검토가 필수적입니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX, AIX, HP-UX ]**
```bash
Step 1) /etc/passwd 파일에 동일한 UID가 존재하는지 확인
Step 2) 명령으로 중복된 UID로 변경
# usermod -u <변경할 UID> <사용자 이름>
# ※ AIX의 경우 chuser id=<변경할 UID> <사용자 이름> 명령어 사용
```

> **💡 실전 팁 & 자주 하는 실수**
> 보통 `grep`이나 `awk`를 사용하여 콜론(`:`) 3번째 필드를 검사하면 중복 UID를 쉽게 파악할 수 있습니다. 중복 UID가 있을 경우, 이를 새로운 UID로 바꾸면 해당 사용자가 예전에 생성했던 파일 소유권을 잃어버리므로 일괄적인 소유권 변경 작업(`chown -R`)이 수반되어야 합니다.

> **📖 전문 용어 해설**
> * **UID(User Identification, 사용자 식별자):** 리눅스/유닉스 커널 단위에서 계정을 문자가 아닌 고유 숫자로 맵핑하여 권한을 판별하는 기준이 되는 식별자입니다.

<br><br><br>

## 11. 사용자 Shell 점검

**🚨 보안 위협:** 데몬 실행 등의 목적으로 생성되어 실제 로그인이 필요 없는 시스템 기본 계정에 Shell(쉘)이 부여되어 있으면, 공격자가 해당 계정으로 터미널 접속을 획득해 악의적인 명령어를 실행할 수 있는 위험이 큽니다.
**⚙️ 조치 시 영향:** 쉘 권한만 회수하는 것이므로, 시스템 내부에서 동작하는 데몬 및 서비스 등에는 영향을 주지 않습니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX, AIX, HP-UX ]**
```bash
Step 1) /etc/passwd 파일을 참고하여 로그인이 불필요한 계정에 /bin/false(/sbin/nologin) 쉘 부여 여부 확인
# cat /etc/passwd | grep -E "^daemon|^bin|^sys|^adm|^listen|^nobody|^nobody4|^noaccess|^diag|^operator|^games|^gopher" | grep -v admin

Step 2) 로그인이 불필요한 계정에 /bin/false 또는 /sbin/nologin 쉘 부여
# usermod -s /bin/false <계정명>
# usermod -s /sbin/nologin <계정명>

# 로그인이 불필요한 계정 목록
# deamon, bin, sys, adm, listen, nobody, nobody4, noaccess, diag, operator, games, gopher
```

> **💡 실전 팁 & 자주 하는 실수**
> 관리자들이 주로 스크립트로 일괄 적용 시, 시스템 구동에 필수적인 서비스 계정임에도 실수로 로그인 쉘을 삭제하여 데몬 자체가 부팅 시 올라오지 못하는 장애를 유발하곤 합니다. 반드시 목록을 대조하고 신중하게 `/sbin/nologin`을 지정해야 합니다.

> **📖 전문 용어 해설**
> * **Shell(쉘):** 운영체제의 가장 외곽 계층에서 대화형 인터페이스를 통해 사용자가 입력한 명령어를 해석하고 실행 결과를 돌려주는 프로그램입니다.

<br><br><br>

## 12. 세션 종료 시간 설정

**🚨 보안 위협:** 콘솔에 접속한 상태로 자리를 비우거나 실수로 터미널을 끄지 않고 방치할 시, 유휴 시간 동안 악의적인 외부자가 시스템에 접근해 불필요한 내부 정보 및 권한을 탈취할 위험이 매우 높습니다.
**⚙️ 조치 시 영향:** 서버 상태를 실시간으로 계속 관제해야 하는 모니터링 콘솔 등의 경우, 타임아웃으로 인해 업무가 빈번히 끊길 수 있으므로 별도의 예외 처리가 필요합니다.

### 점검 및 조치 사례

**[ SOLARIS, LINUX, AIX, HP-UX ]**
```bash
[sh, ksh, bash]
Step 1) /etc/profile 파일 내 TMOUT 값 설정
TMOUT=600
export TMOUT

[csh]
Step 1) /etc/csh.cshrc 또는 /etc/csh.login 파일 내 autologout 값 설정
set autologout=10
```

> **💡 실전 팁 & 자주 하는 실수**
> `/etc/profile`에 `TMOUT`을 설정해도, 개별 사용자가 자신의 홈 디렉터리에 있는 `~/.bash_profile`에서 `TMOUT=0` 등으로 값을 덮어써서 정책을 우회하는 실무적 허점이 종종 발견됩니다. 이를 막기 위해서는 `readonly TMOUT` 명령어를 함께 적용해 변수 수정을 제한하는 것이 좋습니다.

> **📖 전문 용어 해설**
> * **Session(세션):** 인증을 거친 사용자가 시스템과 연결을 수립하여 통신 및 작업을 수행하는 지속적인 상태 구간을 의미합니다.
> * **Timeout(타임아웃):** 보안상 설정된 특정 시간 동안 어떠한 키보드 입력이나 작업 지시도 없을 때, 연결된 세션을 강제적으로 종료하는 조치입니다.

<br><br><br>

## 13. 안전한 비밀번호 암호화 알고리즘 사용

**🚨 보안 위협:** 해독이 수월한 구형(취약한) 암호화 알고리즘을 사용하면, 비밀번호 파일이 노출되었을 때 레인보우 테이블이나 크래킹 툴을 이용한 암호 복호화 공격에 쉽게 무너져 계정이 순식간에 탈취될 위험이 존재합니다.
**⚙️ 조치 시 영향:** 일반적인 시스템 기능 및 성능에는 거의 영향을 미치지 않습니다.

### 점검 및 조치 사례

**[ SOLARIS ]**
```bash
Step 1) /etc/passwd 파일 내 암호화 필드 값 확인
Step 2) /etc/security/policy.conf 파일 내 CRYPT_DEFAULT 값 설정
CRYPT_DEFAULT=5 또는 6
# ※ CRYPT_DEFAULT=5: SHA-256 / 6: SHA-512
```

**[ LINUX - Redhat ]**
```bash
Step 1) /etc/shadow(또는 /etc/passwd) 파일 내 암호화 필드 값 확인
Step 2) /etc/login.defs 파일 내 ENCRYPT_METHOD 값 설정
ENCRYPT_METHOD <SHA-2 이상 암호화 알고리즘(SHA-256 또는 SHA-512)>

Step 3) /etc/pam.d/system-auth 파일 내 안전한 알고리즘 설정
password sufficient pam_unix.so <SHA-2 이상 암호화 알고리즘>
```

**[ LINUX - Debian ]**
```bash
Step 1) /etc/shadow(또는 /etc/passwd) 파일 내의 암호화 필드 값 확인
Step 2) /etc/login.defs 파일 내 ENCRYPT_METHOD 값 설정
ENCRYPT_METHOD <SHA-2 이상 암호화 알고리즘(SHA-256 또는 SHA-512 또는 yescrypt)>

Step 3) /etc/pam.d/common-password 파일 내 안전한 알고리즘 설정
password [success=2 default=ignore] pam_unix.so <SHA-2 이상 암호화 알고리즘>
```

**[ AIX ]**
```bash
Step 1) /etc/security/passwd 파일 내 비밀번호 암호화 알고리즘 확인
password = {<암호화 알고리즘>}<해시값>

Step 2) 안전한 암호화 알고리즘 설정
# chsec -f /etc/security/login.cfg -s usw -a pwd_algorithm=<SHA-2 이상 암호화 알고리즘(SHA-256 또는 SHA-512)>
# ※ /etc/security/pwdalg.cfg 파일을 참조하여 OS에서 정의된 암호화 알고리즘 확인 가능
```

**[ HP-UX ]**
```bash
Step 1) /etc/shadow 파일 내의 암호화 필드 값 확인
Step 2) /etc/default/security 파일 내 CRYPT_DEFAULT 값 설정
CRYPT_DEFAULT = 5 또는 6
# ※ HP-UX 11i v2 이상이며, PHI 및 shadow password를 사용하지 않는 경우 취약
# ※ CRYPT_DEFAULT = 5: SHA-256 / 6: SHA-512
```

> **💡 실전 팁 & 자주 하는 실수**
> 환경설정에서 SHA-512 등으로 암호화 알고리즘 설정을 강화했다고 안심해서는 안 됩니다. 시스템 정책만 바뀐 것이지, 기존 사용자들의 해시까지 일괄 변환되는 것은 아닙니다. 반드시 기존 사용자들에게 **비밀번호 재설정을 유도**해야 비로소 변경된 강력한 알고리즘이 덮어씌워져 적용됩니다.

> **📖 전문 용어 해설**
> * **암호화 알고리즘(Encryption Algorithm):** 원본 데이터(평문)를 수학적 연산을 통해 제3자가 쉽게 해독할 수 없는 형태(해시 또는 암호문)로 변환하는 논리적 규약입니다. SHA-256, SHA-512 등이 강력한 단방향 해시 알고리즘의 대표적 예입니다.
