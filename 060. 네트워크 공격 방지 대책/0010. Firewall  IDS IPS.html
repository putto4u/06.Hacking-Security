Step 01. 접근 제어의 기초

방화벽의 진화와 기술적 메커니즘

방화벽(Firewall, 네트워크 경계에서 트래픽을 필터링하는 보안 장치)은 단순한 패킷 차단을 넘어 네트워크 상태(State)와 애플리케이션 컨텐츠를 분석하는 형태로 발전해 왔습니다.

1세대: 패킷 필터링

(Packet Filtering, 데이터 패킷의 헤더 정보만을 검사하는 방식)

  • IP 주소 및 Port 번호 기반 제어
  • OSI 3/4계층에서 동작
  • 속도는 빠르나 데이터 내부 공격 탐지 불가

2세대: 상태 기반 점검

(Stateful Inspection, 세션 연결 상태를 추적하여 트래픽을 허용하는 방식)

  • TCP 연결의 3-Way Handshake 상태 저장
  • 허용된 세션의 리턴 트래픽 자동 허가
  • 보안성과 성능의 균형 제공

[그림 1-1] 상태 기반 점검(Stateful Inspection) 논리 흐름

방화벽은 세션 테이블을 조회하여 이미 허용된 연결인지 즉각 판단합니다.

* AWS 실무 참고: 클라우드 환경의 Security Group(보안 그룹, 인스턴스 단위의 유입/유출 제어 도구)은 기본적으로 '상태 기반(Stateful)'으로 동작하여, 아웃바운드 허용 시 인바운드 응답 패킷을 자동으로 허용합니다. (※ AWS Network Firewall은 기가바이트당 처리 비용 및 시간당 엔드포인트 유지 비용이 발생하므로 설계 시 주의가 필요합니다.)
Step 02. 능동적 위협 탐지

침입 탐지(IDS) 및 방지 시스템(IPS)

방화벽이 '담장'이라면, IDS/IPS는 담장을 넘으려는 행위나 내부에서의 수상한 움직임을 감시하는 'CCTV와 보안요원'에 비유할 수 있습니다.

배치 방식에 따른 분류

IDS: Mirroring Mode

(미러링 모드, 원본 트래픽의 복사본을 전달받아 분석하는 방식)

서비스 성능 저하가 없으나 공격 차단에 시차가 발생합니다.

IPS: Inline Mode

(인라인 모드, 트래픽이 장비를 직접 통과하며 즉시 차단하는 방식)

실시간 차단이 가능하나 장비 장애 시 네트워크 전체가 단절될 수 있습니다.

탐지 기법 (Methodology) 탐지 논리 (Logic) 장단점 (Pros & Cons)
오용 탐지 (Signature) 기존 공격 패턴 DB와 1:1 대조 정확도가 매우 높으나 신종 공격 탐지 불가
이상 탐지 (Anomaly) 평소 트래픽 임계값 초과 시 탐지 Zero-Day 공격 탐지 가능하나 오탐(False Positive) 발생
Step 03. 심층 방어 전략

현대적 보안 계층: Defense in Depth

단일 장비로는 복합적인 공격을 막을 수 없습니다. 심층 방어(Defense in Depth, 여러 겹의 보안 계층을 쌓아 보호하는 전략)를 통해 한 계층이 뚫리더라도 다음 계층에서 대응할 시간을 벌어야 합니다.

• 경계 보안(Perimeter): 방화벽, DDoS 방어 장비

• 네트워크 보안(Network): IPS, 가상 사설망(VPN, 공용망을 사설망처럼 안전하게 사용하는 기술)

• 애플리케이션 보안(Application): WAF(Web Application Firewall, 웹 전용 공격을 차단하는 방화벽)

• 데이터 보안(Data): DB 암호화, 접근 제어 솔루션

[그림 3-1] 다층 보안 아키텍처 모델

실무 아키텍트의 보안 설계 가이드

1. 차세대 방화벽(NGFW)의 도입

최근에는 방화벽, IPS, VPN, WAF 기능이 하나로 합쳐진 차세대 방화벽(Next Generation Firewall)을 주로 사용합니다. 관리 포인트는 줄어들지만, 성능 부하에 대한 면밀한 사이징이 필요합니다.

2. 오탐(False Positive) 관리의 중요성

보안 장비 도입보다 중요한 것은 정상적인 서비스 트래픽을 차단하지 않도록 정책을 정교화(Tuning)하는 과정입니다. 초기 1~2개월은 '탐지(Detection) 모드'로 운영하며 로그를 분석하는 것이 국룰입니다.

3. 가시성(Visibility) 확보

모든 트래픽은 로그로 남겨야 합니다. 암호화된 트래픽(HTTPS) 내부에 숨겨진 악성 코드를 잡기 위해 SSL 가시성 장비를 병행 구축하는 것을 권장합니다.
