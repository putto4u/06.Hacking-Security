1. 동적 분석의 메커니즘 (Understanding DAST)

DAST (Dynamic Application Security Testing, 동적 애플리케이션 보안 테스트)는 애플리케이션이 실제 실행 중인 런타임 (Runtime, 프로그램이 가동되고 있는 상태) 환경에서 보안 취약점을 찾는 기술입니다. 소스코드를 직접 보지 않고 외부에서 공격자가 수행하는 것과 유사한 방식으로 요청(Request)을 보내고 응답(Response)을 분석하여 취약점을 판단하는 '블랙박스 테스팅' 방식입니다.

※ 설명: DAST는 소스코드가 없는 외부 라이브러리나 API 환경에서도 적용 가능하며, 특히 서버 설정 오류나 인프라 취약점을 찾아내는 데 매우 효과적입니다.

2. 트래픽 인터셉트 및 수동 분석 (Manual Interception & Analysis)

2.1 프록시 서버를 통한 트래픽 변조

수동 점검의 핵심은 클라이언트와 서버 사이에서 프록시 (Proxy, 대리 요청 서버) 도구를 사용하여 전송되는 데이터를 가로채고 수정하는 것입니다. 이를 통해 런타임 환경에서 민감한 정보가 평문으로 전송되는지 확인할 수 있습니다.

주요 점검 도구

  • • Burp Suite (버프 스위트): 전문적인 모의해킹 프록시 도구
  • • OWASP ZAP (오와스프 잽): 강력한 오픈소스 보안 스캐너
  • • Fiddler (피들러): HTTP 트래픽 디버깅 및 분석 도구

변조 시나리오 (Manipulation)

결제 요청 시 금액 파라미터를 price=10000에서 price=1로 변조하여 서버 측 검증 미흡(Insecure Direct Object Reference) 여부를 테스트합니다.

3. 메모리 덤프 분석 기법 (Memory Dump Analysis)

휘발성 데이터 내 민감 정보 추적

프로그램이 실행되는 동안 데이터는 RAM (Random Access Memory, 임의 접근 기억 장치)에 저장됩니다. 공격자가 시스템 권한을 얻어 **메모리 덤프 (Memory Dump, 메모리 전체 내용을 파일로 추출한 것)**를 수행하면, 코드에서는 보호된 변수라도 메모리 상에는 평문으로 존재할 수 있습니다.

[도식: 메모리 구조 내 민감 데이터 잔상 (Data Remnants in RAM)]

# 리눅스 환경에서 프로세스 메모리를 덤프하고 문자열을 추출하는 과정 예시
$ gcore -o dump_file [PID]
$ strings dump_file | grep -Ei "token|password|key"

메모리 보호 전략: 가비지 컬렉션의 한계

Java나 Python 같은 언어의 가비지 컬렉션 (Garbage Collection, 메모리 자동 관리 기법)은 즉각적으로 메모리를 지우지 않습니다. 민감 정보 사용 후에는 변수를 null로 처리하기보다, 직접 메모리를 덮어쓰거나 Secure String (보안 문자열 객체)을 사용하는 설계가 필요합니다.

4. 아키텍트 인사이트 (Expert Insight & Cloud Integration)

무작위 대입 및 퍼징 (Fuzzing)

퍼징 (Fuzzing, 예상치 못한 무작위 데이터를 입력하여 결함을 찾는 기법)은 DAST의 꽃입니다. 수천 개의 비정상적인 요청을 통해 서버가 예외 처리를 하지 못하고 민감한 시스템 정보를 내뱉거나 다운되는 현상을 포착하여 DoS (Denial of Service, 서비스 거부 공격) 취약점까지 탐지할 수 있습니다.

동적 분석의 한계와 보완

DAST는 실행되지 않는 경로(Dead Code)는 분석할 수 없습니다. 따라서 앞서 배운 SAST (정적 분석)와 결합한 **인터랙티브 분석 (IAST, Interactive Application Security Testing)** 방식을 도입하여 두 기법의 장점을 극대화하는 아키텍처 수립이 권장됩니다.

보안 서비스 비용 분석 (Amazon Inspector Pricing)

AWS 환경에서 실행 중인 인스턴스의 런타임 취약점을 자동으로 스캔하려면 Amazon Inspector (아마존 인스펙터, 클라우드 리소스 취약점 스캐닝 서비스)를 활용하십시오. 이 서비스는 EC2 인스턴스 스캔 1회당 약 $1.25의 비용이 발생하며, **람다(Lambda) 함수 스캔 시 $0.30**의 과금이 발생합니다. 비용 최적화를 위해 실시간 스캔보다는 중요 배포 시점에 맞춘 정기 스캔 주기를 설정하는 전략이 필요합니다.
